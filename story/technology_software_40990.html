Defense against XSS Worms

by xloong on 2007-12-31 00:54:55

By Axis

December 29, 2007

http://www.ph4nt0m.org

Recently, XSS Worms have become increasingly rampant. With the development of AJAX technology, these advanced XSS attack techniques have become a current hot topic.

During Christmas, Baidu's personal space suffered from an unprecedented XSS Worm attack. Within a short period, 8,700 blog pages were modified and spread further. Baidu fixed the vulnerability on the 26th and issued an announcement.

http://hi.baidu.com/%B0%D9%B6%C8%BF%D5%BC%E4/blog/item/0e3433fa69eeb61aa8d3110f.html

For this Worm attack, there is a complete technical analysis on Jianxin's blog:

http://www.loveshell.net/blog/blogview.asp?logID=283

In comparison to the previous XSS Worms on MySpace, recent outbreaks such as Orkut and Baidu's XSS Worms indicate that this threat is becoming increasingly serious. It can be foreseen that in 2008, XSS Worms will become the focus of attacks and defenses.

To defend against XSS Worms, you can consider the following points (this will not involve too many details):

1. Cut off the source

The necessary condition for an XSS Worm is that the website has an XSS vulnerability, and this XSS vulnerability must be a persistent (or called stored) type of XSS. This vulnerability must interact with other users on the site.

Since it is a stored type of XSS, it provides the possibility of control at the server end (if it is DOM-based, it may not pass through the server). Therefore, using a security-tier to control XSS is a necessary choice.

There are many options for choosing this security-tier, such as projects like Anti-Samy or tools like Mod-Security. Their purpose is all to perform Input Validation. A more thorough approach is to use htmlencode to escape output, which applies where user output does not need HTML. During code review, you should try to find these places.

If input validation is done thoroughly, it cuts off the source of the XSS Worm, thus preventing the attack from running.

2. Check all places where users interact

This is also a sufficient condition for an XSS Worm to occur.

Check the business logic where user interaction is most intensive, then strengthen the checks of the XSS FILTER for these places, reducing the Attack Surface.

3. Use CAPTCHA or passwords to verify important application processes

Based on the results of point 2, use CAPTCHA or passwords to verify user identity during sensitive applications. The advantage of doing this is that it can promptly interrupt the attack process, preventing the attack from continuously running, minimizing the impact of XSS.

However, the downside is equally clear. Due to commercial needs, many places may not be able to use CAPTCHA, once again testing the balance between usability and security.

4. Use a function in a relatively core position to block consecutive submissions

Generally, the structure of an XSS Worm is on a certain page, meaning that after triggering XSS, the user may execute all JS codes in an instant. These codes include finding friends' lists, then sending new XSS PAYLOADS to these friends' lists.

According to the results of point 2, set submission intervals for these sensitive applications, such as sending messages to friends or submitting articles. After one submission, another submission can only be made after a certain time has passed. Try to minimize losses as much as possible.

5. Complete logging mechanism

Whether it's Apache logs or separate application user behavior logs, they need to be redesigned for XSS Worms, placing checkpoints at sensitive locations. This helps identify problems in the first instance and fix them.

At the same time, during the "criminal apprehension" process, these logs are also very important.

6. Legal systems need further improvement

Clearly, legal systems need to further define XSS Worms.

This article is just throwing out a brick to attract jade, without describing more details.

When XSS Worms are at their most powerful, they will form Worms across different websites, using the interactivity between multiple websites to spread to different sites, making it even harder to trace the origin.

In the field of defending against XSS Worms, there is still a lot of work to be done. You can fully utilize your imagination, and there are more methods that can be used in the process of resistance, such as setting up monitoring points on each page, etc. Before the big wave of XSS Worms in 2008 arrives, I hope everyone can prepare well for flood prevention!
