This article will focus on the first point, speed.

All aspects of performance A modern browser is a platform akin to an operating system. Before Chrome, browsers were single-process applications where all pages shared the same address space and resources. The introduction of a multi-process architecture was one of Chrome's most famous improvements [Translator's Note: Some repetitive details have been omitted].

Within a process, web applications primarily need to perform three tasks: fetching resources, page layout and rendering, and running JavaScript. Rendering and scripting are alternately executed in a single-threaded manner during operation, the reason being to maintain DOM consistency, and because JavaScript itself is a single-threaded language. Therefore, optimizing rendering and script execution is crucial for both page developers and browser developers.

Chrome's rendering engine is WebKit, and its JavaScript engine uses the highly optimized V8 ("V8" JavaScript runtime). However, if the network is not smooth, optimizing V8's JavaScript execution or WebKit's parsing and rendering has very limited effect. You can't make a silk purse out of a sow's ear; data must arrive before proceeding!

In terms of user experience, the most significant aspect is how to optimize the loading order, priority, and latency time (latency) of network resources. You may not notice it, but Chrome's network module improves daily, gradually reducing the loading cost of each resource: learning from DNS lookups, remembering the topology of the web, pre-connecting to likely target URLs, and more. From the outside, it appears as a simple resource loading mechanism, but internally, it's a fascinating world.

Additionally, on average, each resource is only about 12KB, indicating that most network transmissions are short and bursty. This is inconsistent with TCP's direction towards large data and streaming downloads, thus introducing some complications. Let's use an example to unravel this...

The Life of a Resource Request The W3C's Navigation Timing specification defines a set of APIs that allow observation of the timing and performance data of each request in the browser. Here are some details:

After being given a web resource URL, the browser checks the local cache and application cache. If the resource was previously obtained and appropriate cache headers (such as Expires, Cache-Control, etc.) exist, the request is filled with cached data, as the fastest request is one that is not made. Otherwise, we revalidate the resource, and if it has expired or was never seen, a network-consuming request is inevitably sent.

Given a hostname and resource path, Chrome first checks if existing open connections can be reused, i.e., sockets defined by a connection pool (pool) based on (scheme, host, and port). However, if a proxy is configured or a proxy auto-config (PAC) script is specified, Chrome checks the connection with the proxy. PAC scripts provide different proxies based on the URL or specify particular rules. Each proxy can have its own socket pool. Finally, if none of the above apply, the request begins with a DNS query (DNS lookup) to obtain its IP address.

Luckily, the hostname might already be cached. Otherwise, a DNS Query must be initiated. The time required for this process depends on the ISP, the popularity of the page, the likelihood of the hostname being cached in intermediate caches, and the response time of authoritative servers. That is, there are many variables, though it generally doesn't exceed hundreds of milliseconds.

Once the resolved IP is obtained, Chrome opens a new TCP connection to the target address, executing a "three-way handshake": SYN > SYN-ACK > ACK. This operation must be completed for every new TCP connection, with no shortcuts. Depending on distance and route selection, this process may take hundreds of milliseconds, or even seconds. And yet, we haven't received a single valid byte.

When the TCP handshake is complete, if connecting to an HTTPS address, there is also an SSL handshake process, adding up to two additional rounds of latency. If the SSL session is cached, only one round is needed.

Finally, Chrome sends the HTTP request (as shown in the diagram with requestStart). After receiving the request, the server sends response data back to the client. This includes at least one round-trip delay and service processing time. Then the request is complete. But what if it's an HTTP redirect? We have to start the process again. If your page contains redundant redirects, you should reconsider!

Have you calculated all the latency times? Let's assume a typical broadband environment: no local cache, relatively fast DNS lookup (50ms), TCP handshake, SSL negotiation, and a faster server response time (100ms) and one delay (80ms, the average within the U.S.): 50ms for DNS 80ms for TCP handshake (one RTT)

160ms for SSL handshake (two RTTs)

40ms (sending request to server)

100ms (server processing)

40ms (server transmitting response data)

A request takes 470 milliseconds, with 80% of the time consumed by network latency. As you can see, we have a lot to do! In fact, 470 milliseconds is already optimistic:

If the server does not reach the initial TCP congestion window (congestion window), i.e., 4-15KB, it introduces more round-trip delays. SSL latency could also worsen. If a certificate needs to be retrieved or an online certificate status check (OCSP) is performed, we would need a new TCP connection, adding hundreds to thousands of milliseconds of latency.

How fast is 'fast enough'? Previously, server response time accounted for only 20% of total latency, while DNS and handshakes occupied the rest. Past user experience research indicates different user reactions to latency:

Latency and User Reaction 0 – 100ms Quick 100 – 300ms Somewhat slow 300 – 1000ms Still running 1s+ Thinking about something else… 10s+ I'll come back later…

The table applies equally to page performance: rendering a page should respond within at least 250ms to capture user attention. This is simply targeting speed. Studies from Google, Amazon, Microsoft, and thousands of other sites show that additional latency directly impacts page performance: smooth pages attract more browsing, stronger user engagement, and higher conversion rates.

Now we know the ideal latency is 250ms, while the previous example shows that DNS Lookup, TCP and SSL handshakes, and request preparation took 370ms, exceeding the limit by 50%, even without considering server processing time.

For most users and web developers, DNS, TCP, and SSL latency are transparent, rarely considered. This is why Chrome's network module is so complex.

We've identified the problem; now let's delve into the implementation details…

Diving Into Chrome's Network Module Multi-process Architecture

Chrome's multi-process architecture plays a significant role in handling network requests for the browser, currently supporting four different execution modes.

By default, desktop Chrome uses the process-per-site mode, isolating different website pages while grouping pages from the same site together. For example: each tab has an independent process. From a network performance perspective, there is no fundamental difference, though the process-per-tab mode is easier to understand.

Each tab has a rendering process (render process), which includes WebKit's layout engine for interpreting and laying out pages, as shown in the HTML Render in the diagram. There is also the V8 engine and DOM Bindings between them. If you're curious about this part, you can find a great introduction here.

Each such rendering process runs in a sandboxed environment, allowing extremely limited access to the user's computer— including networking. To use these resources, each rendering process must communicate with the browser kernel process to manage security and access policies for each rendering process.

Inter-process Communication (IPC) and Multiprocess Resource Loading Communication between the rendering process and the kernel process is achieved through IPC. On Linux and Mac OS, a socketpair() providing asynchronous named pipe communication is used. Messages from each rendering process are serialized into a dedicated I/O thread, then forwarded to the kernel process. At the receiving end, the kernel process provides a filter interface for parsing resource-related IPC requests (ResourceMessageFilter), which is handled by the network module.

One benefit of this approach is that all resource requests are processed by the I/O process, whether generated by UI activities or triggered by network events. The I/O thread in the browser/kernel process parses resource request messages, forwarding them to a singleton ResourceDispatcherHost object for handling.

This singleton interface allows the browser to control each rendering process's access to the network and achieve effective and consistent resource sharing:

Socket pool and connection limits: The browser can limit each profile to open 256 sockets, each proxy to open 32 sockets, and each {scheme, host, port} group to open 6. Note that simultaneously, up to 6 HTTP and 6 HTTPS connections are allowed for each {host, port} group. Socket reuse: Persistent available TCP connections provided in the Socket Pool can be reused, avoiding extra time spent on establishing DNS, TCP, and SSL (if necessary).

Socket late-binding (delayed binding): Network requests are always associated with a TCP connection when the socket is ready to send data, offering the opportunity to prioritize requests effectively, such as higher-priority requests arriving during socket connection processes. It also improves throughput, like reusing an available socket during connection setup to fully utilize a TCP connection. Traditional TCP pre-connect and other optimizations have similar effects.

Consistent session state: Authorization, cookies, and cache data are shared across all rendering processes.

Global resource and network optimizations: The browser can make better decisions across all rendering processes and unprocessed requests. For instance, giving better priority to requests corresponding to the current tab.

Predictive optimizations: By monitoring network activity, Chrome builds and continuously improves predictive models to enhance performance.

... More projects are ongoing.

For a single rendering process, sending resource requests via IPC is straightforward, requiring only a unique ID communicated to the browser kernel process, leaving the rest to the kernel process.

Cross-platform Resource Loading Cross-platform considerations are a major factor in Chrome's network module, including support for Linux, Windows, OS X, Chrome OS, Android, and iOS. To facilitate this, the network module is implemented as a single-process library (only separate cache and proxy processes), enabling shared infrastructure and identical performance optimizations across platforms, providing simultaneous optimization opportunities for all platforms.

The relevant code can be found in the “src/net” subdirectory. This article won't go into detail on each component, but understanding the code structure can help grasp its capability structure. For example:

Architecture and Performance on Mobile Platforms Mobile browsers are rapidly developing, and the Chrome team prioritizes optimizing mobile experiences. First, it's important to note that the mobile version of Chrome is not a direct port of its desktop counterpart, as doing so would not provide a good user experience. The inherent characteristics of mobile devices dictate a severely resource-constrained environment, leading to fundamental differences in operational parameters:

Desktop users operate with a mouse, overlapping windows, large screens, and no battery concerns. Networks are stable, with ample storage and memory. Mobile users interact via touch and gestures, have smaller screens, limited battery life, slow and expensive networks, and constrained storage and memory.

Moreover, there is no typical mobile device; instead, there's a wide variety of hardware. Chrome aims to accommodate these devices. Fortunately, Chrome's different execution models handle these challenges with ease!

In the Android version, Chrome employs the same multi-process architecture as its desktop counterpart - a browser kernel process and one or more rendering processes. However, due to memory constraints, the mobile version cannot run a specific rendering process for each tab; instead, it determines the optimal number of rendering processes based on conditions like memory availability, then shares these processes among multiple tabs.

If memory is insufficient or other reasons prevent Chrome from running multiple processes, it switches to a single-process, multi-threaded mode. For example, on iOS devices, Chrome operates in this mode due to restrictions imposed by its sandbox mechanism.

Regarding network performance, Chrome uses the same network module on Android and iOS as on other platforms. This enables cross-platform network optimization, one of Chrome's clear advantages. Adjustments are frequently made based on network conditions and device capabilities, including the priority of speculative optimizations, socket timeout settings and management logic, and cache size.

For instance, to extend battery life, the mobile version of Chrome tends to delay closing idle sockets (lazy closing of idle sockets), typically to reduce radio usage by closing old sockets when opening new ones. Additionally, since prerendering consumes certain network and processing resources, it is usually only enabled on Wi-Fi.

A chapter dedicated to the mobile browsing experience will appear independently, perhaps in the next issue of the POSA series.

Optimization Through Chrome Predictor's Predictive Functionality Chrome gets faster with use. This feature is implemented through a singleton object called Predictor. This object is instantiated in the browser kernel process, and its sole responsibility is to observe and learn current network activity patterns, predicting the user's next steps. Here's an example:

When a user hovers over a link, it suggests a preference and predicts the next browsing behavior. At this point, Chrome can preemptively perform DNS Lookups and TCP handshakes. The average click operation takes nearly 200ms, during which DNS and TCP-related operations can be completed, saving hundreds of milliseconds in latency. When high-probability options are triggered in the address bar (Omnibox/URL bar), DNS lookups and TCP pre-connections (pre-connect) are similarly triggered, and even pre-rendering occurs in an invisible tab!

We each visit a string of websites daily, and Chrome studies the sub-resources on these pages, attempting pre-resolution (pre-resolve) and potentially pre-fetching to optimize the browsing experience.

Besides the above three points, there are many more...

Chrome learns the topology of the web during use, not just your browsing patterns. Ideally, it saves hundreds of milliseconds in latency, bringing you closer to instant page loading. To achieve this goal, Chrome implements the following core optimization techniques:

DNS Pre-Resolve: Pre-resolving host addresses to reduce DNS latency TCP Pre-Connect: Pre-connecting to target servers to reduce TCP handshake latency Resource Prefetching: Pre-loading core page resources to load and display the page Pre-rendering: Pre-fetching the entire page and related sub-resources to enable immediate display

Every decision involves one or more optimizations to overcome numerous limitations. However, since these are predictive optimizations, they may introduce redundant processing and network transmission if not effective, potentially leading to negative loading time experiences.

How does Chrome handle these issues? The Predictor tries to gather various information, such as user actions, historical browsing data, and information from the render engine and network module itself.
