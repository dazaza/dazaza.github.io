Abstract: This article introduces the reasons for using the Singleton pattern in the C# programming language, and discusses the issues that should be considered when using the Singleton pattern.

Keywords: Pattern; Singleton

Often, we need an object of a type whose instance count is determined to be exactly one within a program itself, such as various functionalities and object concepts in mathematics which are generally stateless. In other words, we may need these functionalities, or they don't have any state because there might only be a conceptual connection between these functionalities (like things in mathematics), but no state or self-contained system; or even if it's a self-contained system, there is only one such system in the entire application (such as in a large-scale application).

For setting up this functionality, we need a method to set it specifically. There are two patterns: one called Monostate, and the other is Singleton. What is Monostate? This pattern is so common that it's not worth analyzing in detail. The VB.NET module and static classes in C# are what we call Monostate—they only contain static methods without a static class structure. Singleton is different—it ensures that a class has only one instance and provides a global access point to it. The main purpose of Singleton is to ensure that there is only one instance of an object relevant to other objects' existence, without requiring clients to manage this. First, in C#, static methods cannot be inherited, so more effort is required when writing code. Second, Monostate does not have a way to automatically release resources, and it will remain throughout the entire lifecycle of the program, even if it's unnecessary to use them. Similarly, the corresponding resource release process is very cumbersome, and it's annoying during this process to prepare resources again after forced release. Finally, the Monostate pattern makes it impossible to reference or pass members further, which is a very serious problem. Therefore, how does Singleton solve these problems? The key part is that all members are not static, so it only requires maintaining a unique object, making the maintenance of current members quite simple. However, applications must pay attention to the Singleton context. For example, in Windows, you can design a Singleton window, which allows users to close it again. If the usage rate of the window is not high, then when users use windows designed with the Monostate pattern, the resources used are difficult to release.
