Development Exercise of Notepad Application (Part I)

by dream2012 on 2012-03-06 11:18:56

In this exercise, you will learn how to add a second Activity to your Notepad application. This Activity allows users to create, edit, and delete notes. It assumes that a new note is created in response to user input and packages itself into a return value Bundle provided by an intent. The goal of this exercise is:

- Build a new Activity and add it to the Android manifest.

- Invoke another Activity using the `startActivityForResult()` method via asynchronous calls.

- Pass data between Activities using various Bundle objects.

- Learn how to use a more advanced screen layout.

If you encounter an error regarding AndroidManifest.xml or android.zip, right-click on the project, select **Android Tools > Fix Project Properties** to resolve the issue.

Open the Notepadv2 project and review the following:

- Observe the class member variable used to store the note pointer called `mNotesCursor`. The leading 'm' indicates it's a member variable conforming to Android coding style standards.

- Implement two new overridden interfaces: `onListItemClick()` and `onActivityResult()`.

mDbHelper.deleteNote(getListView().getSelectedItemId());

### Step 4:

Implement the `createNote()` interface:

Create a new Intent with the class `NoteEdit` to create a note (ACTIVITY_CREATE). Then, invoke the Intent using the `startActivityForResult()` interface:

```java

Intent i = new Intent(this, NoteEdit.class);

startActivityForResult(i, ACTIVITY_CREATE);

```

This form of Intent invocation is specific to the particular class in our Activity, which in this case is `NoteEdit`. Since the Intent class requires requests to be sent through the Android operating system, we must provide a Context (`this`).

The `startActivityForResult()` interface is triggered after the new Activity completes and returns to our Activity. In our Activity, the callback interface is named `onActivityResult()`, which we will implement later. Another interface named `startActivity()` can also invoke another Activity, but it operates in a fire-and-forget manner, meaning the current Activity won't receive any notifications when the other Activity completes, nor can any results be returned from the invoked Activity.

Don’t worry about the fact that `NoteEdit` does not yet exist; we will address that soon.

### Step 5:

Implement the overridden `onListItemClick()` interface:

`onListItemClick()` is a callback function we will override. It is called when the user clicks an item in the note list. It has four parameters: the ListView object invoking the interface, the clicked View (contained within the ListView), the position (the location of the item in the list), and the row index (`mRowId`) of the clicked item. In this example, we ignore the first two parameters (as we only have one ListView) and the row index. Our focus is entirely on the position of the selected item in the list. Using this parameter, we can retrieve the desired data from the corresponding row and package it to send to the `NoteEdit` Activity.

In the implementation of this callback function, we create an Intent with the `NoteEdit` class to modify the note content, then add data to an additional package passed as a parameter to the invoked Activity’s Intent. We pass the title, body, and row index of the currently edited note. Finally, it invokes the `startActivityForResult()` interface to start the Intent. Below is the complete code for the `onListItemClick()` interface:

```java

super.onListItemClick(l, v, position, id);

Cursor c = mNotesCursor;

c.moveToPosition(position);

Intent i = new Intent(this, NoteEdit.class);

i.putExtra(NotesDbAdapter.KEY_ROWID, id);

i.putExtra(NotesDbAdapter.KEY_TITLE, c.getString(c.getColumnIndexOrThrow(NotesDbAdapter.KEY_TITLE)));

i.putExtra(NotesDbAdapter.KEY_BODY, c.getString(c.getColumnIndexOrThrow(NotesDbAdapter.KEY_BODY)));

startActivityForResult(i, ACTIVITY_EDIT);

```

`putExtra()` is an interface that adds data to the additional package to trigger the Intent. Here, we pass the title, body, and row index of the note we want to edit.

When we locate the selected object in the corresponding position in the list using `moveToPosition()`, the details of the note are available in the query pointer.

Through the added information to the Intent and the `startActivityForResult()` interface, we trigger the Intent and request type code in the `NoteEdit` class (the request type code will be returned as a parameter `requestCode` to the `onActivityResult()` interface).

### Step 6:

Open the `note_edit.xml` file and browse through it. This is the code related to the note editing interface.

This is the most complex interface layout we've dealt with so far. The file is provided here to avoid issues with fading in and out during code formatting. (XML has strict rules about case sensitivity and structure, and these errors often stem from layout problems.)

A new parameter we haven't seen before is used here: `android:layout_weight` (in this case, set to 1 in all instances). `layout_weight` assigns importance values to views within a linear layout. All views have a `layout_weight` value, defaulting to zero, meaning they occupy the screen space required to display their size. If a value greater than zero is assigned, the available space in the parent view is divided according to each view's `layout_weight` value and its ratio within the overall `layout_weight` value of the current screen layout compared to other views.

For example: Suppose we have a text label and two text editing elements horizontally aligned. The text label has no specified `layout_weight`, so it will occupy the minimum required space. If both text editing elements have a `layout_weight` value of 1, they will equally divide the remaining width in the parent view layout (because we declared them of equal importance). If the first text editing element has a `layout_weight` value of 1 and the second has a value of 2, one-third of the remaining space goes to the first, and two-thirds go to the second (because we declared the second as having higher importance).

This layout also demonstrates how to nest multiple layouts within other layouts to achieve more complex and aesthetically pleasing designs. In this example, a horizontal linear layout is nested within another vertical layout to align the title label and text field horizontally.

### Step 8:

Implement the definition of the `onCreate()` method for the `NoteEdit` class:

This method sets the title of our new Activity to "Edit Note" (a string defined in the `strings.xml` file) and sets the note content view using the `note_edit.xml` layout file. We obtain handles for the note title, text body views, and confirm button. This allows us to set the note's title and body using these handles and bind the confirm button to respond to user button presses.

We then unpack the parameters passed in the extras bundle of the invoked Intent and pre-fill the title and note text body and views, allowing users to edit them. Next, we obtain and store the note row index (`mRowId`) to know which note the user is currently editing.

Set up the layout in the `onCreate()` method:

```java

setContentView(R.layout.note_edit);

```

Find the necessary editing and button components:

```java

mTitleText = (EditText) findViewById(R.id.title);

mBodyText = (EditText) findViewById(R.id.body);

Button confirmButton = (Button) findViewById(R.id.confirm);

```

Note that `mTitleText` and `mBodyText` are class member variables (you need to declare them in the class before using them here).

Declare a private class member variable `Long mRowId` in the class to store the current row index value `mRowId` (if it exists).

Continue adding code in the `onCreate()` method to initialize the note title, body, and row index ID variables using the Intent extras data package (if present):

```java

mRowId = null;

Bundle extras = getIntent().getExtras();

if (extras != null) {

String title = extras.getString(NotesDbAdapter.KEY_TITLE);

String body = extras.getString(NotesDbAdapter.KEY_BODY);

mRowId = extras.getLong(NotesDbAdapter.KEY_ROWID);

if (title != null) {

mTitleText.setText(title);

}

if (body != null) {

mBodyText.setText(body);

}

}

```

We extract the title and body values set in the extras data package when the Intent was triggered.

We also protect against assigning empty strings to the text fields.

Create an `onClickListener()` method for the button:

Listeners might be one of the more confusing aspects of UI implementation, but in this example, the effect we want to achieve is simple: calling the `onClick()` method when the user presses the confirm button. Use this to do something and return some values to the Intent caller after the note has been edited. We achieve this using an anonymous inner class, which may seem confusing at first unless you've seen it before. However, you should set aside your confusion because there will be opportunities to revisit this code later and understand how to create a Listener and associate it with a button (Listener is a common term in Java development, especially in UI design and implementation).

### Step 9:

Complete the implementation of the `onClick()` method in our Listener:

When the user presses the confirm button, this interface runs. We want this method to retrieve the title and text fields of the edited note and place these data into a return package so they can be passed back to the Activity that initially triggered this `NoteEdit` Activity. If the interface responds to an edit rather than a create operation, we also need to place the row index ID into the return package so the `Notepadv2` class can save the edits made to the note.

Create a return data package (Bundle) and pass the note title and body values using the constants defined in `Notepadv2`:

```java

Bundle bundle = new Bundle();

bundle.putString(NotesDbAdapter.KEY_TITLE, mTitleText.getText().toString());

bundle.putString(NotesDbAdapter.KEY_BODY, mBodyText.getText().toString());

if (mRowId != null) {

bundle.putLong(NotesDbAdapter.KEY_ROWID, mRowId);

}

```

Pass this data package into a new Intent and end the Activity:

```java

Intent mIntent = new Intent();

mIntent.putExtras(bundle);

setResult(RESULT_OK, mIntent);

finish();

```

This new Intent simply acts as a carrier for the data package (including the note title, body, and row index ID we want to pass).

The `setResult()` method sets the result type code and passes the Intent back to its caller. In this example, everything works fine, so we return the result code `RESULT_OK`.

Calling the `finish()` method signals that the Activity is complete (equivalent to a return call). Any interfaces set in the result return control information to the result caller.

The complete implementation of the `onCreate()` method (along with some supporting fields for the class) is as follows:

```java

private EditText mTitleText;

private EditText mBodyText;

private Long mRowId;

@Override

protected void onCreate(Bundle savedInstanceState) {

super.onCreate(savedInstanceState);

setContentView(R.layout.note_edit);

mTitleText = (EditText) findViewById(R.id.title);

mBodyText = (EditText) findViewById(R.id.body);

Button confirmButton = (Button) findViewById(R.id.confirm);

mRowId = null;

Bundle extras = getIntent().getExtras();

if (extras != null) {

String title = extras.getString(NotesDbAdapter.KEY_TITLE);

String body = extras.getString(NotesDbAdapter.KEY_BODY);

mRowId = extras.getLong(NotesDbAdapter.KEY_ROWID);

if (title != null) {

mTitleText.setText(title);

}

if (body != null) {

mBodyText.setText(body);

}

}

confirmButton.setOnClickListener(new View.OnClickListener() {

public void onClick(View view) {

Bundle bundle = new Bundle();

bundle.putString(NotesDbAdapter.KEY_TITLE, mTitleText.getText().toString());

bundle.putString(NotesDbAdapter.KEY_BODY, mBodyText.getText().toString());

if (mRowId != null) {

bundle.putLong(NotesDbAdapter.KEY_ROWID, mRowId);

}

Intent mIntent = new Intent();

mIntent.putExtras(bundle);

setResult(RESULT_OK, mIntent);

finish();

}

});

}

```

### Step 10:

Finally, define the new Activity in the manifest file:

Before the new Activity can be recognized by the Android operating system, it needs its own Activity entry in the `AndroidManifest.xml` file. This way, the Android operating system knows the Activity is in the `AndroidManifest.xml` file and can invoke it. We can also specify which `IntentFilter`s the Activity implements here, but we will skip that part for now and just let the Android operating system know the Activity is defined.

The Eclipse plugin includes a Manifest editor that makes it easy to edit the `AndroidManifest.xml` file, which we will use. If you prefer to directly edit the file or not use the Eclipse plugin, refer to the information below on how to edit the Manifest file without using the new Manifest editor.

Double-click the `AndroidManifest.xml` file in the folder tree to open it.

Click the Application tab at the bottom of the Manifest editor.

In the application section, click Add....

If you see a radio button at the top of the dialog, select the one that says "Create a new element at the application level."

Before clicking "OK," ensure that the "(A)Activity" option is selected in the dialog's tab.

In the Application Node section, click the new "Activity" and enter `.NoteEdit` in the Name* field, then press Enter.

The Android Manifest editor allows you to add more complex Activity entries to the file and explore other available options (but don't select them, as doing so will add them to your Manifest file).
