How I Developed an Android Game from Scratch in 23 Days – Part 1~3

On the 9th day, this is a first-person shooter game, but it's anything but ordinary.

When talking to people about this game, it was quite difficult to define its exact genre. While it can be seen as a traditional arcade game, unlike those games where you move your spaceship around and fire straight shots, your position is fixed and you can shoot in any direction according to instructions. After careful consideration, I've never seen a game exactly like this one, so don't try to categorize it into existing genres. The closest description I can give is: similar to Space Invaders, but without the spaceship. Such a description completely confused people.

Today I rewrote the code for generating aliens and changed all alien colors to white. This allows me to update their colors during gameplay using the same sub-screen, assuming you haven't noticed that the aliens are monochrome. I plan to handle them all this way except for the boss. The boss will have some glowing dynamic effects added. Who knows, maybe an artist will discover the potential of the game and create impressive effects. If this happens, please contact me.

I'm considering giving a hint when enemies are about to "upgrade," like displaying metallic text saying "Aliens Upgraded" alongside the aliens.

The explosion code has been unified. Initially, a 2D "pixel" array displayed enemy fragments after explosions. Now this code has been optimized so that all aliens use the same effect, except for the boss. The boss explodes in phases. This design still needs careful consideration.

In any case, if you're interested in playing the game, this .apk file contains all the progress made so far. Although there's nothing special yet, you can see my continuous effort.

On the 10th day, making the game more fun.

I've started doubting whether this concept really has long-term playability. If the aliens could move left and right, the game would be more fun. Adding two new attack formations feels much better. I also added a screen shake effect when destroying aliens during explosions, making the game feel more alive. The reload functionality has been implemented; now if you empty a 10-round magazine, you'll have to reload. Interrupting shooting to reload adds to the gameplay. I replaced some ugly magenta-colored aliens with cold blue-white tones, which look much better.

I think if players only control the lower half of the screen — the part shielded — the upper half can continuously show the status of enemies without being obstructed by the player's fingers. This situation often occurs. At first, the shield might cover half the screen, gradually shrinking as enemies attack until the gun muzzle is exposed without any protection. Although this means possibly drawing a gun, it's not a bad idea, and I might try it.

The name "Stellar Battle" has been chosen. I tried over 20 other names, but due to the abundance of space-themed games, it's hard to find a free name.

On the 11th day, adding alien types, attack patterns, and the first level boss appears.

The first boss appears on level 10, its shape being a large square. I call it Borg, though it doesn't resemble Borg at all. It continuously fires aliens, and you must hit it many times to destroy it. I set a large vibration effect for defeating the boss. The boss is composed of 10x10 pixel blocks, and the explosion effect is stunning.

I also added a new snake-like movement attack where aliens move left and right while descending repeatedly until reaching the bottom. I slightly increased the speed in this attack pattern, making it more challenging.

I also drew some good-looking aliens and named them, each type named after the Java class. They are: Hairy, Glider, Worker, and Eater.

This is today's .jar file, which can run on Windows, Linux, or Mac if you have Java installed. On Linux, I executed the command:

java -jar stardust.jar

To ensure proper gameplay, note that the window height should not be less than 800. It runs well on my 1680x1050 desktop, but on a 1280x800 laptop, the taskbar occupies some space, compressing the window vertically. Therefore, I had to click below the aliens to shoot. I think if we release a PC version, this issue must be resolved.

Day 12 (Part 1): New Game Name DRONE INVADER

The name is finally chosen. There were many options, but only six or seven were available. This one fits the game theme best. A new theme has also been created, using the Ruslan font.

Today I understood the difference between Comparable and Comparator in Java. I modified the bullet code to allow firing multiple bullets (laser shards) simultaneously. Bullets don't need to touch aliens directly to hit them; they just fire in a direction, and bullets automatically attack aliens. This simplifies judging which alien is hit first based on the trajectory at the moment of firing, replacing full collision detection. This method applies to all levels except Boss levels. Since the Boss spits out other aliens, it should be placed first in the array so bullets fly past all aliens to directly hit the boss. If bullets directly hit the boss, the previous handling is retained. But if shooting in a direction, aliens are sorted by y-coordinate, and the nearest one is hit.

This new feature changes the gameplay experience, making it easier and more fun. Shooting aliens no longer causes misoperations due to finger clicks. The game transitions from "precise collision" to "shooting test," with a 10-round clip to prevent instant destruction of everything.

I think I should make the alien movements more challenging to avoid the game becoming too easy.

Day 12 (Part 2)

I'm starting to like this game, and I really like it. I decided to work a bit more before sleeping, adding a big moon on levels 16 and 26, with aliens hiding behind the moon. The moon slowly moves across the screen, exiting just as the Boss appears. This increases difficulty because it's hard to tell if aliens are hiding behind the moon. Since the moon is circular, this complicates rectangular operations. I investigated precise pixel detection methods, showing individual pixels whenever hitting an alien, especially when the moon keeps rotating. However, this might slow down the game. I reduced the original rectangle by 12% and used rectangular detection instead. Although it's not precise pixel detection, it works well. This allows bullets to pass through the moon and hit aliens using the moon as cover.

I'm very satisfied with the current gameplay experience. It's interesting, challenging, and engaging. Now I need more bosses, more alien types, and power upgrades. Without upgrading power, reaching level 60 is very difficult, so I'm trying to add enough content so that no boss repeats up to level 100. I think adding power upgrades will be easier, and I'll see how the game evolves with these additions.

For the first time, I feel this will be a great game, standing out among regular space shooters.

Day 13: Shields, New Bosses

I added a new boss, bringing the total to two bosses. Here are the current alien names: Worker, Eater, Hairy, Glider. The bosses are called Worker Boss (which looks like a larger Worker and spits out many small Workers) and Borg (which is cube-shaped and breaks into large cubes when destroyed).

I also added shields. I previously drew these nice shield icons using Inkscape and got inspiration from a YT tutorial. I tried following the tutorial (which uses Adobe Illustrator, not Inkscape), but failed. I then observed some favorite shields and noticed they consist of branches, curves, or gradients. I put Hairy on it, and it looked great. This can also serve as an Android icon.

Anyway, the shield can be activated anytime and lasts for 20 seconds. Later upgrades can extend the duration. If aliens touch the shield, they accelerate energy depletion, allowing you to destroy them even after the shield disappears.

The shield graphic looks like a force field. Using Gimp with manual adjustments makes it look better. I want a smooth curve rather than a straight line, as straight lines seem unable to add gradient shadow effects. Perhaps there are techniques to achieve this, but I don't know yet. Finally, I combined multi-angle linear gradients, resulting in a fantastic effect.

I tried adding some glowing effects to the shield icon, but it seemed too glaring. I'll leave this for use when collecting power-ups during flight.

With these new graphics, the game looks more complete. I'm still considering where to place score multipliers and whether to display the current attack wave.

I'm also thinking about letting players buy upgrades before playing, requiring some coins or money. Currently, quick play obviously isn't supported. Maybe leaving crystals or random upgrades behind some aliens could appear. Or, the number of waves you survive determines the number of coins you get.

Day 15: Android "Back" Button, Main Menu, Fixed Coordinate Bug

Do you remember the issue on Day 11 where screen coordinates and mouse clicks couldn't shoot aliens? Yes, that was all my fault. Fortunately, it made me realize that many Android users downloading the game don't have a screen resolution of 800x400. Before that, I was directly converting touch coordinates to actual coordinates:

float x = Gdx.input.getX() - 240f; float y = 400 - Gdx.input.getY();

This wasn't the correct approach. The simple and appropriate method is to use GDX for conversion:

Vector3 touchPos; touchPos.set(Gdx.input.getX(), Gdx.input.getY(), 0); camera.unproject(touchPos);

Handling the "Back" Button on Android

Most online examples mention overriding the KeyDown method when handling the "Back" button. Unfortunately, this requires using Stage, which I didn't do. I know there's a lot of Actor and Stage duplication in the current code, but that doesn't matter. I'll use Stage in the next project.

Luckily, I found a solution. Just add the following function in the create() method of the Game subclass:

Gdx.input.setCatchBackKey(true);

Then check in the render() method if the "Back" button has been pressed:

if (Gdx.input.isKeyPressed(Keys.BACK)) { Gdx.app.exit(); }

Since render() is called many times per second, you might need a boolean flag variable to check if the "Back" button has been released.

if (backReleased && Gdx.input.isKeyPressed(Keys.BACK)) { backReleased = false; Gdx.app.exit(); } else { backReleased = true; }

You can now enter the game, go into the store menu, and return to the main menu. Of course, the menu only displays options and hasn't implemented functions yet.

Day 16: Recording Videos from a GDX Game

Video address: www.youtube.com/embed/RUy177pvT8I?rel=0

I thought about uploading a game video to YouTube and recording it with recordmydesktop, but the result was a mess. Due to libGDX and RMD not syncing, I saw a bunch of pieces on the screen, such as sprites cut in half. I searched and found a few useful articles. Basically, they involved making each frame into a PNG file and compiling them into a video. As expected, this consumes a lot of disk space, which isn't a big problem for me. I found a useful post:

http://www.wendytech.de/2012/07/opengl-screen-capture-in-real-time/

However, there were issues with their code. For some reason, when I overlaid semi-transparent sprites on the background, the resulting PNG files showed semi-transparent pixels in those areas. This generated a lot of messy stuff in the video. I tried different settings and even changed the rendering code, but the problem persisted. Now, a simple processing step—using ImageMagick (adding a black background)—solves this issue. So I thought, if I have to do this step anyway, I might as well do vertical flipping in ImageMagick. So I turned off Y-axis flipping in the code, making it more efficient and eliminating the need to allocate w*h*4 bytes of memory per frame. On an 800x480 screen, each frame requires about 1.5MB!

At the same time, the code handling frame rate (frame skipping) wasn't optimized. The process skipped several file numbers, which wasn't a problem. But it also created corresponding ScreenShot objects for each frame, which was unnecessary. For example, if you're recording a 30fps video while the game runs at 60fps, you're spending half the time creating objects that aren't used.

Finally, the FPS handling code seemed not to release pixel maps. So if you run it for a long time, RAM gets eaten up.

So, I extracted all the FPS code from the ScreenShot class, leaving the rest to handle continuous video. I also noticed some variables that were initialized but never used. Now the ScreenShot class is more intuitive and easier to understand:

public class ScreenShot implements Runnable { private static int fileCounter = 0; private Pixmap pixmap;

@Override public void run() { saveScreenshot(); }

public void prepare() { getScreenshot(0, 0, Gdx.graphics.getWidth(), Gdx.graphics.getHeight(), false); }

public void saveScreenshot() { FileHandle file = new FileHandle("/tmp/shot_" + String.format("%06d", fileCounter++) + ".png"); PixmapIO.writePNG(file, pixmap); pixmap.dispose(); }

public void getScreenshot(int x, int y, int w, int h, boolean flipY) { Gdx.gl.glPixelStorei(GL10.GL_PACK_ALIGNMENT, 1); pixmap = new Pixmap(w, h, Pixmap.Format.RGBA8888); Gdx.gl.glReadPixels(x, y, w, h, GL10.GL_RGBA, GL10.GL_UNSIGNED_BYTE, pixmap.getPixels()); }

Alright, that's all. I added the following at the end of each render in the render loop:

ScreenShot worker = new ScreenShot(); worker.prepare(); // grab screenshot executor.execute(worker); // delayed save in another thread

Considering completeness, I added the executor in the Screen subclass:

private ExecutorService executor; ... executor = Executors.newFixedThreadPool(25);

Now, my Core 2 can't keep up with the frame rate. This is good because, on one hand, the slower game speed allows me to record better videos, and on the other hand, it better records screenshots for later exporting. So I added a screenshot hotkey.
