I believe most friends have been victims of iframe-based malware. Some friends' websites have been injected with iframes multiple times, and their frustration is understandable. Moreover, nowadays ARP attacks and iframe injections are easily achieved, posing constant threats even within local networks. Sigh, what kind of world is this?

Ling'er once posted on a classic forum: "A one-line solution to prevent IFRAME malware injection" at http://bbs.blueidea.com/thread-2785512-1-1.html. Many friends contacted Ling'er, some expressing gratitude, but more had questions. Today, let me explain the principle in detail:

**IE Only** —— Generally, only IE is vulnerable to iframe-based malware, so Ling'er focuses on addressing this issue for IE.

Before reading further, let's first understand **expression**:

Starting from IE5 and later versions, CSS supports the use of **expression**, which allows associating CSS properties with JavaScript scripts. The CSS property can be either an inherent element property or a custom property. This means that after the CSS property, you can write a piece of JavaScript expression, and the value of the CSS property equals the result of executing the JavaScript expression. In the expression, you can directly reference the element's own properties and methods, as well as other browser objects. This expression behaves as if it were a member function of the element.

Many friends know that CSS can directly describe the appearance of a visible tag. For example, `p {color:red}` makes all the text inside `` tags on the webpage red. Isn't an iframe also a tag? Let's start writing the code with Ling'er, haha:

```css

iframe{... /* Here write the CSS code describing the appearance */;}

```

Carefully consider: What is the best way to prevent things inside the iframe from being downloaded? The core idea comes here: cut off the request inside the iframe. Cutting off the request means quickly destroying the iframe object. How to achieve this? Didn't we just introduce **expression**? Expression can execute JS scripts. There are two syntax formats:

1. Native CSS property name: `expression(JS expression);`

2. Custom attribute name: `expression(JS expression);`

Here, we choose the second option, and the code should look roughly like this:

```css

iframe{v:expression(JS expression);}

```

The next question is how to destroy all iframe objects on the webpage. The principle of achieving this using JS is as follows: make the request address inside the iframe blank (`about:blank`), then remove the iframe object from the DOM (Document Object Model) to cut off all requests inside the iframe. There are several ways to remove DOM nodes; here I'll use the `outerHTML` property. Below is the CSS code:

```css

iframe{v:expression(this.src='about:blank',this.outerHTML='');}

```

Explanation: The preceding `v` is a self-defined CSS attribute by Ling'er. Here, `this` represents all iframe objects that will be described in terms of appearance. The comma between the two expressions means both lines of code will be executed simultaneously without any precedence order — this is a powerful guarantee. `about:blank` represents a blank page, which everyone knows. `outerHTML` is the DOM object containing its own HTML code, while `innerHTML` is the HTML code contained within the DOM object (excluding itself).

Great, the code is done. Let's test it and see if it works.

First, create a new webpage and insert the above CSS code (or add it to your existing CSS code):

```html

iframe{v:expression(this.src='about:blank',this.outerHTML='');}

```

Then, insert a few IFRAME codes into this page, assuming they are maliciously injected pages. Code as follows:

```html

Baidu

126 Mailbox

NetEase

```

Save it as `noiframe.htm`, and open it in the browser for testing (for local testing, enable the script warning bar at the top). Here, I use a packet capture tool for testing, but there's no need to use such a tool. A simple and effective method is to open the IE cache folder, clear it, refresh the page, and check if there are any files from these three websites in the cache folder. If there aren't any, it means no request results were returned — the test result is satisfactory. A slight smile appears on my face ^^*. At this moment, a colleague hands me a pancake, which tastes pretty good.

Tip: The cache folder location for Windows XP SP2 is `C:\Documents and Settings\Administrator\Local Settings\Temporary Internet Files`.
