Composition of Android Applications

by dream2012 on 2012-02-28 10:17:07

Android applications are generally constructed from four types of components:

- Activities

- Broadcast Receivers

- Services

- Content Providers

It's important to note that not every Android application needs to include all four components. Some may consist of a combination of these components. Once you determine which components your application requires, you should list them in the **AndroidManifest.xml** file. This is an XML configuration file used to define the components required by the application, their functionalities, and necessary conditions. This file is mandatory. For more details, refer to the [Android manifest file documentation](https://developer.android.com/guide/topics/manifest/manifest-intro).

Below is a detailed explanation of the four components:

### - Activities

An activity is the most basic component of an Android application. In an application, an activity typically represents a single screen. Each activity is implemented as a separate class and inherits from the base Activity class. The activity class displays a user interface composed of view controls and responds to events. Most applications consist of multiple screens. For example, a text messaging app might have one screen displaying a list of contacts to send messages to, another screen for composing text messages and selecting recipients, and yet another screen for viewing message history or performing message settings. Each of these screens is an activity, and it is easy to transition from one screen to a new one to perform a new activity.

In some cases, the current screen may need to provide a return value to the previous screen—such as allowing the user to select a photo from the phone to use as the caller's avatar in the contact list. When opening a new screen, the previous screen is paused and pushed onto the back stack. Users can navigate back to previously opened screens using the back button. Optionally, we can remove some unnecessary screens since Android retains each program launched from the desktop in the stack.

### - Intents and Intent Filters

To switch between screens, the Android-specific class **Intent** is used. An intent describes what the application wants to do. The two most important parts of the Intent data structure are the action and the associated data. Typical action types include MAIN (the entry point of the activity), VIEW, PICK, EDIT, etc. The associated data is represented in URI format. For example, to view a person's contact information, you would create an intent with the VIEW action type and a URI representing that person.

A related class called **IntentFilter** is used to describe which intents an activity (or BroadcastReceiver, see below) can handle when an intent is requested to perform an action. If an activity is intended to display a person's contact information, it must declare an IntentFilter that knows how to handle the VIEW action and URIs representing a person. IntentFilters must be defined in the **AndroidManifest.xml** file.

By parsing various intents, switching between screens becomes straightforward. When navigating forward, the activity will call the **startActivity(myIntent)** method. Then, the system will search through all the IntentFilters defined by installed applications to find the activity that best matches **myIntent**. The new activity receives the notification of **myIntent** and begins running. When the start activity method is called, it triggers the parsing of **myIntent**, providing two key benefits:

1. Activities can reuse requests generated by other components in the form of intents.

2. Activities can be replaced at any time by a new activity with the same IntentFilter.

### - Broadcast Receivers

You can use **BroadcastReceiver** to make your application respond to external events, such as incoming calls, data network availability, or when it gets dark. BroadcastReceivers cannot display a UI; they can only notify users of interesting events through the **NotificationManager**. BroadcastReceivers can be registered either in the **AndroidManifest.xml** file or programmatically using **Context.registerReceiver()**. When these interesting events occur, your application does not need to explicitly call BroadcastReceivers. The system will launch your application when needed and trigger the BroadcastReceivers if necessary. Applications can also broadcast their own intents to other mobile applications using **Context.sendBroadcast()**.

### - Services

A service is a program with a longer lifecycle that does not have a user interface. A good example is a media player playing songs from a playlist. In a media player application, there may be multiple activities that allow users to select songs and play music. However, the functionality of playing music does not correspond to an activity because users naturally expect the music to continue playing while navigating to other screens. In this case, the media player activity would use **Context.startService()** to start a service, keeping the music playing in the background. Meanwhile, the system will keep this service running until it completes its task. Additionally, we can connect to a service using the **Context.bindService()** method (starting it if it isn't already running). Once connected, we can communicate with the service via the interfaces it provides. Using the media player example, we could also perform actions like pausing or replaying the music.

### - Content Providers

Applications can store their data in files, SQLite databases, or even any valid device storage. When you want to share your application's data with other applications, content providers become very useful. A content provider class implements a set of standard methods that allow other applications to save or read various data types handled by the content provider.
