After enabling the Remi repository, deploying LNMP on the server is no longer a headache.

by ownlinux on 2011-07-30 13:45:02

Every time a new server is deployed, it feels like a tedious and physically demanding task. You need to download a bunch of software required for deploying the LNMP environment and other dependencies. Not only does this take a lot of time, but one small oversight could leave some hidden issues (compilation errors that are hard to detect). Of course, experienced users have their own methods—writing an automatic compilation and deployment shell script. But what about beginners? Should they just copy the scripts from experienced users? It might work when you copy it, but what happens when the environment changes? It’s entirely possible that this could lead to some errors, and if you can’t even read their scripts, how will you modify them according to your needs? Troubleshooting won't even know where to start, let alone the fact that the compilation process itself takes a lot of time, especially if the network isn’t ideal, the hardware configuration is average, or you're using a VPS. Imagine trying to compile MySQL on a VPS with 256MB of memory—doesn’t that sound painful?

By the time you get here, you might say that compiling provides more customization and flexibility. But ask yourself: are you familiar with all the compilation parameters? If you haven’t fully understood each parameter, how can you customize effectively? If you make a few customizations that introduce potential errors, that would be a disaster. Let me tell you, the advantage of software in the repository is that these programs have been precompiled by Linux experts and placed in the repository. Their familiarity with the system environment shouldn't be questioned, right? Moreover, software installed via the repository is very easy to upgrade, and adding/removing modules and extensions is also convenient. I used to be obsessed with compiling everything, but now I realize that what's available in the repository is almost the same as what I compile myself, so I no longer want to go through all that trouble. After all, yum really saves a lot of effort. Of course, it doesn’t mean that after using yum, you don’t need to do anything else; you still need to flexibly customize configuration files and extensions based on your own needs.

In Red Hat-based Linux systems, there is a repository called "remi" which contains almost all the latest versions of software, and upgrades are very convenient. Before enabling remi, you need to enable the epel source first. According to the official Fedora WIKI, EPEL (Extra Packages for Enterprise Linux) is a Fedora Special Interest Group that creates, maintains, and manages a high-quality set of additional packages for Enterprise Linux, including, but not limited to, Red Hat Enterprise Linux (RHEL), CentOS, and Scientific Linux (SL). Once successfully enabled, your RHEL can also use yum to install various software—isn't that exciting?

Run the following commands according to your version to install:

**For RHEL 6.x/CentOS 6.x/Scientific Linux 6.x:**

```

rpm -ivh http://rpms.famillecollet.com/enterprise/remi-release-6.rpm

rpm -ivh http://download.fedora.redhat.com/pub/epel/6/i386/epel-release-6-5.noarch.rpm

```

**For RHEL 5.x/CentOS 5.x/Scientific Linux 5.x:**

```

rpm -ivh http://download.fedora.redhat.com/pub/epel/5/i386/epel-release-5-4.noarch.rpm

rpm -ivh http://rpms.famillecollet.com/enterprise/remi-release-5.rpm

```

After completing these steps, you will see relevant files in the `/etc/yum.repos.d` directory.
