Upgrading the Development Environment and Applications to the SDK Guide

by dream2012 on 2012-03-01 10:59:22

This guide will help you upgrade your development environment and applications to the latest version of the SDK. If you have already applied a previous version of the Android SDK, this guide is also applicable to you. To ensure that your application is compatible with the Android 1.0 system, you need to install the new SDK and migrate existing Android mobile applications using the new API. The following sections will guide you through this process.

### I. Installing the New SDK

**Download the SDK** and extract it to a secure location. After extracting the new SDK, you should perform the following steps:

1. **Erase your emulator's data**

Since the new SDK version has been released, some data formats have changed. Therefore, any previously saved emulator data must be cleared. Open a console/terminal and operate in the `/tools` directory of the SDK.

- Windows: `emulator -wipe-data`

- Mac/Linux: `./emulator -wipe-data`

2. **Update your PATH variable (Mac/Linux; optional)**

If you previously set the PATH variable to point to the SDK tools directory, you must update it to point to the new SDK. E.g., in the `.bashrc` or `.bash_profile` file:

```bash

export PATH=$PATH:/tools

```

### II. Updating the ADT Eclipse Plugin

If you are developing using the ADT plugin in Eclipse, follow these steps to install the new plugin matching the new SDK.

- **Eclipse 3.3 (Europa)**

- **Eclipse 3.4 (Ganymede)**

**Steps:**

1. Select `Help > Software Updates > Find and Install...`

2. Select `Search for updates of the currently installed features` and click `Finish`.

3. If any ADT updates are available, select and install them.

4. Restart Eclipse.

Alternatively,

1. Select `Help > Software Updates...`

2. Select the `Installed Software` tab.

3. Click `Update...`

4. If any ADT updates are available, select and click `Finish`.

5. Restart Eclipse.

After restarting, update your Eclipse settings to point to the SDK directory:

1. Select `Window > Preferences...` to open the Preferences panel. (Mac OSX: `Eclipse > Preferences`)

2. Select `Android` from the left panel.

3. For the SDK location in the main panel, click `Browse...` and locate the SDK directory.

4. Click `Apply`, then `OK`.

### III. Signing Your Application

All applications must be signed before they can be installed. The ADT plugin and ant-based development tools support this requirement by signing compiled APKs with a debug key. To do this, the build tools use Keytool, included in the JDK, to create a keystore and a key with a known alias and password.

For more details, refer to **Signing Your Application**.

To support signing, first confirm that Keytool is valid for the SDK build tools. In most cases, you can tell the SDK build tools how to find Keytool by setting your `JAVA_HOME` environment variable to an appropriate JDK. Alternatively, you can add the Keytool version from the JDK to your `PATH` variable.

If you are developing on a Linux version that originally used GNU Java compilers, ensure that the system is using the Keytool version from the JDK rather than gcj. If Keytool is already in your path, it might be pointing to a symbolic link at `/usr/bin/keytool`. In this case, check the target of the symbolic link to ensure it points to the correct Keytool.

If you are using ant to compile your `.apk` files instead of ADT, you must regenerate your `build.xml` file. To do this, follow these steps:

1. In your Android application project directory, find and delete the current `build.xml` file.

2. Run `activitycreator`, directly outputting to the folder containing your application project:

```bash

activitycreator --out your.activity.YourActivity

```

Running this way, `activityCreator` will not erase or create new Java files (or manifest files) for those activities and packages that already exist. It is important that the package and activity actually exist. This tool creates a new `build.xml` file as well as a new directory called `libs` where third-party jar files will be placed, enabling you to handle them automatically with the ant script.

### IV. Porting Your Application

After updating your SDK, you may encounter broken code due to changes in the framework and API. You need to update your code to match the changes in the Android API.

One method is to open your project in Eclipse and view the errors marked by ADT in your application. From here, you can look up corresponding changes in the **Changes Preview** and **API Changes Report**.

If you encounter other difficulties while updating your code, visit the **Android Group Discussions** for help or seek assistance from other Android mobile application developers.

If you have modified the ApiDemos application and wish to port it to the new SDK, note that you will need to uninstall the pre-installed version of ApiDemos on the emulator.
