Tips and tricks about Android errors

by dream2012 on 2012-03-05 13:03:09

1. **ADT Installation Error: "requires plug-in org.eclipse.wst.sse.ui"**

The "Android Editor" component of the ADT plugin requires specific Eclipse components, such as WST (Web Standard Tools). If you encounter this error during ADT installation, you need to first install the required Eclipse components and then attempt to reinstall ADT. The simplest way to install the necessary Eclipse components for the "Android Editor" component of the ADT plugin is as follows:

- From the dialog box where you select Update sites to visit, check both the ADT site and the Callisto/Europa/Ganymede Discovery Site checkboxes (you may also need to check the Automatically select mirrors option below). Click Finish.

- In the Next dialog box, select Android Plugins. Now, expand the node item for the site; if you don't do this, the site's content will not load. After loading successfully, click Select Required. This will select all the necessary components for installing the Android plugin (wts, emf, etc.). Click Next, accept the license agreement, click Install All, and restart after completion.

2. **ADB Reports "No Device" While Emulator is Running**

Try restarting ADB: stop ADB first (`adb kill-server`), then use another ADB command to restart it.

3. **My New Application/Activity Doesn't Appear in the Device App List**

Before a new activity appears in the phone application list, you often need to restart your device or emulator. This is especially true when it is a completely new application with a new `AndroidManifest.xml` file. If it is a new activity within an existing `AndroidManifest.xml`, have you referenced the correct tags (or defined a service using tags, defined a receiver, etc.)? Ensure your `AndroidManifest.xml` is valid; errors often occur in attribute values, such as when the compiler does not receive the property value within the `>` because the intent filter does not match, causing your application to not display correctly. Have you sent your `.apk` file to the device (`adb install`)? Run `logcat` on your device (`adb logcat`) and install the `.apk` file. Check the output of `logcat` to see if the application is installed and correctly recognized. Below is an example of successful installation output:

```

I/FileObserver( 414): *** onEvent wfd: 3 mask: 8 path: MyRSSReader.apk

D/PackageManager( 414): Scanning package: /data/app/MyRSSReader.apk

D/PackageManager( 414): Adding package com.example.codelab.rssexample

D/PackageManager( 414): Registered content provider: my_rss_item, className = com.example.codelab.rssexample.RssContentProvider, isSyncable = false

D/PackageManager( 414): Providers: com.example.codelab.rssexample.RssContentProvider

D/PackageManager( 414): Activities: com.example.codelab.rssexample.MyRssReader com.example.codelab.rssexample.MyRssReader2

```

If `logcat` shows that the package manager encountered an error while loading the manifest file, add a space in your manifest file to force it to recompile.

4. **I've Updated My UI, But It Doesn't Seem to Reflect on the Device**

Did you remember to send your `.apk` file to the device (`adb install`)?

5. **When Starting an Application, You Get the Error "Binary XML file line #2: You must supply a layout_width attribute"**

Ensure the SDK version used for compilation matches the Android OS version currently running. Ensure you called `setContentView()` before other methods in your `onCreate()` method; calling methods like `setListAdapter()` before `setContentView()` can cause Android to try accessing screen elements that haven't been set up yet, leading to odd errors.

6. **My Request (Make a call, Receive an SMS, Receive a Notification, Send an Intent to Android App) Is Ignored**

You might not have the permission to call this activity or receive this intent (or possibly the requested permission has not been granted). Many standard Android activities, such as making a call, require permissions to prevent arbitrary applications from sending or receiving requests. Refer to "Security and Permissions" for more information about permissions and `manifest.permission` for a list of restricted Android permissions.

7. **Help! My Project Won't Compile in Eclipse**

If your project won't compile, look for symptoms such as newly added resources in the `res/` subdirectory not showing up in the R class, the emulator not starting, inability to run the application, or even running an old version of the application. To resolve these issues, try the following steps:

- Switch to the DDMS view in Eclipse (if it isn't already open): In the menu bar, select Window > Open Perspective > Other, choose DDMS from the list, and click OK.

- In the Devices panel (default top-right panel), click the downward triangle icon to bring up the panel menu.

- Select Reset ADB from the menu and try running your application again.

If the problem persists, check the Console and Problems tabs at the bottom of the Eclipse UI. If there are issues shown in these tabs, they should provide some clues about where the error occurred. If you're unsure whether the issues are new or old, right-click in the Console and select Clear to clear the console, then clean the project. Clean the project (a good way to handle some compilation errors) by selecting Project > Clean from the main Eclipse menu and selecting the project you're currently running (or clean all projects).

8. **Eclipse Cannot Interact with the Emulator**

When it seems there is no communication between Eclipse and the emulator, common scenarios include no response after pressing Run, the emulator hanging waiting for a debug connection, or Eclipse reporting that it cannot find the emulator or shell. The most frequent issue occurs when you press Run, the emulator starts (or is already running), but the application doesn't start. You'll likely find the following steps help fix the problem. Through practice, you may determine which step resolves your specific issue, but for now, the safest approach is to follow the sequence below:

- If the emulator is running, exit it first.

- Ensure all emulator processes are killed (they may sometimes hang; use the `ps` command on Unix/Mac or the Process tab in Task Manager on Windows).

- Exit Eclipse.

- In the command line, type: `adb kill-server`.

- Start Eclipse and attempt to connect to the emulator.

9. **When Entering Preferences in Eclipse and Selecting "Android," You Get the Following Error: Unsupported major.minor version 49.0**

This error occurs if you're using an older JDK version. Ensure you're using JDK 5 or JDK 6.

10. **Due to Signature Errors, I Can't Install the ApiDemos Application in My IDE**

Android requires all applications to be signed, as described in signing your application. The ApiDemos application is included in the SDK and pre-installed on the emulator, so it has already been compiled and signed with a private key. If you want to run or modify any ApiDemo apps in Eclipse/ADT or another IDE, simply uninstall the pre-installed app version via the emulator. If you attempt to run ApiDemos apps in your IDE without uninstalling the pre-installed version, you'll get an error message similar to the following:

```

[2008-08-13 15:14:15 - ApiDemos] Re-installation failed due to different application signatures.

[2008-08-13 15:14:15 - ApiDemos] You must perform a full uninstall of the application. WARNING: ...This will remove the application data!

[2008-08-13 15:14:15 - ApiDemos] Please execute 'adb uninstall com.android.samples' in a shell.

```

This error occurs because you're trying to install a copy of ApiDemos on the emulator with a different certificate signature (Android IDE tools sign the app with a debug certificate, while the existing version was signed with a private certificate). The system does not allow such reinstallation. To resolve this, fully uninstall the pre-installed version and then reinstall it using the adb tool. Follow these steps:

- In the terminal, change to the SDK tools directory.

- If no emulator instance is running, start the emulator using the `emulator &` command.

- Uninstall the pre-installed app using the `adb uninstall com.android.samples` command.

- Reinstall the app using the `adb install` command.

- If working in Eclipse/ADT, you can compile and run the app normally.

Note: If you have multiple emulator instances running, specify the emulator instance in your uninstall/install commands by adding `-s`, e.g., `adb -s emulator-5556 install`.

11. **Unable to Compile My Application Due to Expired Debug Signing Certificate Generated by Build Tools**

If your development machine uses a regional setting that does not include a Gregorian calendar, you may encounter many issues the first time you compile and run your program. Specifically, you may find that the Android build tools cannot compile your application because the debug key has expired. This happens because the key and certificate management tools included in the JDK, which Android build tools use internally, fail to properly handle non-Gregorian regional settings and create the correct date. More specifically, the generated debug key may have expired, resulting in a compilation error. If you encounter this issue, follow these steps to resolve it:

- First, delete the debug password-protected file/key (keystore/key) generated by the Android build tools. Specifically, delete the `debug.keystore` file. On Linux/Mac OSX, the file is located in `~/.android`. On Windows XP, it is located in `C:\Documents and Settings\\Local Settings\Application Data\Android`. On Windows Vista, it is located in `C:\Users\\AppData\Local\Android`.

- Then, choose one of the following options:

- Temporarily change your development machine's regional settings (date and time) to one that uses the Gregorian calendar, such as the United States. Once the regional setting is changed, use the Android build tools to compile and install your application. The build tools will regenerate a new password-protected file and debug key with an invalid date. Once the new debug key is generated, reset the regional settings on your development machine to the original state.

- Alternatively, if you don't want to change your machine's regional settings, generate the password-protected file/key (keystore/key) on another machine that uses the Gregorian calendar, then copy the `debug.keystore` file from that computer to the appropriate location on your development machine. This issue has been verified on Windows and may apply to other platforms.

12. **Cannot Run JUnit Test Classes in Eclipse/ADT**

If you use Eclipse/ADT for development, you can add JUnit test classes to your mobile application. However, when you attempt to run a class as a JUnit test class, you may encounter the following error:

```

Error occurred during initialization of VM

java/lang/NoClassDefFoundError: java/lang/ref/FinalReference

```

This error occurs because `android.jar` contains only stub classes and does not include the full implementation of JUnit.* classes. To add a JUnit class, you must create a JUnit configuration:

- In the Package Explorer view, select your project.

- Open the Run Configuration Manager. In Eclipse 3.3 (Europa), select Run > Open Run Dialog... or Run > Open Debug Dialog.... In Eclipse 3.4 (Ganymede), select Run > Run Configurations... or Run > Debug Configurations....

- In the Configuration Manager, right-click the "JUnit" configuration type and select New.

- In the Test tab of the new configuration, specify the project and test class, along with other options for running the tests.

- In the Classpath tab of the new configuration, find "Android Library" under Bootstrap Entries and delete it.

- Still in the Classpath tab, select Bootstrap Entries and click the Advanced button. Choose Add Library, click OK. Select JRE System Library, click Next. Select Workspace Default JRE, click Finish.

- Again select Bootstrap Entries, click Advanced. Choose Add Library, click OK. Select JUnit 3, click Finish.

After configuring in this manner, your JUnit test class should now run correctly.
