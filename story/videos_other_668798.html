Four Criteria for Evaluating Virtual Hosting Servers

by yyy852456 on 2012-02-18 15:02:18

How many virtual hosts can you fit on a single host? This is a common question that IT personnel face when considering what hardware to purchase for virtual hosts. In this article, virtualization expert Eric Siebert will explain how to select the right server for different types of virtual machines and how to meet the current and future needs of your virtual environment.

You may be able to fit over 100 virtual machines on a single host or just two. The type of application running on the virtual machine will greatly determine how many virtual machines can reside on a host server. For example, servers with minimal resource requirements, such as web, file, and print servers, can run more virtual machines compared to resource-intensive servers like SQL and Exchange servers. Generally, we should analyze the current environment's performance usage to better understand the demands of the virtual environment.

Four criteria for evaluating host servers

When deciding on server hardware size, there are four main criteria: memory, CPU, network, and disk resources. Let’s start with memory, which tends to get consumed first on host servers.

Memory: When it comes to calculating how much RAM a host server should have, I recommend having as much as possible. However, when it comes to allocating memory to virtual machines, the situation is completely opposite—less is more; only give the virtual machine the memory it needs. Typically, physical servers allocated with more memory end up wasting it. For virtual machines, increasing RAM at any time is simple, so initially allocate only a small amount of memory that suffices, then increase as demand grows. It's easy to allocate more physical memory than the host actually possesses, leading to risks of virtual machine swapping to disk when memory runs out, which can degrade performance.

CPU: With the advent of multi-core CPUs, adding more CPUs to host servers has become easier and less expensive. Nowadays, almost all servers come with two or four cores per physical CPU. A general rule is that each CPU core can support four single-CPU virtual machines. This number varies based on the average CPU usage of the applications running on the virtual machines, potentially ranging from 12 per core to a maximum of 8-10 virtual machines per core.

There is a common misconception about virtual servers: that virtual machines can use the total sum of CPU megahertz. For example, a server with four dual-core 2.6GHz CPUs has a total of 20,800 MHz (8 x 2.6 GHz) of resources. However, a virtual machine with one virtual CPU cannot use more than the megahertz available from a single CPU or core. If a virtual machine has two virtual CPUs, it cannot use more than the maximum megahertz of each CPU or core. The number of cores required also depends on whether multiple virtual CPUs are used by virtual machines.

Typically, you should have at least one more core than the maximum number of virtual CPUs allocated to a single virtual machine. For instance, if you plan to run a virtual machine with four virtual CPUs, avoid purchasing a dual-core, dual-processor server because such a server only has four cores. This is because the hypervisor's CPU resource scheduler must find four idle cores simultaneously every time the virtual machine makes a CPU request. If there are only four cores available in total, performance will suffer. I recommend using quad-core CPUs, as they provide more flexibility for CPU scheduling.

Network: The number of network interface cards (NICs) required for virtual servers varies based on redundancy needs, whether network storage is required, and the features selected. Using 802.1Q VLAN tagging provides the flexibility to use multiple virtual LANs (VLANs) on a single NIC, thus eliminating the need for a separate NIC for each VLAN on the host server. For smaller servers, two NICs might suffice, but it's best to have at least four NICs on the host server. If you use network storage like iSCSI, it's preferable to have more than four NICs, especially if you plan to use VMware's vMotion feature. When creating switches, it's best to assign multiple NICs to them for redundancy and to increase available capacity for virtual machines.

Disk: The final aspect to evaluate is disk resources. There are several options to consider, depending on your budget and whether your environment includes a Storage Area Network (SAN). Using local disks is the cheapest option but does not offer the advanced features of shared storage between virtual host servers, such as vMotion. SAN (Fibre Channel) disks generally provide the best-performing disk solution but are usually the most expensive. Network disks are a good alternative, offering performance close to that of SANs. Similarly, using 15K hard drives offers better performance than using 10K hard drives, but having larger RAID groups is also important, as it helps spread disk I/O across drive spindles as much as possible.

When deciding how many disks to purchase, ensure that all virtual machines have sufficient disk resources available, and add an extra 10%-20% disk space for additional virtual machine files and snapshots. If you plan to use snapshots frequently, you may need even more disk space. In most cases, disk resources are combined with the host, such as storing development and testing virtual machines on local disks and production virtual machines on shared storage.

In general, you want your virtual machines to use at least 80% of the virtual host server's capacity to maximize your investment. However, leave enough capacity for future growth and ensure there are sufficient resources to support additional virtual machines in case of host failure. More capacity is always better, as it avoids resource contention and the need to purchase additional virtual host servers.
