So how big was your SYN Flood?

by xloong on 2007-12-08 17:24:20

Team: http://www.ph4nt0m.org

Author: Yun Shu (http://www.icylife.net)

Date: 2007-12-07

This is an interesting article, but it does not discuss the principles of SYN Flood attacks or describe solutions for defending against such attacks. Here, I will casually talk about a few details that are often hidden by device manufacturers, either intentionally or unintentionally. If you are considering purchasing equipment to defend against attacks, I hope this article can help you—least of all, when negotiating with manufacturers, you'll know what questions to ask and won't be easily misled.

### 1. Ethernet Padding

In current SYN Flood programs, in order to improve sending efficiency, optional fields in the packet headers are not filled out. Therefore, the IP header length is exactly 20 bytes (Byte), as shown below:

```c

typedef struct ip_hdr

{

unsigned char h_verlen;

unsigned char tos;

unsigned short total_len;

unsigned short ident;

unsigned short frag_and_flags;

unsigned char ttl;

unsigned char proto;

unsigned short checksum;

unsigned int sourceIP;

unsigned int destIP;

}IP_HEADER;

```

Similarly, the TCP header is also 20 bytes long, as shown below:

```c

typedef struct tcp_hdr

{

unsigned short th_sport;

unsigned short th_dport;

unsigned int th_seq;

unsigned int th_ack;

unsigned char th_lenres;

unsigned char th_flag;

unsigned short th_win;

unsigned short th_sum;

unsigned short th_urp;

}TCP_HEADER;

```

That means the attack program fills a 40-byte data packet and starts sending it. However, for Ethernet, the minimum packet length requires the data segment to be at least 46 bytes. Since our packet is only 40 bytes, the network card will add some processing during transmission. The end of our 40-byte IP header plus TCP header packet will be padded with six 0s by the network card to meet the minimum packet length requirement. At this point, the entire data packet consists of a 14-byte Ethernet header, a 20-byte IP header, a 20-byte TCP header, and six 0 bytes added due to the minimum packet length requirement, totaling 60 bytes. But this is still not complete because Ethernet transmission also requires CRC checking. Before sending the data, the network card will perform a CRC check on the data packet and append a 4-byte CRC value to the end of the packet. At this point, the length of our data packet is no longer 40 bytes but has become 64 bytes. The structure of the data packet is as follows:

| 14-byte Ethernet Header | 20-byte IP Header | 20-byte TCP Header | 6-byte Padding | 4-byte Checksum |

| Destination MAC Address | Source MAC Address | Upper-layer Protocol Type | IP Header | TCP Header | Ethernet Padding | CRC Check |

After all these steps are completed, the network card begins sending the data packet. Unfortunately, if you use general packet capture tools like TcpDump or WireShark on the attacking side, you cannot see this process. This process can only be verified by capturing packets on the attacked host. However, using similar tools on the attacked end clearly shows that the SYN packet containing only the IP header and TCP header sent from the attacker will have six additional 0-filled bytes appended when it reaches the attacked machine. This is the Ethernet minimum packet length padding mentioned earlier. Regrettably, the 4-byte CRC check is discarded after the network card receives and checks the data packet, so it cannot be captured using common tools on the attacked end.

In other words, the 40-byte message you send is no longer 40 bytes when it reaches the attacked machine's network card—it becomes 64 bytes.

### 2. Ethernet Transmission

We just discussed the padding operations performed by the network card when sending packets, but there's more. Let's take a look at the transmission process of the attack packet over the network. Since the attack packet is very small, far less than the maximum transmission unit (MTU) of 1500 bytes, it will not be fragmented. Are these packets transmitted tightly packed together like cans on a production line? In fact, they are not.

Ethernet transmission also involves a preamble and inter-frame gap. The preamble occupies 8 bytes (bits), consisting of 64 bits. The first seven bytes are all 10101010, alternating between 1 and 0. The eighth byte changes to 10101011. When the host detects two consecutive 1s, it knows that the data begins next, starting with the destination MAC address, etc., which are parsed sequentially. In other words, the preamble tells the host to prepare for receiving data. The inter-frame gap is the space between Ethernet packets, which is 12 bytes (96 bits). Therefore, the structure of the data during network transmission is as follows:

| 8-byte Preamble | 6-byte Destination MAC Address | 6-byte Source MAC Address | 2-byte Upper-layer Protocol Type | 20-byte IP Header | 20-byte TCP Header | 6-byte Ethernet Padding | 4-byte CRC Check | 12-byte Inter-frame Gap |

That means a single SYN packet consumes 84 bytes of bandwidth during network transmission.

### 3. 100Mbps Line Speed or 1Gbps Line Speed

With the above foundation, we can now start discussing the line speed issues of network security devices. These devices could be Anti-DDOS devices, IDS, IPS, etc. When the smallest SYN packet with only the IP header and TCP header is transmitted over an Ethernet network, the maximum PPS supported by a 100Mbit network is 100*10^6 / (8 * (64 + 8 + 12)) = 148809 pps, and for a 1000Mbit network, the maximum PPS is 1488090 pps.

When manufacturers promote their security devices, do their numbers align with these figures? How is the maximum traffic supported by their IDS calculated? What content is included? Is the claimed line speed of their Anti-DDOS devices consistent with the actual situation? Can they really defend against attacks corresponding to the stated PPS? When you tell others that your IDS indicates you suffered an N Gbit SYN Flood attack with PPS reaching N million, were you really under a large-scale attack?
