How will HTML5 develop next? A senior chief engineer at Intel discusses the future of HTML5.

by anonymous on 2013-11-17 18:34:39

Computer programmers have long been plagued by cross-platform issues, ever since the need arose to involve a second platform. Since then, the number of problems has rapidly increased. Today, developers must at least target four operating systems (and more specific ones) simultaneously. Devices running these operating systems come in various appearances, sizes, resolutions, durability levels, input methods, carrier networks, connection speeds and states, user interface regulations, application stores, deployment, and update mechanisms.

Many global developers once regarded Java as the preferred choice for cross-platform development. Indeed, Sun (now Oracle)’s structured language has continuously addressed many cross-platform issues, but it also introduces numerous obstacles, one of the most significant being that class structures can impose heavy burdens even on the smallest program functionalities. When developers shifted to using browsers for application delivery, the burden brought by Java became heavier; Java applets are opaque black boxes to browsers because the language is closed to developers (despite our great respect for JCP).

As Java intensified the browser wars, a similarly named parsing language emerged. This language initially called Mocha, later LiveScript, and finally JavaScript, proved to be more practical in some aspects than Java because it could interact with browsers and use HTML's Cascading Style Sheets (CSS) to control content display. JavaScript support quickly became a standard feature for every browser. Now, it has become the HTML5 programming language, which the World Wide Web Consortium considers the next standard for markup languages.

To better understand the reasons for HTML5’s success, its current state, and its future direction, Intel Software Adrenaline magazine interviewed Senior Principal Engineer Moh Haghighat from Intel's Software and Services Division in the Software Development Products Department. Moh was the technical lead for Intel's first JavaScript Just-In-Time (JIT) compiler developed for the Firefox browser. He also led the development of the first parallel JavaScript JIT and the prototype for a parallel browser layout engine, both developed within the Firefox environment. He currently leads Intel's HTML5 technology strategy efforts.

Intel Software Adrenaline: Why does HTML5 surpass Java for cross-platform development?

Moh Haghighat: Launching Java applets reminds me of the infamous "loading Java" status bar. Instead of seeing quick responsiveness like today's web browsers, you see lots of progress bars, which at least becomes annoying. This is a fundamental issue with the language, and [Adobe] Flash addressed this problem to some extent. Even running a small piece of Java code requires loading the entire JVM into the browser, and executing a large sequence of class initializations before starting the actual execution. For server-side code, this situation is acceptable when you run the same code repeatedly on the server for multiple clients. But on the client side, you want to run code from many different applications, and seeing so many progress bars is unbearable. The best place to run Java turned out to be middleware on the server side, exemplified by the excellent WebLogic, a pioneer in Java application servers. This is one of the technical reasons why HTML5 surpasses Java for client-side development. Nonetheless, Java played an important role in establishing the feasibility of manageable programming languages and laying the foundation for more efficient high-level languages such as JavaScript, PHP, Python, and Ruby.

Java also has some truly outstanding features that support efficient code generation with almost no expensive compiler analysis, such as type-based and offset-based ambiguity resolution for registers.

Another issue lies in how Sun managed the Java project. Java was not an open technology from the start, and web technologies evolved over time. I believe the main reason Java didn't succeed is that Sun ignored web browsers; during the rise of JavaScript, we saw little innovation from Java in the browser space. JavaScript's asynchronous usage model (AJAX) made the browser an excellent interactive platform, leading to the popularity of many web-based applications like Google Docs. At the same time, the complexity of these applications increased. For instance, Gmail's JavaScript code grew from nearly ten thousand lines in 2004 to nearly half a million lines in 2010, increasing fiftyfold in just six years (see Figure 1). With the rapid global growth of the web, millions of web pages now use JavaScript, making it the most influential language in terms of installed codebases. In terms of browser integration, Java cannot compare with JavaScript, as JavaScript is the browser's "native" language. Additionally, browsers using CSS and JavaScript achieve complete separation of UI and logic, something the Java platform lacks.

HTML5 offers a much more attractive alternative. Your applications can run on any type of computing device, whether it's a phone, tablet, laptop, desktop computer, or TV. If the device supports HTML5, your application will run on it. You don't even need to compile; you just write the program, and it gets distributed and executed.

ISA: JavaScript and HTML have existed for many years. What recent changes make HTML5 so effective for cross-platform development?

MH: First, in the past five years, JavaScript's speed has significantly improved, becoming up to 100 times faster (see Figure 2), largely due to the emergence of JavaScript JIT. Because of the significant increase in the complexity and maturity of web applications, JavaScript JIT has become an indispensable technology. In terms of JavaScript performance, the latest version of Internet Explorer in 2011 is over 100 times faster than the Internet Explorer of 2001. Second, through HTML5, a vast array of new features were suddenly introduced. This may be the most innovative period for HTML since the advent of browsers. Now, the way HTML renders has changed completely compared to the past. After HTML5 began to be widely adopted, its performance improved significantly, and new features continued to emerge.

In terms of technology adoption, I observed another noteworthy trend: IT departments increasingly find HTML5 appealing because people use various devices at work. In today's business environment, there are hundreds of different types of smartphones, tablets, and portable computers involved. HTML5 is the only solution that allows IT applications to run on all devices. Moreover, companies lack the resources to develop and support applications for each device type. Now, responsive design patterns based on HTML5 are emerging, allowing content to adapt to the screen size of the device. These are all important components of HTML because with CSS and JavaScript, you can dynamically adapt to different devices.

ISA: Due to the lack of static typing, JavaScript has often been criticized for being limited to small applications. Is this criticism justified?

MH: JavaScript will eventually provide classes, modules, and typing capabilities. The ECMAScript committee responsible for the next version of JavaScript is studying the concept of classes, which will help with maintenance. Java has classes, and C++ has classes. JavaScript uses a prototype-based inheritance model, which allows you to implement classes but does not enforce a specific way of doing so. Java has its own object pattern for classes; currently, the JavaScript language does not adopt a specific class pattern. It allows users to implement their own versions of classes.

For software engineering reasons, when a large number of developers are involved in a project, specific class patterns might be necessary. For large projects, you may need to impose some restrictions, such as defining specific coding principles. If everyone uses JavaScript in their own way, its flexibility could potentially have negative effects.

Let’s draw an analogy from earlier times when programming languages like FORTRAN used GOTO statements. In the 1970s, Edsgar Dijkstra and others discovered that using GOTO statements in an unstructured manner was harmful. Therefore, programmers were required to use structured programming principles instead of arbitrarily using GOTO statements, making programming languages and applications easier to understand and maintain. Similarly, as JavaScript evolves into a low-level language and enables the creation of many high-level languages, the same thing is happening. JavaScript has become what is known as the "assembly language of the web."

ISA: Can you give us an example of this scenario?

MH: Recently, the most exciting examples regarding constraints and structure are Mozilla's project named "asm.js" and a related project called "Emscripten," which is essentially a compiler that converts LLVM bitcode into JavaScript and HTML5 APIs. (LLVM is a project sponsored by Apple aimed at developing low-level virtual machine compiler infrastructure and language front-ends for C/C++ and other languages.) Emscripten essentially converts C++ into JavaScript.

Asm.js is an optional target for Emscripten that can provide near-native performance, with overhead typically reduced by two times compared to optimized native code, whereas current full JavaScript is about five times slower. Although JavaScript uses dynamic typing, asm.js requirements (such as type inference and annotations) remain entirely based on the JavaScript specification. Using asm.js, users can write code that is proven to be statically typed. Thus, at load time, users can verify that the code is indeed statically typed. Ahead-of-time compilers can generate extremely efficient code without additional checks, unlike dynamic typing languages, helping improve both flexibility and code efficiency.

As we said, these languages continue to evolve and will enable the conversion of large chunks of compute-intensive code (such as performance libraries and game engines) into JavaScript, making them usable on every device globally. JavaScript is becoming the target language for higher-level languages.

ISA: Are there any current language projects related to JavaScript?

MH: There is a language called CoffeeScript, and Microsoft is developing TypeScript, which is a superset of JavaScript. Google's Dart is a new language with class concepts that compiles to JavaScript. I don’t think anyone can convince the entire web to switch the primary language of web browsers. However, I believe that the excellent characteristics of various languages will influence the JavaScript specification, and these characteristics will be added to the specification. This basically achieves evolution, backward compatibility, and incremental change.

ISA: Intel directly drives the development of HTML5 and JavaScript. Could you tell us about the relevant work?

MH: Intel has been a leader in parallel technology, and parallelism makes HTML5 meet the requirements of multi-core processors. You need to leverage parallelism to enhance responsiveness and power utilization. We have parallelized CSS rule matching in the Firefox layout engine to make it scalable. From a programming language perspective, HTML5 has a parallel API called "Web Workers." It is very suitable for coarse-grained background threads, but if you want to execute a large number of small parallel tasks, Web Workers won't suffice. Intel has been collaborating with Mozilla on parallel arrays; colleagues in our Intel Labs first built its prototype. Currently, we are working on resolving all implementation issues to bring it into ECMAScript, the official language specification for JavaScript.

Additionally, we are collaborating with Mozilla and Google to ensure that existing SIMD vector functionality (such as Sandy Bridge and Ivy Bridge) can truly be programmed using JavaScript. That is, if your JavaScript code performs parallel operations on data, the code will actually be vectorized, ensuring that developers can use these functions effectively. These features will eventually enter the standards.

Conclusion

Faced with the growing number of operating systems and devices, many developers worldwide are looking for cost-effective ways to create their applications. Developers once viewed Java as a cross-platform development solution. While Java has many advantages compared to older languages, it has not kept pace with the rapid innovation of mobile platforms. Developers now turn to HTML5 and its core technologies, JavaScript and CSS, to achieve the latest features and cross-platform operability needed to succeed in today's application market.

According to Haghighat, Intel engineers continue to optimize HTML5 engines on Intel platforms and add new features to HTML5. Intel views HTML5 as an important step towards the era of transparent computing. To further accelerate the widespread adoption of HTML5, Intel now provides the Intel XDK, a complete development kit that helps developers build, test, debug, package, and deploy their HTML5 applications across various platforms such as Android, Firefox OS, and iOS.
