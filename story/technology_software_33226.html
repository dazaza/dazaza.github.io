"8ffe2740 and Server Application Error encountered in IIS 5.1"

by xloong on 2007-11-03 21:31:51

Today, in order to better study "Mastering Script Hacking" and test the upcoming z-blog 1.8, I installed IIS 5.1 on my local machine. However, I encountered a series of issues, so I'm summarizing them here for friends who may encounter similar problems in the future (this sounds awkward, as if I’m some kind of expert, but actually, I’m just a noob—sweatdrop~).

My machine is running the Professional Edition of XP, and the IIS Information Service was not hidden, which saved me the trouble of restoring it. The selection and installation process went relatively smoothly. However, after installation, I couldn’t access http://localhost. Checking the management console, I found that the default site was not started. When I tried to start it manually, an error message appeared:

IIS cannot start due to an unexpected error 8ffe2740.

Using Baidu, I searched for related content, and many people said that port 80 was occupied. Several users mentioned that web-based Xunlei (a download tool) had taken over this port. But I never use web-based Xunlei; that’s more of a “basic” product for ordinary users. I prefer the professional-grade ad-free version of Xunlei. Could it also be occupying port 80?

To check port usage, I entered "Start->Run", typed cmd, and opened a DOS window. Then, I ran the following command:

```

netstat -ano

```

Looking at the information for port 80, which should have been used by IIS, I found:

```

Proto Local Address Foreign Address State PID

TCP XXX.XXX.XXX.127:80 59.XX.X.39:1281 ESTABLISHED 1448

TCP XXX.XXX.XXX.127:80 59.XX.X.27:3700 ESTABLISHED 1448

```

The PID using port 80 was 1448.

Then, I opened the Task Manager, selected the "Processes" tab, clicked "View->Select Columns", and checked the "PID (Process Identifier)" option. This added a PID column to the "Processes" list. Sorting by PID from smallest to largest, I found that the image name corresponding to PID 1448 was indeed Xunlei—so it was Xunlei causing the problem again.

Since another program was occupying IIS's port 80, errors were bound to occur. Knowing the source of the issue makes it easier to resolve. Either change Xunlei's port or change IIS's port. Since IIS typically uses port 80, I decided to change Xunlei's port instead.

Opening the Xunlei interface, I clicked "Settings->BT/Port Settings" and saw that its TCP port setting was 80 (as shown in the figure below). I changed this port to an uncommon one or used the "Random Generate" option. I manually set it to 9999 and saved the changes.

After closing and reopening Xunlei to make the settings take effect, freeing up port 80 allowed IIS to start normally.

Unfortunately, this damn IIS kept giving me trouble. It could start, but it wouldn't display properly, showing the following prompt:

```

Server Application Error. The server has encountered an error while loading an application during the processing of your request. Please refer to the event log for more detailed information. Please contact the server administrator for assistance.

```

It was clearly taking advantage of my poor understanding of foreign languages. After further searching on Baidu, I found a solution:

1. Right-click "My Computer" -> Manage -> Local Users and Groups, and set passwords for the two users IUSR_computername and IWAM_computername. The passwords must be the same.

2. Start -> Run -> type cmd,

```

cd c:\Inetpub\AdminScripts

cscript.exe adsutil.vbs set w3svc/wamuserpass yourpassword

cscript.exe adsutil.vbs set w3svc/anonymoususerpass yourpassword

```

Check if it works now. If it still doesn't work, then run:

```

cscript.exe synciwam.vbs -v

iisreset

```

For the Server Application Error continuation, 8004EOOF error,

When running C:\Inetpub\AdminScripts\synciwam.vbs, an 8004EOOF error occurred. After searching online, I discovered it was an MSDTC issue: the msdtc service wasn't starting properly. Finding the cause made it easier to resolve ^_^.

**Step 1**

Delete the following registry keys:

```

HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\MSDTC

HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\MSDTC

HKEY_CLASSES_ROOT\CID

```

**Step 2**

Stop the MSDTC service:

```

net stop msdtc

```

**Step 3**

Uninstall the MSDTC service:

```

msdtc -uninstall

```

**Step 4**

Reinstall the MSDTC service:

```

msdtc -install

```

Then follow the original solution for the Server Application Error.

If it still doesn't work, uninstall IIS and reinstall it.

I continued until the final step—reinstalling IIS—and finally resolved the issue. Exhausted!

I hope this article helps anyone encountering similar problems. All this trouble with IIS wasn’t in vain (I know I’m not very skilled, so please don’t point it out if you already know~).
