Top 5 Surprises When Starting Out as a Software Developer

It has been over 20 years since I started working. Even so, I still remember the anticipation of graduating from university and starting my career. Before that, apart from a few summer jobs unrelated to programming, my life was mostly spent in school. Although most of my expectations about work were met, I also discovered many surprising aspects of this field during my early years in the workplace. Below are the top five:

5. Interpersonal Communication

Programming seems like a rather solitary job – you need to complete a feature, so you sit down and implement it with code. However, in reality, you will have quite a lot of interaction with your colleagues. You will discuss designs with them, review new features during meetings, and discuss your code with testers.

Learning to be tactful and flexible in communication is very helpful. If you haven’t yet mastered this skill, you should learn it. There’s a best-selling book on this topic – Carnegie's "How to Win Friends and Influence People." If you haven't read it, I strongly recommend that you do.

4. Writing Matters

Expressing ideas clearly through writing is very helpful. In a way, writing code is similar to writing, both requiring you to structure and express your thoughts clearly and unambiguously. Emails alone are endless, but there are also documents accompanying the product features you develop, bug reports that need clear descriptions, and explanations for the bugs you fix. Writing wasn’t emphasized in university, but if you're good at it, it can definitely be an asset in your work.

3. There Is No Such Thing As Finished Software

Before I started working, I thought that once I developed a feature, it was done. However, in reality, you often have to revisit that feature. Sometimes it's because it doesn't fully meet the client's requirements, or because you need to add more functionality, or because you want to merge similar functions, or even to fix a bug. In any case, continuing to work on your code is normal.

In university, we often built programs from scratch, but this rarely happens in the real world. Of course, unless you're developing a new feature, but these new codes must integrate into the old ones. As a result, a significant part of developing new features involves reading and understanding old code. This is a skill we didn’t practice much in school.

2. Few Elegant Algorithms

I obtained a master's degree in computer science and engineering, studying courses such as communication systems, including signal processing, error-correcting codes, queuing theory, etc. We also had core computer science courses like algorithms and data structures, which I enjoyed a lot. I thought it was cool to learn these clever algorithms and data structures, so I was really looking forward to seeing them in action at work.

My first job was as a software engineer at Ericsson in Montreal, working on a mobile telephone switching center for cellular networks. There was a large amount of code controlling call setup, hang-ups, roaming, etc., but when I saw that these functions were implemented using the most basic data structures and algorithms, I felt a bit disappointed. Interestingly, I found that the code used to track user roaming states consisted of over a thousand binary trees. The last three digits of a user's phone number determined which tree they belonged to. To find a user, you locate their tree based on the last three digits of their phone number, then traverse the tree to find them. Aside from this, everything else was linked lists or even simpler data structures.

1. Complexity After System Integration

Given the lack of sophisticated algorithms and the use of basic data structures throughout the application, one might think that working here wouldn’t be challenging. Wrong! I soon realized that this system was highly complex, not because of complicated functionalities, but because of the integration of too many simple functions.

In all the systems I’ve worked on, I’ve noticed a commonality: most features are very simple, but precisely because there are so many simple features, the subtle (or not-so-subtle) interactions encountered while integrating these features lead to bugs.
