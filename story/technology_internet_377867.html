How to Improve the Running Efficiency of Informix Database

by jimwang on 2011-01-13 11:05:36

Informix is a widely used relational database. As the amount of data in the database and the volume of application transactions increase, its operational efficiency becomes increasingly prominent. Therefore, how to improve the performance of the database is a topic of concern for administrators. The operational efficiency of the Informix database is directly related to the configuration of its system parameters (online), performance tuning, the way database tables are created, indexing strategies, and the quality of ESQL/C programs. Below, we will introduce optimization strategies and measures to improve the operational efficiency of the Informix database.

### 1. Optimization of Informix System Performance Parameters

System performance is related to shared memory, CPU, disk, and network.

- Shared memory is generally 1/3 to 1/4 of the system memory. One CPU VP (virtual processor) is configured with 4 LRU queues, and n LRU queues are configured with n page_cleaner processes. Tuning should ensure that the hit rate of high-speed buffer reads exceeds 95%, and the write hit rate exceeds 85%. Setting multiple network cards can improve performance, and the network collision rate should be within 8%.

- The principle of disk tuning is to reduce the number of disk reads, maximize the amount of data read per disk operation, evenly distribute data, and prevent bottlenecks from occurring. Online disk space should use raw devices rather than file systems. The former is much faster than the latter and more reliable.

- Physical log cache space should be between 30 to 50 MB; it doesn't need to be too large.

- The number of CPU VPs should be set to the number of CPUs minus 1 (if there is only one CPU, then the number of CPU VPs should also be 1).

- System kernel parameters should be adjusted according to the configuration requirements proposed by different versions of Informix. If the adjustment is incorrect, the online construction may fail. Debugging the performance of the Informix-online data server often requires dynamic repeated adjustments and tests based on certain empirical values to achieve satisfactory results. The quality of Informix database system parameter performance adjustments directly affects the operational efficiency of the Informix database. Specific methods and steps for performance tuning are detailed in Informix documentation, which we will not focus on here.

### 2. Optimization of Database Table Creation Methods

1. **Optimization of tablespace allocation for each database table**:

- Before creating a table, estimate the size of the data for each table so that the first "extent" (physically contiguous pages) space allocation matches the estimated value as closely as possible. The next "extent" space allocation should be determined based on the estimated growth of the table's data. This reduces data fragmentation and saves space, improving the efficiency of the database system.

2. **Table fragmentation**:

- Introduce table fragmentation to distribute data uniformly both physically and logically, which helps enhance parallel processing performance.

3. **Record locking method during table creation**:

- For batch processing tables, use page-level locking (page). For real-time transaction tables, use row-level locking (row). The locking method can be determined when creating the table or changed using the `alter tabname lock mode (row)` and `alter tabname lock mode (page)` commands. Note that tables generated through dbimport and dbexport default to page-level locking. For frequently accessed real-time transaction tables, you should switch to row-level locking. Use the `oncheck -pt` command to check the locking status of a table.

4. **Logging method for database creation**:

- (1) No logging: Cannot perform transaction processing.

- (2) Buffered log: Writes to disk when the shared buffer is full.

- (3) Unbuffered log: Writes to disk immediately after a transaction is completed.

- (4) ANSI mode: Only differs in date format (month/day/year), otherwise identical to unbuffered.

- Generally, for real-time processing systems, we use unbuffered logging. For bulk data loading/unloading, no logging is recommended.
