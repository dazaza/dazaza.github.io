JavaEE5 with Struts, Spring, and Hibernate

by anonymous on 2013-11-17 17:24:33

Because of the openness of Java technology and the many choices offered to developers by the JCP, as a Java developer or enthusiast, when working on project development, one inevitably faces numerous frameworks in the Java world: Struts, Spring, Hibernate/iBatis, or Servlets, EJBs, etc. — various open-source or standard frameworks defined by the JCP organization. Faced with so many frameworks, what should developers choose?

As we all know, before the official release of the Java EE 5 specification, many open-source frameworks were very famous, loved, and widely used, such as Struts, Spring, Hibernate, etc. These open-source works once became de facto standards for Java enterprise application development to a certain extent. However, in May 2006, with the official release of the Java EE 5 specification, and with the launch of numerous products or technologies from various vendors supporting the Java EE 5 specification, an inevitable competition arose between open-source and standards...

The emergence of Java EE 5 might be one of the most significant shocks since the birth of J2EE. It has been more than half a year since the specification was released, and the industry's attention to Java EE 5 is becoming increasingly intense. A Google search for the keyword "Java EE" yields over 5 million related results. From a search on Sun's website (http://java.sun.com/javaee/overview/compatibility.jsp), it can also be seen that professional vendors have quickly followed up. In addition to Sun itself, three other globally renowned companies, including SAP and Kingdee Apusic, have already launched application server products that fully support the Java EE 5 specification.

Java EE 5 includes new features such as JSF 1.2, EJB 3.0, and JAX-WS 2.0, attempting to address issues of simplicity, flexibility, and ease of use in Java enterprise application development. Before the appearance of Java EE 5, many open-source frameworks (Open Source Frameworks) emerged like mushrooms after rain, trying to solve application development problems not covered by "committee" specifications from certain angles or aspects, such as separation of concerns in web development (MVC), business model implementation (ORM), etc. Many open-source frameworks are very famous, loved, and widely used, such as Struts, Spring, Hibernate, etc. These "non-standard" works once became de facto standards for Java enterprise application development to a certain extent. Will the appearance of Java EE 5 change this situation? Or, when people reselect application frameworks, will they prioritize the entirely new Java EE 5 technology? With these doubts, the author attempts to make a simple technical comparison and offer some superficial comments, hoping to stimulate discussion and provide entertainment for the public.

MVC is a layered structure of Model, View, and Controller, which achieves separation of concerns through layering and reduces common repetitive work in traditional development. Both Struts and JSF support MVC. The core of Struts' support for MVC is the Controller, which obtains HTTP requests through a common entry Servlet, passes HTTP parameters into ActionForm, and then passes ActionForm to the Action class. Struts only has one event handler for HTTP requests; when a request comes in, the corresponding Action returns a result to the front-end Controller, which then selects how to navigate. JSF usually also uses a front-end Controller (some commercial JSFs can intelligently recognize JSF requests without additional controllers), but the work done in the Controller is completely different from Struts' Controller—it triggers events in JSF page components, generates component presentations using Render tools, and binds data from the Model to components, etc. It can be seen that JSF is more flexible in the control layer, supports transformations in the view layer using Render tools to generate flexible presentations, and achieves complete separation from the framework in the model layer, thus offering higher flexibility.

POJO stands for Plain Old Java Object. POJO combined with AOP is widely used for rapid business modeling. Struts was initially designed mainly to solve view and control problems, paying little attention to the flexibility of the model. The ActionForm model in Struts must inherit from framework classes, and although some helper classes can be provided through custom libraries to achieve model independence from the framework, it is essentially tightly coupled to JSP- and HTTP-centric methods. JSF provides excellent natural support for POJOs and supports rapid business model generation through RAD tools, thus offering higher productivity.

The greatest significance of page navigation is helping to realize visual page development and quickly define page flow on the UI interface. Page navigation is divided into static navigation and dynamic navigation. Static navigation directly flows from one page to another, while dynamic navigation determines the flow of pages based on specific actions or logic. Both Struts and JSF support static and dynamic page navigation. However, in Struts, navigation rules are bound to Actions, meaning that repeated navigation rule definitions may be required for different Actions on the same page, whereas JSF navigation can be independent of Actions and share the same navigation rules between different components and Actions within a page.

Struts itself does not provide a UI component mechanism, while JSF offers a complete UI component mechanism. Customizing and reusing UI components can greatly simplify development and enhance user experience. Commercial products often go further, providing powerful and easy-to-use development tools to achieve drag-and-drop WYSIWYG Web UI development.

The Spring framework is an open-source project and is not a standardized thing. Spring defines its own XML configuration file outline and program interfaces, which carries great uncertainty in the long term. The main developers of Spring come from Interface21 (a company I respect very much). Interface21 maintains itself through related consulting and services. As a commercial group, their interest orientation could completely sway the future direction of Spring. Compared to EJB 3.0’s prestigious lineage as part of the standard with support from many mainstream commercial vendors, Spring's non-standard nature could pose significant risks.

The Spring framework itself does not include persistence implementation, but it supports JDBC, Hibernate, JDO, and iBatis persistence frameworks. Spring uses different DAO and Helper classes to leverage JDBC, Hibernate, iBatis, or JDO, so there is no isolation from the final service provider. In short, code refactoring is needed to switch persistence frameworks. EJB 3.0 integrates persistence into the application service, allowing the underlying persistence implementation to be changed via JPA, such as switching from Hibernate to Toplink. EJB 3.0's persistence offers greater flexibility and facilitates vendor performance optimization and expansion.

Both Spring and EJB3.0 provide runtime services for enterprise applications (such as transactions, security, logging messages, configuration services). Since these services are not directly associated with the application's business logic, they are not managed by the application itself. EJB3.0 uses Java annotations to configure declarative services, while Spring uses XML configuration files. In most cases, EJB3.0 annotations appear simpler and more elegant. The result of Spring using XML to define properties and configure declarative services is a lengthy and unstable configuration file.

The dependency injection pattern (DI) is the best practice for achieving loose coupling in applications. Both Spring and EJB3.0 support DI patterns, but they have profound differences. Spring supports a general (but complex) XML-based dependency injection API. EJB3.0 supports injecting most service objects (such as EJBs and context objects) and JNDI objects declared through simple annotations.

EJB 3.0 fully supports clustering. Applications deployed in service clusters will automatically gain load balancing, distributed caching, state replication, and other functions. The underlying cluster services are hidden behind the EJB3.0 programming interface, shielding all complexities. Spring does not have a simple way to utilize clustering.

Hibernate vs. EJB 3.0

Hibernate and EJB 3.0 are not highly comparable, as Hibernate focuses solely on ORM, while EJB 3.0 is more of a component framework that includes an ORM part. During the design of EJB 3.0, Gavin King, the author of Hibernate, contributed significantly, and it is said that the design philosophy of EJB 3.0 EntityBeans is entirely derived from Hibernate. Simply converting EJB 3.0 EntityBean API calls into Hibernate API calls allows Hibernate to become the Implementation of EntityBeans in EJB 3.0.

When open-source frameworks have become habitual forces and brought people many joys or fatigue, is the arrival of Java EE 5 timely? Regardless, whether to continue choosing open-source or embrace Java EE 5, today this is not an easy choice. Perhaps, as more vendors release products supporting Java EE 5 and provide better tool support, the answer will become clearer.
