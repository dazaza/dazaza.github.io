Basic Knowledge of Hard Disk Data Recovery

by lesile7hl0 on 2010-04-13 06:20:20

**Basics of Hard Disk Data Recovery**

When it comes to data recovery, we cannot avoid discussing data structures, file storage principles, and even the operating system's boot process. These are the fundamental knowledge you must rely on when recovering data from a hard disk. Even if you don't need to recover data, understanding these concepts (even just a little more) can be very helpful for your daily computer operations and applications. Let’s start with the data structure of the hard disk…

### **Hard Disk Data Structure**

When you first buy a hard drive, you can't use it right away. You need to partition and format it, then install an operating system before it becomes usable. For example, in the Win9x/Me series that we have been using until now, a hard disk is typically divided into five parts: the Master Boot Sector, the Operating System Boot Sector, FAT (File Allocation Table), DIR (Directory), and Data. Among these, only the Master Boot Sector is unique, while the others increase in number depending on the number of partitions.

#### **Master Boot Sector Data Recovery**

The Master Boot Sector is located at track 0, cylinder 0, sector 1 of the entire hard disk, and includes the Master Boot Record (MBR) and the Partition Table (DPT). The role of the MBR is to check whether the partition table is correct and determine which partition is the boot partition. At the end of the program, it loads the boot program of that partition (i.e., the operating system boot sector) into memory and executes it. As for the partition table, many people know that it starts with 80H or 00H and ends with 55AAH, covering 64 bytes and located at the end of this sector. It's worth noting that the MBR is generated by the partitioning program (such as DOS's Fdisk.exe), and different operating systems may have slightly different sectors. If you wish, you can write one yourself, as long as it completes the aforementioned tasks. This is also why multi-system booting is possible (as an aside: because the MBR is easy to write, there are many boot sector viruses).

#### **Operating System Boot Sector Data Recovery**

FAT (File Allocation Table) is the file addressing system used by DOS/Win9x systems. For data safety, FAT is usually created twice, with the second FAT serving as a backup for the first. The FAT area immediately follows the OBR (Operating System Boot Record), and its size is determined by the size of the partition and the size of the file allocation unit. Regarding FAT formats, there have always been many options. Microsoft's DOS and Windows use the familiar FAT12, FAT16, and FAT32 formats, but other FAT formats do exist, such as those used by Windows NT, OS/2, UNIX/Linux, Novell, etc.

#### **Directory Area Data Recovery**

OBR (OS Boot Record) refers to the operating system boot sector, which is typically located at track 0, cylinder 1, sector 1 (for DOS; for systems with multiple boot options, it is located in the first sector of the corresponding primary or extended partition). It is the first sector that the operating system can directly access and includes a boot program and a parameter record table called BPB (BIOS Parameter Block) for this partition. Each logical partition has its own OBR, and its parameters vary depending on the partition size and the type of operating system. The main task of the boot program is to determine whether the first two files in the root directory of the partition are the operating system boot files (e.g., IO.SYS and MSDOS.SYS for MSDOS-based Win9x/Me). If so, it reads the first file into memory and passes control to that file. The BPB parameter block records important parameters such as the starting sector, ending sector, file storage format, hard disk media descriptor, root directory size, number of FATs, and size of the allocation unit (formerly known as a cluster). The OBR is produced by a high-level formatting program (such as DOS's Format.com).

#### **File Allocation Table Data Recovery**

Although DATA occupies most of the space on the hard disk, without the previous sections, it is nothing more than boring binary code and has no meaning. It's worth mentioning that the formatting program we usually refer to (high-level formatting, such as DOS's Format program) does not clear the data in the DATA area but only rewrites the FAT table. As for partitioned hard disks, only the MBR and OBR are modified, and most of the data in the DATA area remains unchanged. This is why much hard disk data can be restored. However, even if any one of the MBR/OBR/FAT/DIR is damaged, it will keep our so-called DIY experts busy for quite some time... A reminder: if you frequently organize your disk, your data may be continuous, so even if the MBR/FAT/DIR is completely damaged, we can still use disk editing software (like DOS's DiskEdit) to restore the file as long as we find the starting location of the file (of course, this requires a premise: you haven't overwritten the file...).

#### **Hard Disk Partition Recovery**

DIR stands for Directory, i.e., the root directory area. DIR immediately follows the second FAT table. Having only FAT cannot locate the position of files on the disk; FAT must work together with DIR to accurately locate the file's position. DIR records the starting unit of each file (directory, which is the most important) and the file's attributes. When locating the file's position, the operating system uses the starting unit in DIR combined with the FAT table to determine the specific location and size of the file on the disk. After the DIR area, it is the actual data storage area, i.e., the DATA area.

### **Data Storage Principles**

Since we are talking about data recovery, we naturally cannot ignore the principles of data storage. We will also briefly discuss issues related to data deletion and hard disk formatting...

#### **File Data Reading and Data Recovery**

The operating system reads file information from the directory area (including filename, extension, file size, modification date, and the cluster number of the first cluster where the file is saved in the data area). Here, let's assume the first cluster number is 0023.

The operating system reads the corresponding data from cluster 0023 and then finds FAT unit 0023. If the content is the file end marker (FF), it indicates the end of the file; otherwise, the content saves the cluster number of the next data cluster. This repeats until the file end marker is encountered.

#### **File Writing and Data Recovery**

When we want to save a file, the operating system first finds an empty area in the DIR area to write the filename, size, and creation time, etc. Then, it finds idle space in the Data area to save the file and writes the first cluster of the Data area into the DIR area. The rest of the actions are similar to the reading actions described above.

#### **File Deletion and Data Recovery**

After reading the above about file reading and writing, you might lose confidence in continuing to read further. But don’t worry, deleting files in Win9x is quite simple—just a small change in the directory area—changing the first character of the file in the directory area to E5 indicates that the file has been deleted.

#### **A Small Note on Fdisk and Format**

Similar to file deletion, using Fdisk to delete and recreate partitions and using Format to format logical drives (assuming you did not use the /U unconditional formatting parameter) do not directly delete data from the DATA area. The former only changes the partition table, and the latter only modifies the FAT table. Therefore, mistakenly deleted partitions and formatted hard disks can potentially be recovered...
