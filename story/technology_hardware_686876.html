Building Custom Components for Android Mobile Applications

by dream2012 on 2012-03-02 11:36:38

Android mobile applications have a fixed connection with View class components, such as buttons (Button), text boxes (TextView), editable text boxes (EditText), list boxes (ListView), checkboxes (CheckBox), radio buttons (RadioButton), scroll bars (Gallery), spinners (Spinner), and so on. There are also some more advanced View components with specific purposes, like AutoCompleteTextView, ImageSwitcher, and TextSwitcher. Additionally, various layout components like LinearLayout and FrameLayout are also considered View components since they are derived from the View class.

Your application combines these control and layout components in a certain way to display them on the screen. Generally, these components should be sufficient for your needs. However, you should know that you can create your own components through inheritance. You can inherit from classes like View or Layouts, or even some more advanced control classes.

Below, we will discuss why inheritance is necessary:

- You can create a completely customized component for a specific function, such as creating a control component using 2D graphics for sound control, similar to electronic controls.

- You can combine several components into a new one, such as a ComboBox (a list of texts that can be input) or dual-pane selector control (two List windows where you can assign each item's relationship).

- You can create your own layout components. The SDK provides a series of options to help you build your application, but advanced developers may find it necessary to develop new layouts based on existing ones, or even create new components from scratch.

- You can override the appearance or functionality of an existing component. For example, change how the EditText (editable text) component is displayed on the screen (you can refer to the Notepad example, which teaches you how to create an underlined display page).

- You can capture events like key presses and handle them in a general way (for example, in a game).

To achieve a specific goal, it might be necessary to extend an existing View component. Below, we'll use some examples to teach you how to do this.

### Basic Approach

Here are some general steps to guide you in creating your own components:

1. Let your class (Class) inherit from an existing View class or subclass of View.

2. Override some methods of the parent class: Methods that need to be overridden generally start with "on", such as onDraw(), onMeasure(), and onKeyDown().

This also applies to Activity or ListActivity derivatives, where you need to override some lifecycle functions and other functional HOOK functions.

3. Use your inherited class: Once your inherited class is created, you can use it wherever the base class can be used, but completing the functionality is up to you.

4. Inheritable classes can be defined within activities, making them easier to call, though this is not mandatory (perhaps in your application, you want to create a component that everyone can use).

### Fully Customized Components

The method for fully customizing components can create graphical components used for display, perhaps a graphical meter like a voltmeter, or a ball showing lyrics in karaoke that rolls with the music. No matter which way, you cannot simply complete it by combining existing components, no matter how you combine them.

Fortunately, you can easily create a component entirely according to your requirements, and you will find that the only limitations are your imagination, the size of the screen, and the performance of the processor (remember that your application will ultimately run on platforms with lower performance than desktop computers).

Here's a brief introduction on how to create a fully customized component:

- The most universal parent class of VIEW is undoubtedly the View class, so at first, you need to create a subclass based on this class.

- You can write a constructor to extract attributes and parameters from XML files, or you can define these attributes and parameters yourself (maybe the color and size of the graphic meter, or the width and amplitude of the pointer, etc.).

- You may need to write your own event handlers, property access and modification functions, and some code related to the functionality of the component itself.

- If you want the component to display something, you are likely to override the onMeasure() function, and therefore you must override the onDraw() function. If both functions use the default settings, the onDraw() function will do nothing, and the default onMeasure() function will automatically set a dimension of 100x100, which may not be what you want.

- Other on... series functions that need to be overridden must be rewritten.

#### onDraw() and onMeasure()

- The onDraw() function will pass you a Canvas object, through which you can do anything on a 2D graphic, including other standard and common components, text formatting, and anything else you can think of.

- Note: This does not include 3D graphics. If you want to use 3D graphics, you should change your parent class from View to SurfaceView and use a separate thread. You can refer to the GLSurfaceViewActivity example.

- The onMeasure() function is a bit tricky because it is the key part of the interaction between the component and the container. onMeasure() should be overridden to accurately represent the measurements of its contained parts. This gets a bit complicated because we need to consider the restrictions of the parent class (passed through onMeasure()), and we should know that once the width and height are measured, the setMeasuredDimension() method must be called immediately.

In summary, executing the onMeasure() function goes through the following stages:

1. The overridden onMeasure() method is called, along with the height and width parameters (widthMeasureSpec and heightMeasureSpec are both integer types), and you should consider the size constraints of your product.

2. Your component calculates the necessary measurement length and width through onMeasure() to display your component, and it should conform to the specifications, although it can implement some functions outside the specifications (in this case, the parent class can choose what to do, including clipping, sliding, throwing exceptions, or calling the onMeasure() function again with different parameters).

3. Once the height and width are calculated, the setMeasuredDimension(int width, int height) method must be called, otherwise an exception will occur.

### A Customized Component Example

In the API Demos, the CustomView provides an example of a customized component, which is defined in the LabelView class.

The LabelView example involves all aspects of custom components:

- First, let the custom component derive from the View class.

- Write a parameterized constructor (parameters can come from XML files). Some processing has already been done in the parent View class, but there are still some new parameters specific to the Labelview custom component that need to be handled.

- Standard public functions such as setText(), setTextSize(), setTextColor().

- Override the onMeasure() method to determine the size of the component (Note: In LabelView, it is achieved through a private function measureWidth()).

- Override the onDraw() function to display the Lable on the provided canvas.

In the example, you can see the usage of the custom component LabelView through custom_view_1.xml. In the XML file, pay special attention to the mixed use of android: and app: parameters. The app: parameters indicate individuals in the application that are considered LabelView components, and these will also be defined as resources in the R class.

### Compound Components (or Compound Controls)

If you don't want to create a fully customized component but rather a new component composed of several existing components, then compound component technology is more suitable. Simply put, combining several existing components into a logical group can encapsulate them into a new component. For example, a Combo Box component can be seen as a combination of an EditText and a Button component with a pop-up list. If you click the button to select an item from the list, in Android, there are actually two other View classes that can achieve a similar effect: Spinner and AutoCompleteTextView, but Combo Box as an example is easier to understand.

Here's a simple introduction on how to create a compound component:

- Generally start from the Layout class and create a derivative class of the Layout class. Perhaps in the Combo box, we would choose Horizontal LinearLayout as the parent class.

- Remember, other Layout classes can be nested inside, so the compound component can be a mix of any components. Note that just like Activity, you can declare your components in external XML files or nest them in the code.

- In the constructor of the new compound component, first call all the constructors of the parent class and pass in the corresponding parameters. Then set other aspects of your compound component, where to create the EditText component, and where to create the PopupList component. Note: You can also introduce some of your own properties and parameters in the XML file, which can also be used by your compound component.

- You can create event listeners to listen to events triggered by the View class in the new component, such as listening to the single-click event of the List option, and you must update the value of your EditText after this event occurs.

- You may create some of your own properties with access and modification methods. For example, allow setting the initial value of EditText and provide access to it.

- In the derivative class of Layout, there is no need to override the onDraw() and onMeasure() methods because Layout has good default handling. However, if you feel it is necessary, you can also override it.

- You may also override some on-series functions, such as overriding onKeyDown(), you can choose the corresponding value in the list by pressing a certain key.

In summary, using the Layout class as the base class has the following advantages:

- Just like activity, you can declare your new component in XML files, or you can nest it in the code.

- There is no need to override the onDraw() and onMeasure() functions, as they already work well.

- You can quickly create your compound component and use it like a single component.

### Tweaking an Existing Component

In some cases, you may have simpler ways to create your components. If you already have a very similar component, all you need to do is simply derive your component from this component and override some necessary methods. You can also achieve this through a fully customized component method, but by deriving a new component from View, you can easily obtain some existing processing mechanisms, which are likely what you want, without having to start from scratch.

For example, in the SDK, there is a NotePad example (NotePad application). This example demonstrates many practical details of the Android platform, such as learning how to derive a notepad that can automatically wrap lines from EditView. This is not a perfect example because compared to earlier versions, these APIs have changed a lot, but it does illustrate some points.

If you haven't checked this program yet, you can now import the Notepad example into Eclipse (or just view the corresponding source code through the provided link). Specifically, check the definition of MyEditText in NoteEditor.java.

There are a few things to note:

- Declaration (The Definition)

- This class is defined by the following line of code: `public static class MyEditText extends EditText`

- It is defined inside the NoteEditor activity class, but it is public, so if necessary, it can be called from outside NoteEditor as NoteEditor.MyEditText.

- It is a static class (static class), meaning there will be no "virtual method" for accessing data through the parent class, making it a standalone class that doesn't heavily depend on NoteEditor. This is a clear idea for creating inline classes that do not need to access the outer class, ensuring that the generated class is small and can be conveniently called by other classes.

- It is an extension of the EditText class, which is the parent class we chose to customize. Once completed, the new class can be used as a regular EditText.

Initialization of the Class:

Generally, the parent class is called first. Furthermore, this is not a default constructor but a parameterized constructor. Since EditText is created using parameters extracted from XML layout files, our constructor also extracts parameters and passes them to the parent class.

Method Overriding:

In this example, only the onDraw() method is overridden. But you can easily override other necessary methods for your custom component.

For the Notepad example, by overriding the onDraw() method, we can draw blue lines on the canvas of EidtView (the canvas class is passed through the overridden onDraw() method). The super.onDraw() function should be called near the end of the function. The method of the parent class should be called, but in this example, we call it after drawing the blue lines.

Using Custom Components:

Now that we have our custom component, how do we use it? In the Notepad example, the custom component is directly used in the predefined layout file. Let's take a look at the note_editor.xml file in the res/layout directory.

This custom component is created in XML as a general View class and described using the full path package. Note that the inline class is represented as NoteEditor$MyEditText, which is the standard method of referencing inline classes in Java programming.

Other attributes and parameters in the definition will be passed to the constructor of the custom component and then to the EditText constructor, so these parameters are also the parameters you use for the EditText component. Note that you can also add your own parameters, which we will discuss below.

That's all you need to do, indeed it is a simple example. But the key point is: your needs determine the complexity of your custom component.

A more complex component may need to override more on-series functions and have many special functions to fully realize the functionality of the custom component. The only limitation is your imagination and what tasks you need the component to perform.

### Start Your Componentization Journey Now

As you can see, Android provides a clever and powerful component model that allows you to accomplish your tasks as much as possible. From simple component adjustments to component combinations, and even fully customized components, flexible use of these techniques should give you an Android application that perfectly meets your visual requirements.
