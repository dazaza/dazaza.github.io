Use the Dalvik Debug Monitor Service (DDMS) tool

by dream2012 on 2012-03-08 11:43:00

Android comes with a debugging tool called Dalvik Debug Monitor Service (DDMS). It provides port forwarding services, screen capture on the device, information on threads and heap usage, logcat, process, and radio state information, call and SMS spoofing, location data spoofing, and more. This article only introduces the general features of DDMS and does not provide detailed descriptions of its features. DDMS is located in the SDK's tools/ directory. To run it, enter the directory from a terminal or command prompt and type ddms (or ./ddms on Mac/Linux). DDMS will operate on either the emulator or a connected device; if both the emulator and a connected device are running, DDMS defaults to the emulator.

### How DDMS Works

DDMS acts as an intermediary, connecting the IDE to applications running on the device. Each Android application runs in its own process, managing its own virtual machine (VM), and each process listens on a different port for the debugger. When it starts, DDMS connects to adb and establishes a device monitor service between them, notifying DDMS when devices connect or disconnect. A virtual monitor service is established between adb and DDMS, which notifies DDMS when a VM on the device starts or terminates. Once the VM is running, DDMS obtains the process ID via adb and establishes a connection to the VM debugger through the adb daemon (adbd) program on the device. DDMS can communicate with the VM using a custom wire protocol.

On each VM on the device, DDMS opens a port to listen for the debugger. For the first VM, DDMS listens on port 8600, the next on port 8601, and so on. When a debugger connects to one of these ports, all communication is forwarded between the debugger and the corresponding VM, allowing DDMS to function like a remote debugging session.

DDMS also opens another local port, the DDMS "base port" (default 8700), which also serves as a listener for the debugger. When a debugger connects to this port, all traffic selected in DDMS is forwarded to the VM, making it the default connection port for the debugger.

**Tip:** You can set the DDMS configuration port in the File > Preferences menu. The settings are saved in the "$HOME/.ddmsrc" file.

Learn about debugging in Dalvik: Debugging applications on the Dalvik Virtual Machine is similar to debugging on other VMs, but when stepping out of synchronized code, the "current line" cursor jumps to the last line of the method.

### Left Panel

The left panel of the Debug Monitor shows the current status of each emulator/device, displaying a list of VMs on each emulator/device. Each VM is uniquely identified by the package name of the application running on it. Use this list to find and attach the process you want to debug to the VM. Near the VM name is a "debug communication" port (in the far-right column); if you connect a debugger to one of these ports in the list, you will be connected to the corresponding VM on the device. However, when using DDMS, you only need to connect to port 8700, at which point DDMS forwards all traffic to the currently selected VM (note: when you select a VM in the list, the listed port includes 8700), so you don't have to reconfigure the debugger port every time you switch VMs.

When an application running on the device calls waitForDebugger() (or selects this option in Developer Options), a red icon will appear next to the client name while it waits for the debugger to attach to the VM, and the icon will turn green once a debugger connects.

If you see a bug icon with a strike-through, it means that DDMS cannot establish a connection between the debugger and the VM because it cannot open the local port on the VM. If you see this icon on all VMs on the device, it is likely that another instance of DDMS is running (including Eclipse plugins).

If you see a question mark in an application package space, it means that DDMS has received the application's PID from adb but somehow cannot handshake with the VM process. Try restarting DDMS.

### Right Panel

On the right side, the Debug Monitor displays useful information and some pretty cool tools via tabs.

- **Info**

This view shows general information related to the currently selected VM, including process ID, package name, and VM version.

- **Threads**

The Threads view has a list of threads running on the target VM. To reduce the amount of data sent, thread update information is only sent when the "Threads" button on the toolbar is pressed, corresponding to each VM. This list contains the following information:

- ID: A unique thread ID assigned by the VM. In Dalvik, these are odd numbers starting from 3.

- Tid: Linux thread ID. As the main thread of a process, it equals the process ID.

- Status: VM thread status. Daemon threads display with a star and will be one of the following states:

- running: Executing application code

- sleeping: Calling Thread.sleep()

- monitor: Waiting to acquire a monitor lock

- wait: In Object.wait()

- native: Executing native code

- vmwait: Waiting for a VM resource

- zombie: Threads in a dead process

- init: Thread is initializing (you won't see this step)

- starting: Thread is preparing to run (you won't see this either)

- utime: Total time executing user code, in "jiffies" (usually 10ms), available only on Linux.

- stime: Total time executing system code, in "jiffies" (usually 10ms).

- Name: Thread name. "ID" and "Name" are set when the thread starts and reserved areas are updated periodically (default is every 4 seconds).

- **VM Heap**

Displays some heap status, updated during garbage collection. If a VM is selected, the VM Heap view prompts to disable heap updates. Click the "Show Heap Updates" button in the top-left toolbar to return to the VM Heap view, and click Cause GC to perform garbage collection and update heap status.

- **Allocation Tracker**

In this view, you can track memory allocations for each VM. Select a VM on the left, then click Start Tracking, and click Get Allocations to view memory allocation information since tracking began. Corresponding data will be displayed in the table below, and clicking it again will refresh the list.

- **Control**

With these controls, you can simulate specific device states and activities, including:

- Phone State: Change voice call status and data plan (local, roaming, searching, etc.) and simulate different network speeds and delays (GPRS, EDGE, UTMS, etc.)

- Phone Activity: Simulate phone calls and SMS messages to the emulator.

- Location Control: Send false location data to the emulator, enabling operations like GPS mapping. To use location control, start your application in the Android emulator and open DDMS, click the Emulator Control tab and press Location Control. From there, you can:

- Manually send individual longitude/latitude coordinates to the device.

- Click Manual, select coordinate format, fill in the fields, and click Send.

- Use a GPX file to describe a path on the device for playback.

- Select a GPX file and load it; once loaded, click the Play button to replay the path.

- When starting playback from GPX, you can adjust the playback speed and control it with pause and skip buttons in the DDMS panel. DDMS parses track points (in the first table) and tracks (in the second table, combined simply via fields to support multi-segments). Only tracks can be played; clicking a track point in the first list sends only its coordinates to the device. After selecting a track, you can play it.

- Use a KML file to describe sequential location points on the device for playback.

- Click and open a KML file; once loaded, click the Play button to send coordinates to your location program.

- When using a KML file, it parses an element whose value consists of a set of longitude, latitude, and altitude values. Example: -122.084143,37.421972,4.

- In your file, you can include multiple elements, each containing an element. When you do this, a set of location markers will be added to the track, and DDMS will send one location marker per second to the device.

- A way to generate an appropriate KML file is: Find a location in Google Earth, right-click the location point, select "Save Place As" from the menu on the left, and choose Kml as the storage format.

Note: DDMS does not support creating paths via lat1, long1, lat2, long2,... methods. Similarly, nodes in the middle are currently unsupported, and future versions may support time-based locations and paths with a single location element.

For more information on location data spoofing methods, check the Location-Based Services APIs documentation.

### File Browser

Through the File Browser, you can browse the device file system and perform basic management such as dragging and pulling files. These operations revolve around adb drag commands, providing a GUI experience.

Open DDMS, select Device > File Browser... to open the File Browser window. You can drag files into device directories but cannot drag files out of the device. To copy files from the device, select the file and click the Pull File from Device button on the toolbar. To delete files, use the Delete button on the toolbar.

If you're interested in the SD card image on the emulator, you can create an image with the mksdcard command and load it when starting the emulator. For example, in the /tools directory, execute:

```

$ mksdcard 1024M ./img

$ emulator -sdcard ./img

```

Now, when the emulator is running, the DDMS File Browser will be able to read and write to the sdcard directory. However, your files will not automatically display. For example, if you add an MP3 file to the sdcard directory, the media player will not see the file until after you restart the emulator (when restarting the emulator from the command line, ensure that the sdcard is loaded again).

### Screen Capture

You can capture a screen image from the device or emulator by selecting Device > Screen Capture from the menu or pressing the CTRL-S shortcut key.

### Browse Processes

You can get the same output as using the ps -x command on a specific VM by selecting Device > Show Process Status from the menu bar.

### Trigger GC

Pressing the trash can button on the toolbar performs garbage collection.

Run Dumpsys and Dumpstate on the Device (logcat)

From Dalvik, run dumpsys (logcat) by selecting Device > Run logcat... from the menu bar.

From Dalvik, run dumpstate by selecting Device > Show Device State... from the menu bar.

### Test Wireless State

By default, wireless state is not output on a standard logcat (which contains a lot of information). To view wireless information, click Device > Show Wireless State... from the menu or run the logcat described in Logging Wireless Information.

### Stop a VM

You can stop a VM by selecting Action > Suspend VM from the menu, which causes the VM to call System.exit(1).

### Learn About DDMS Defects

DDMS has the following known defects:

- If you connect and disconnect a debugger, after DDMS reconnects to the client, the VM thinks the debugger has been closed. This will eventually be fixed.
