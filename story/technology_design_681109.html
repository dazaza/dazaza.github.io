Lifecycle of an Android Application

by dream2012 on 2012-02-28 11:16:01

In most cases, each Android application runs in its own Linux process. When some code of an application needs to run, the process is created and continues to run until the system determines that the process is no longer useful. The system then recovers the memory used by the process so it can be allocated to other applications. It is crucial for application developers to understand how different application components (especially Activity, Service, and BroadcastReceiver) affect the lifecycle of the application process. Incorrect use of these components may cause the system to kill a process that is executing important tasks. A common example of a lifecycle bug is with BroadcastReceiver: when a BroadcastReceiver receives an Intent in its BroadcastReceiver.onReceive() method, it starts a thread and then returns. Once it returns, the system considers the BroadcastReceiver inactive, and therefore the process hosting the BroadcastReceiver is no longer useful (unless there are other active components in the process). As a result, the system may kill the process at any time to recover memory. This will also terminate the thread spawned within the process. To resolve this issue, you can start a service from the BroadcastReceiver so the system knows there is still active work happening in the process.

To decide which process to kill when memory is low, Android places processes into an "importance hierarchy" based on the components running in each process and their states. Process types include (in order of importance):

- **Foreground process**, closely related to what the user is currently doing. Different application components can bring their host process to the foreground through various methods. A process can be considered as being moved to the foreground when any of the following conditions are met:

- The process is running an interactive Activity on the frontmost screen (its onResume() method has been called).

- The process has a running BroadcastReceiver (its BroadcastReceiver.onReceive() method is executing).

- The process has a Service, and there is executing code within one of the Service's callback functions (Service.onCreate(), Service.onStart(), or Service.onDestroy()).

- **Visible process**, which has an Activity that can be seen by the user on the screen, but is not in the foreground (its onPause() method has been called). For example, if the foreground Activity is a dialog, the previous Activity hidden behind the dialog would belong to such a process. These processes are very important and generally should not be killed unless it is necessary to ensure the operation of foreground processes.

- **Service process**, which has a Service that has been started using the startService() method. Although users cannot directly see what these processes are doing, the tasks they perform are of concern to the user (such as background MP3 playback or uploading/downloading network data). Therefore, the system will keep these processes running unless there is insufficient memory to maintain all foreground and visible processes.

- **Background process**, which has an Activity that the current user cannot see (its onStop() method has been called). These processes have no direct impact on the user experience. If they correctly follow the Activity lifecycle (for more details, refer to Activity), the system can kill these processes at any time to recover memory for use by the first three types of processes. There are usually many such processes running in the system, so they are kept in an LRU list to ensure that the most recently seen processes by the user are the last to be killed when memory is low.

- **Empty process**, which does not contain any active application components. The only reason to retain such a process is to improve startup speed when one of the application's components needs to run next time.

The system classifies processes based on the importance of the currently active components within them. Refer to the documentation for Activity, Service, and BroadcastReceiver to learn more about how these components function throughout the entire lifecycle of a process. Each process category's documentation describes in detail how they affect the entire lifecycle of the application. The priority of a process may also increase based on its dependencies with other processes. For example, if Process A binds to a service in Process B via Context.BIND_AUTO_CREATE or uses a ContentProvider, then Process B must be considered at least as important as Process A during classification.
