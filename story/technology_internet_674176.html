A Brief Discussion on the Symptoms of a WEB Server Under CC Attack and Its Prevention

by qingian007 on 2012-02-24 11:03:01

This article -- Roaring Life Integration -- Weilan Xia reprinted

CC attacks are a type of DDoS (Distributed Denial of Service), and compared to other DDoS attacks, CC seems to be more technically sophisticated. In this attack, you won't see fake IPs or exceptionally large abnormal traffic, but it causes the server to fail in establishing normal connections. A regular user with an ADSL line is enough to bring down a high-performance Web server. From this, we can see its harmfulness; calling it a "Web Killer" is not an exaggeration. What's most worrying for webmasters is that the technical threshold for this kind of attack isn't very high. Using tools and some IP proxies, even a beginner or intermediate-level computer user can carry out a DDoS attack.

Attack Symptoms

CC attacks have a certain degree of concealment, so how do we determine if the server is currently suffering from or has previously suffered from a CC attack? We can confirm through the following three methods:

(1). Command Line Method

When generally under a CC attack, the Web server will show port 80 being externally closed, as this port has already been clogged by a large amount of junk data, causing normal connections to be terminated. We can check this by entering the command 'netstat -an' in the command line. If you see a large number of similar connection records like the following, it basically means you're under a CC attack:

...

TCP 192.168.1.3:80 192.168.1.6:2205 SYN_RECEIVED 4

TCP 192.168.1.10:80 192.168.1.6:2205 SYN_RECEIVED 4

TCP 192.168.1.10:80 192.168.1.6:2205 SYN_RECEIVED 4

TCP 192.168.1.10:80 192.168.1.6:2205 SYN_RECEIVED 4

TCP 192.168.1.10:80 192.168.1.6:2205 SYN_RECEIVED 4 ...

Here, [192.168.1.6] is the IP address of the host used for proxying the attack, [SYN_RECEIVED] is the TCP connection status flag, meaning "[initial synchronization state during connection]", indicating that the handshake response cannot be established and remains in a waiting state. This is the characteristic of the attack, and usually, there will be many such records, representing attacks from different proxy IPs.

(2). Batch Processing Method

The aforementioned method requires manually inputting commands, and if there are too many IP connections on the Web server, it can be quite labor-intensive. We can create a batch processing file to determine whether a CC attack exists through this script code. Open Notepad and enter the following code, saving it as CC.bat:

@echo off

time /t >>log.log

netstat -n -p tcp |find ":80">>Log.log

notepad log.log

exit

The above script filters all current connections to port 80. When we feel the server is abnormal, we can double-click to run this batch processing file, then open the log.log file to view all connections. If there are many connections from the same IP to the server, it can be determined that the IP is carrying out a CC attack on the server.

(3). Check System Logs

The downside of the two methods mentioned above is that they can only check the current CC attack, and are powerless to determine if the Web server was previously subjected to a CC attack. At this point, we can check through Web logs, since Web logs faithfully record all IP accesses to Web resources. By reviewing the logs, we can determine if the Web server was previously subjected to a CC attack and identify the attacker's IP to take further measures.

Web logs are generally located under the C:\WINDOWS\system32\LogFiles\HTTPERR directory, where files named similarly to httperr1.log exist. These files record Web access errors. Administrators can choose the appropriate logs based on the time attributes of the logs and analyze them to determine if the Web site has been attacked by CC. By default, the items recorded in Web logs are not many, but we can configure IIS to record more items for security analysis. The steps are:

[Start -> Administrative Tools] open [Internet Information Server], expand the left items to locate the corresponding Web site, right-click and select [Properties] to open the site properties window, click the [Properties] button under the [Website] tab, and in the [Log Properties] window under the [Advanced] tab, check the corresponding [Extended Properties] to allow Web logs to record more details. For example, the [Bytes Sent], [Bytes Received], [Time Taken] items are not selected by default but are very useful in recording and judging CC attacks, so they can be checked. Additionally, if your security requirements are higher, you can set the [New Log Schedule] under the [General] tab to record [every hour] or [every day]. To facilitate future analysis and determination of time, you can check [File Naming and Creation Uses Local Time].

CC Attack Prevention Strategies

After determining that the Web server is currently or has previously suffered from a CC attack, how can we effectively prevent it?

(1). Unbind Domain Name

Generally, CC attacks target the domain name of the website. For instance, if our website domain is [www.vvppss.com], the attacker sets the attack target as this domain in the attack tool and then carries out the attack.

To counter such attacks, we can unbind this domain name in IIS, making the CC attack lose its target. The detailed steps are: open [IIS Manager], locate the specific site, right-click [Properties] to open the site's property panel, click the [Advanced] button next to the IP address, select the domain item for editing, delete the [Host Header Value] or change it to another value (domain).

Through simulated testing, after unbinding the domain name, the CPU of the Web server immediately returns to normal, and connections via IP work normally. However, the drawback is also evident; canceling or changing the domain name affects visitors' access. Also, this method is ineffective against CC attacks targeting the IP address. Even if the domain name is changed, once the attacker discovers it, they can attack the new domain name.

(2). Domain Spoofing Resolution

If a CC attack against the domain name is detected, we can resolve the attacked domain to the address 127.0.0.1. Since 127.0.0.1 is the local loopback IP used for network testing, resolving the attacked domain to this IP achieves the purpose of the attacker attacking themselves. As a result, no matter how many bots or proxies the attacker uses, their system will crash, making them suffer the consequences.

Additionally, when our Web server suffers from a CC attack, we can resolve the attacked domain to authoritative government websites or police websites, allowing the police to deal with them.

Nowadays, most Web sites use dynamic domain resolution services provided by service providers like Xinwang. Users can log in and make settings accordingly.

(3). Change Web Port

In general, Web servers provide services externally through port 80, so attackers typically attack the default port 80. Therefore, we can modify the Web port to prevent CC attacks. Run the IIS Manager, locate the relevant site, open the [Properties] panel of the site, and under [Website Identifier], there is a TCP port which defaults to 80. We can change it to another port.

(4). IIS IP Blocking

By using commands or checking logs, we can discover the source IPs of CC attacks. We can then set up IIS to block these IPs from accessing the Web site, thereby preventing IIS attacks. In the [Properties] panel of the relevant site, click the [Directory Security] tab, and under [IP Address and Domain Restrictions], click the [Edit] button to open the settings dialog box. In this window, we can set [Allow Access], i.e., [Whitelist], or [Deny Access], i.e., [Blacklist]. For instance, we can add the attacker's IP to the [Deny Access] list, thus blocking the IP's access to the Web.

(5). Group Policy Block IP Range

Open Local Security Settings, click [IP Security Policies on Local Machine]—Create IP Security Policy—Next—Enter any name, such as 'Block', and keep clicking Next until Finish, at which point a policy named 'Block' will be created.

Next, click [IP Security Policies on Local Machine]—Manage IP Filter List and Filter Actions—Add—Name it 118.123.5 (to recognize it, preferably fill in the corresponding IP range)—Add—Next—Select a specific IP subnet, input 118.123.5.0 and change the subnet mask to 255.255.255.0—Next—Select my IP address as the target—Next—Protocol type as any—Next—Complete—Fully block.

Then click the policy named 'Block' we started creating, click Properties—Add—Next—Next, select all network types—Next—When prompted, click Yes—to the IP filter list, select the one we just created named 118.123.5—Next—Select Block—Next to Complete—Close.

Finally, right-click the 'Block' policy, assign it, and at this point, we have successfully blocked the IP range starting with 118.123.5.
