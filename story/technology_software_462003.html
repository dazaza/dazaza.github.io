Abstract: Due to the openness of the Internet and the incomplete design of information and system security confidentiality, there have been network attacks and damages in the past, which have caused a lot of trouble in people's daily lives and economic activities. As the most widely used service network currently, the WWW service has seen frequent occurrences of JSP websites being attacked by hackers, leading to increasing attention on its security. This article focuses on the security issues of JSP applications, analyzes some important current vulnerabilities in JSP, proposes solutions, and ultimately summarizes the research on JSP security.

Keywords: Security issues of JSP applications

1. Overview of JSP: JSP is a standard for dynamic web page technology. JSP technology integrates segments of traditional web HTML files with Java programs (using extensions like *.HTM, *.HTML) into a JSP file (*.JSP). JSP technology uses Java programming language and XML-like tags and scripts to encapsulate the logic for generating dynamic web pages. Page tags and scripts can exist through server resources within the application logic. JSP web pages support independent design for web page design and display logic, enabling reusable component-based design, making the development of web-based applications more agile and easy. JSP executes on the server side, usually returning an HTML text to the client side, so only a browser client is needed to view it. JSP technology leverages the advantages of existing tools and technologies and can be extended to support enterprise-level distributed applications. As part of the Java 2 family and an indivisible part of Java technology, JSP technology can support highly complex web-based applications. Since the embedded scripting language in JSP pages is based on Java, and all JSP pages are compiled into Java Servlets, JSP benefits from all the advantages of Java technology, including robust memory management and security. As part of the Java platform, JSP is written in the Java programming language, and JSP pages are typically compiled into Java Servlets, which are standard Java extensions. Page developers can access all Java application environments using Java technology, ensuring scalability and portability. When a JSP page is called for the first time, if it does not already exist, it will be compiled into a Java Servlet class and stored in the server’s memory. This allows the server to respond very quickly the next time the page is called.

JSP Application Servers provide completely open access to resources, from servlets to JSP pages, and security issues may threaten any or all accesses from the server, network server, client to the page, even distributed DDoS or worm attacks, potentially affecting the entire Internet. Especially, preparation for validating input and controlling access to resources is always critical. The JSP architecture is complex, involving many subsystems working together. The interaction between these subsystems often becomes the source of security vulnerabilities. Cookies store small amounts of information on the client side provided by the server to maintain session status or track client browser activities. There are two reasons why security is threatened if sensitive data is stored in cookies: first, the entire content of cookies on the client side is visible; second, although browsers generally do not provide the ability to forge cookies, there is nothing to prevent users from fully forging cookie responses to the server. 1) Malicious tag embedding: CERT Advisory CA-2000-02 introduces the issue of malicious HTML tags embedded in client requests. This problem generally refers to attacks that can target other users or the server, but the latter is less common.
