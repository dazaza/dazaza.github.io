WINDOWS Programming Options

by cg4fh5rr2 on 2010-03-29 22:14:51

To illustrate the various techniques of Windows programming, this book provides many example programs. These programs are written in C and use the Windows API directly to develop applications. I refer to this approach as "classic" Windows programming. This is how we wrote programs for Windows 1.0 in 1985, and it remains an effective way to write Windows programs today.

### APIs and Memory Models

For programmers, the operating system is defined by its API. The API includes all the system calls available to applications, along with the relevant data types and structures. In Windows, the API also implies a specific program structure that we will examine at the beginning of each chapter. Generally speaking, the Windows API has remained consistent since Windows 1.0, with little significant change. Programmers experienced in writing for Windows 98 would find the source code of Windows 1.0 programs very familiar. One way the API changes is through enhancements. Windows 1.0 supported fewer than 450 function calls, whereas today there are thousands.

The biggest change in the Windows API and its syntax came from the transition from a 16-bit architecture to a 32-bit architecture. From version 1.0 to version 3.1, Windows used the segmented memory model on 16-bit Intel 8086, 8088, and 286 microprocessors. For compatibility reasons, 32-bit Intel microprocessors starting from the 386 also support this mode. In this mode, the size of the microprocessor registers is 16 bits, so the int data type in C is also 16 bits wide. In the segmented memory model, memory addresses consist of two parts: a 16-bit segment pointer and a 16-bit offset pointer. From the programmer's perspective, this is quite messy and introduces distinctions between long or far pointers (which include both the segment address and the offset address) and short or near pointers (which include only the offset address with an assumed segment address).

Starting with Windows NT and Windows 95, Windows supports the 32-bit flat addressing memory model in the 32-bit mode of Intel 386, 486, and Pentium processors. The int data type in C has also been extended to a 32-bit value. Programs written for the 32-bit versions of Windows use simple 32-bit pointer values for flat linear space addressing.

The API used for the 16-bit versions of Windows (Windows 1.0 to Windows 3.1) is now called Win16. The API used for the 32-bit versions of Windows (Windows 95, Windows 98, and all versions of Windows NT) is now called Win32. Many function calls remain the same when transitioning from Win16 to Win32, but some require enhancement. For example, graphic coordinate points changed from 16-bit values in Win16 to 32-bit values in Win32. Additionally, some Win16 function calls returned a two-dimensional coordinate point contained within a 32-bit integer value. This is not possible in Win32, so new function calls were added that operate differently.

All 32-bit versions of Windows support the Win16 API (to ensure compatibility with older applications) and the Win32 API (to run newer applications). Interestingly, Windows NT works differently from Windows 95 and Windows 98. In Windows NT, Win16 function calls are translated into Win32 function calls through a translation layer before being processed by the operating system. In Windows 95 and Windows 98, the operation is reversed: Win32 function calls are translated into Win16 function calls through a translation layer and then processed by the operating system.

At one time, there were two different sets of Windows APIs (at least in name). Win32s ("s" stands for "subset") is an API that allows programmers to write 32-bit applications that run on Windows 3.1. This API only supports 32-bit versions of functions already supported by Win16. Additionally, the Windows 95 API was once called Win32c ("c" stands for "compatibility"), but this term has been abandoned.

Currently, both Windows NT and Windows 98 are considered to support the Win32 API. However, each operating system still supports certain features not supported by the other. Because of their substantial similarities, it is possible to write programs that can run under both operating systems. Moreover, it is generally believed that these two products will eventually merge into one.
