Essential file for each app: AndroidManifest.xml

by dream2012 on 2012-03-07 14:24:10

The AndroidManifest.xml file is required for every application and is located in the root directory of the application. It describes the global variables of the package, including exposed application components (activities, services, etc.) and the implementation classes for each component, what kind of data can be manipulated, and where they run. An important aspect of this file is the intent filters. These filters describe when and under what circumstances an activity should be launched. When an activity (or the operating system) wants to perform an action, such as opening a web page or opening a contact selection screen, it will create an Intent object. This object contains several descriptors that describe the operation you want to perform, the data you want to handle, the type of data, and some other information. Android compares the information in the Intent object with all the intent filters exposed by the applications to find the activity that best handles the requested data and action. More information about intents can be found on the Intent page.

In addition to declaring the application's activities, content providers, services, and intent receivers, you can also specify permissions and instrumentation (security controls and testing) in the AndroidManifest.xml file. Some common points to note:

Almost all AndroidManifest.xml files (like other Android files) include a namespace declaration xmlns:android="http://schemas.android.com/apk/res/android" in their first element. This declaration makes standard Android attributes available in the file, providing most of the data for the XML elements in the file.

Most manifests contain an element that defines all components and properties available at the application level within this package. All top-level applications that need to be presented to users from the program launcher must include at least one activity component that supports the MAIN action and appears in the LAUNCHER category.

Below is a detailed structural outline of the AndroidManifest.xml file, describing all available tags:

The root node of the file. Describes all contents of the package. The following can be placed within its node:

Request a security authorization; your package must be granted this permission to operate correctly. Refer to the Security Model documentation for more information about authorizations. A manifest can contain zero or more of these nodes.

Declare a security authorization used to restrict which applications can access components and special features within your (or another) package. Refer to the Security Model documentation for more information about authorizations. A manifest can contain zero or more of these nodes.

Declare an instrumentation component that can be used to test the code of this or another package. For more details, see Instrumentation. A manifest can contain zero or more of these nodes.

Declare the root node for application-level components within the package. This node can describe the global (and/or) default properties of the application, such as labels, icons, themes, required permissions, etc. A manifest can contain zero or one such node (multiple application nodes are not allowed). Under this node, you can include zero or more declarations for each of the following components:

An Activity is the primary mechanism through which an application interacts with the user. When an application runs, the first screen the user sees is an activity. Additionally, most other screens used by the user are implemented by separate activities, declared with additional activity tags.

Note: Every activity, whether externally visible or only used within the package, must have a tag defined in the manifest. If there is no matching tag in the manifest for an activity, you won't be able to launch it.

Additionally, to support runtime activity lookup, you can include one or more elements to describe the actions supported by the activity.

Declare a set of specific Intent values supported by the component in the form of an IntentFilter. In addition to the various values that can be specified under this element, attributes can also be assigned to support unique labels, icons, and other information about the described action.

The Intent action supported by this component.

The Intent category supported by this component.

The Intent data MIME type, Intent data URI scheme, Intent data URI authority, or Intent data URI path supported by this component.

You can also associate one or more meta-data entries with your activity, which other clients can retrieve to obtain any information about this activity.

Add new metadata to the activity, which clients can obtain through ComponentInfo.metaData.

A BroadcastReceiver allows the application to receive notifications about data changes and events occurring, even if the application is not running. Similar to the activity tag, you can optionally include one or more elements or values supported by the receiver. For more information, see the descriptions of and for activities.

A Service is a component that can run for an arbitrary length of time in the background. Similar to the activity tag, you can optionally include one or more elements or values supported by the service. For more information, see the descriptions of and for activities.

A ContentProvider is a component used to manage persistent data and publish it to other applications. You can also attach one or more values as described in the activities section.
