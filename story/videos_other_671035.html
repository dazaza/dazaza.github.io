Server Virtualization: Tricks for Managing a Virtual Environment

by xwkj0983 on 2012-02-21 14:54:03

Truly measurable ROI is widespread, and almost every enterprise wants to achieve multiple objectives in areas such as consolidation, configuration, disaster recovery, security, and regulatory compliance (the survey results show an average of 5 objectives, with some respondents potentially reaching 10 or more). Therefore, it's easy for people to only see the benefits of virtualization technology while ignoring the need for Virtual System Management (VSM).

When I refer to VSM, I'm talking about the combination of rules required to manage the entire virtualization system—not just the management of virtual machines (VMs) or hypervisors. VSM includes traditional system management processes such as configuration, performance management, patch management, capacity planning, backup and recovery, configuration management, and process automation, as well as new processes like VM management, hypervisor management, live migration, and virtual image management.

When there is insufficient virtual system management, deploying virtualization can lead to issues, which are prominently reflected in the following aspects:

If virtual machines are deployed without any control, their security and compliance will be affected; uncontrolled virtual host deployment increases marginal costs for additional software licenses, system management manpower, and IT costs, and performance may also be impacted, leading to availability and downtime issues for end users.

Improper deployment of virtual hosts may fail to meet the consolidation-efficiency ratio; when the number of virtual machines that managers need to handle increases, the productivity of the managers also decreases.

Virtual system management controls can prevent these issues, and the corresponding tools mainly fall into two categories.

Process automation tools ensure that only pre-authorized virtual machines are deployed and checked according to standard configuration templates. Any exceptions that lead to rejection must be manually approved, thus eliminating virtual machine deployment errors caused by human or other anomalies.

Resource discovery and inventory management can quickly detect a new server joining the production network and treat it as an affiliated DHCP server, thereby accelerating the detection speed of problems and reducing problem resolution time.

Performance and application management tools can detect IP address conflicts between clients and servers. With the presence of comprehensive and real transaction monitoring, problems will be captured even before the call center receives such issues, allowing for earlier resolution.

In the real world, fragile management can lead to various problems, and we can only rely on virtual system management to avoid these issues as much as possible. Of course, the potential problems caused by fragile VSM and the advantages of robust VSM are numerous, so I won't list them all here.

It is worth noting that if you do not apply virtual system management protocols to your virtualization deployment, it will lead to serious problems such as high costs, low productivity and performance, increased downtime, and high risks.

To achieve enhanced performance and availability management

Performance and availability management are key indicators of virtual systems, and they are crucial for enterprises deploying virtualization. In a recent EMA study involving over 600 enterprises, 62% of surveyed users mentioned reducing downtime, and 60% mentioned improved business continuity—these two factors were important in their decision to deploy virtualization.

To ensure performance and availability, we recommend that users understand the five key performance areas in the virtual architecture:

Discovery Tools should automatically locate, identify, and provide the complete topology of each IT service and keep all parts up-to-date. "Discovery" should be able to detect physical servers, virtualization platforms, virtual hosts and guests, applications built on top of them, and their interconnections. Ideally, it will also provide a "single source of truth" for storing discovered systems, such as a federated configuration management database.

Physical Architecture Monitoring EMA research shows that most enterprises plan to retain a robust non-virtual environment for the foreseeable future. At the same time, the virtual environment always runs on top of the physical architecture, so monitoring the availability and performance of the underlying physical system components remains important. This includes details such as granular-level resources, network performance, file I/O, system uptime, response time, etc.

Virtual Architecture Monitoring Virtual System Management (VSM) tools must monitor dynamic virtual system performance matrices (response time, resource utilization, I/O rates, etc.) in real-time. When applications and components migrate, they should be tracked, and multiple virtual environments across platforms, technologies, vendors, hosts, subnets, and even data centers should be monitored simultaneously.

Operational Service Monitoring To fully understand the performance and availability of end-to-end services, tools must completely monitor the operational environment for the service. Monitoring content includes servers, applications, databases, middleware, networks, storage, client connections, etc., giving users deep insights into the virtual platform, understanding the results of dynamic resource utilization, and how component changes affect overall service performance.

Partial Integration The above capabilities must be connected into a single view, fully integrating images, time correlation, detailed reports, and predictive warnings, connecting all different physical and virtual components. Not only measuring from inside out (measuring the performance and availability of components within the data center), but also meeting the outside-in approach (measuring response time, availability, and end-user experience at the client side).

With these five core performances, enterprise users can discover and understand complex dynamic physical and virtual architectures, see how all components interact to deliver IT services, predict, and prevent potential problems.

Balancing Load to Improve Security and Performance

To improve system security and performance, balancing the workload between virtual hosts is very important. The goal of virtualization is to increasingly tap into the potential performance of the host. Many virtualization users have server utilization below 40%, while generally speaking, achieving 80% utilization is reasonable, including disk, network, CPU, and memory. There are many ways to bring system utilization closer to this number, one of which is to run the virtual machines for development, testing, and production tasks on the same virtual host cluster.

However, when you balance workloads, there is one particular point to note: how much data mixing can occur on your Host Bus Adapter (HBA) or an IP-based network. Data mixing can occur when using VLANs, NPIVs, or accessing data with the same HBA in two or more security zones.

Some government agencies and companies do not require data mixing, so these organizations use separate physical network interface cards (PNICs), pSwitches, and HBAs within a single virtual host for each active network and data storage to balance the load, and some even implement 100% separation of virtual hosts. This is not a universal method because it increases costs, but it is indeed a feasible method.

Sometimes avoiding VLANs has reasons. For example, if a pSwitch fails, data streams from one VLAN can appear in another VLAN, allowing data to cross security zones. This situation happens whenever a pSwitch malfunctions.

The issue of different security zones can be resolved through the popularization of VLANs or private pNICs. For example, the pSwitch problem mentioned above allows an ESX host to serve VMs from different security zones, although it raises the overall server utilization to 80%, it requires additional hardware investment such as pSwitches, pNICs, and HBAs.

Besides hardware, balancing the workload of virtual machines also requires administrators to be doubly vigilant, preventing data leakage from a failed pSwitch and preventing administrators from operating virtual machines in inappropriate security zones. Unfortunately, there are almost no tools available in the current market to control these issues. Catbird’s V-Security and Reflex Systems’ Virtual Security Appliance can assist in protecting network security by rejecting suspicious virtual machine access, but these tools are specific to VMware. Tools for other virtual hosts are still under development.

Finally, it is important to remind that to ensure balanced loads within the same virtual host collection, combine shared security zones. Meanwhile, unless you are sure that it can increase administrator vigilance, do not mix security zones.
