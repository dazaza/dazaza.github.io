1: **SQL Injection (SQL Injection)**

This vulnerability is quite common, such as ASP+Access injection, ASP+MSSQL injection, ASPX+MSSQL injection, etc. Nowadays, there are also many PHP+MYSQL cases.

PS: If someone says static pages are easier to hack, it’s likely they’re either delusional or just a noob. For example, someone recently said: "I only hack dynamic pages because static pages are too fragile."

PS: Nowadays, agencies like the CIA, Department of Defense, and X Pentagon (using XXXBSD+Apache HTML) are all static. Want to pretend to be cool by using static pages? Go ahead—good luck getting famous that way...

Let's start with ASP.

Nowadays, most garbage websites use ASP. Why? Generally speaking, ASP is bundled with IIS, and if you're using IIS, you're probably on Windows. Gaining elevated privileges in the Windows system is usually pretty straightforward.

Preventive measures include filtering sensitive characters such as `and`, `or`, `'`, `select`, `update`, `chr`, `delete`, `%20from`, `;`, `insert`, `mid`, `master.`, `set`, `=`, etc. However, there are ways to bypass these filters, such as submitting URL-encoded encryption to break through the filter (e.g., `or 1=1` becomes `%6f%72%20%31%3d%31`) or submitting ASCII `CHAR(101)+CHAR(97)+CHAR(115)+CHAR(116)`.

Key points:

1. Database passwords must be encrypted with MD5. Ideally, both usernames and passwords should be encrypted. Passwords should ideally be over 13 characters long, combining letters, numbers, and special characters. How do you remember them? For example, if a site is called `redhatd`, set the password to `redhatd1~redhatd1~`. If it's too short, repeat it. Another method is remembering lyrics, like "Still a virgin at 21" (`esyshscnzs`), where each syllable represents the first letter of the sound. Then add special characters at the end.

Next, let's talk about PHP and JSP:

Such sites, aside from ASP, often use PHP. However, hackers who target PHP sites are generally more skilled. The preventive methods are similar: filter sensitive characters like `and`, `or`, `'`, `select`, `update`, `chr`, `delete`, `%20from`, `;`, `insert`, `mid`, `master.`, `set`, `=`, etc. Password length should be as long as possible.

FAQ:

Q: Why should passwords be so long?

A: Because many people will take the hashed passwords obtained from SQL injections and try to crack them on certain websites. Some well-known ones include:

- http://www.xmd5.com/

- http://www.cmd5.com/

There are also some foreign sites I won't mention here. They all have one thing in common: instead of directly decrypting the password, they search their own databases for matches. Although a certain doctor has researched the reverse algorithm for MD5, it hasn’t been widely applied yet. So, MD5 is essentially irreversible.

Q: Does having a longer password make it secure?

A: No. Some people might brute-force MD5 hashes. (Don’t call me crazy.) PS: There’s a software called MD5Crack2 that can brute-force MD5 hashes extremely quickly. Administrators, give it a try.

Although such people are rare, once brute-forcing starts, no password is uncrackable—it’s just a matter of time. What can you do? You could consider adding Chinese characters or Japanese kana (e.g., “ァ” or “◇”) into your password to make it nearly impossible to guess.

PS: Social Engineering:

Be mindful of password uniqueness. If you manage multiple servers, strongly recommend not using the same password for all. Otherwise, if one server gets compromised, all of them are at risk.

Redhatd's personal tips:

Teach you a relatively simple database isolation method: Install a virtual machine on the server, such as VirtualPC or VMware, and run Win2003 inside it. If needed, you can rent a server yourself and use shared internet access between your computer and the virtual machine. Place the database inside the virtual machine. This way, the database is effectively on an internal network, just like on your local machine. Now, even if the database is exposed on the external network, it cannot be accessed via TELNET on port 1433. Even if someone gets access to the database server, it won’t be useful.

2: **SQL & Permissions (Competence chmod)**

Microsoft's MSSQL runs on the Windows platform and supports two types of user authentication: MSSQL users and Windows identity verification.

Here lies the extreme insecurity: if an attacker gains system privileges, they can modify the administrator password and then log in to the local MSSQL using Windows identity verification. They can view, modify, or delete any data in the corresponding database and even create a SYSTEM ADMIN privilege SQL account.

The general solution: When setting up MSSQL management accounts, avoid using the default login mode. If MSSQL uses the default login mode, Windows user verification can be used. Therefore, only MSSQL accounts should be used. Delete `xp_cmdshell` unless absolutely necessary. It’s also recommended to remove the `WSCRIPT.SHELL` component.

For websites whose permissions don’t need to be frequently changed, set them to read-only. If there’s no need to use `sa` privileges, it’s best not to. On Linux systems, it’s recommended to set website directory permissions to `chmod 544` (root user can read and execute, other users can only read).

3: **Upload and Download (Upload and Download)**

This type of vulnerability is quite common. Uploads are typically done through deception, such as uploading a file despite restrictions on certain file types. Since website programs often filter headers only, modifying the header can allow the upload of any program.

Classic examples include packet capture uploads and NC submissions.

Download vulnerabilities involve constructing special URLs to download files that are useful to the attacker. For example, in Discuz!:

```php

$dbhost = 'localhost'; // Database server

$dbuser = 'root'; // Database username

$dbpw = 'hook789'; // Database password

$dbname = 'dz'; // Database name

$pconnect = 0; // Persistent database connection (0=off, 1=on)

```

If this PHP file can be downloaded normally, it means you already have the credentials to connect to the SQL database. Therefore, as mentioned earlier, it’s generally better not to use the highest SQL user privileges.

The causes of upload and download vulnerabilities vary. Some are due to system vulnerabilities, others to website program vulnerabilities. Properly setting permissions can prevent further exploitation.

4: **Deception and Being Deceived (Deception and Cheated)**

These issues often arise due to human negligence or lack of thoroughness.

For example, XSS (Cross-Site Scripting) is popular abroad. Although XSS cannot directly obtain passwords or sensitive information, it can deceive users by creating fake login boxes. If you believe the box belongs to the system and enter your correct username and password, your credentials will be stored elsewhere and used by hackers to infiltrate the site.

Another example is cookie spoofing. Some websites use cookies for backend authentication. If someone constructs your cookie, they can bypass entering a password and directly access the admin panel.

Although cookie-based authentication is insecure, many commercial programs still use it. Similarly, session-based authentication isn’t foolproof because tools like SESSIONIE can keep sessions active indefinitely by setting periodic connections from compromised machines. This method is less known but effective. Overall, cookie spoofing remains more common.

Social Engineering:

No need to elaborate—just be cautious. Avoid using personal information like ID numbers, birthdays, or email addresses as passwords.

5: **ARP-Sniffer**

This vulnerability deserves attention. Recently, sites like Hei Fang, Security China, Hacker X Archives, and Xiumu Blog were hacked due to ARP spoofing, which allowed attackers to obtain backend account credentials.

Defense is possible, but many hosting providers lack awareness. Defending against ARP requires binding MAC addresses, which must be executed with high-level privileges.

SNIFER must operate within the same subnet (the same LAN segment). C-class IP addresses must match. To resolve, use `ipconfig /all` to check your MAC address, then use `arp -s ip mac_address` to bind the MAC address. If unsure, search Google or use `arp /?` for command help.

Example:

```bash

> arp -s 157.55.85.212 00-aa-00-62-c6-09 .... Adds a static entry.

> arp -a .... Displays the ARP table.

```

This format allows checking if someone is ARP-spoofing your server via the gateway.

Binding MAC addresses in Linux:

Create an `/etc/ethers` file containing the correct IP/MAC mappings:

```

192.168.2.32 xx:xx:xx:xx:xx:xx

```

Add the following line to `/etc/rc.d/rc.local`:

```

arp -f

```

For Debian:

For MAC address `xx:xx:xx:xx:xx:xx`, add `hwaddress ether xx:xx:xx:xx:xx:xx` to the corresponding network card section in `/etc/network/interfaces` or add a line in `/etc/init.d/networking`: `/sbin/ifconfig eth0 hw ether xx:xx:xx:xx:xx:xx`.

Restart the service: `/etc/init.d/network restart`.

5: **Shellcode and Exploits**

Because some web applications are publicly downloadable (e.g., New Cloud Website Management System, DVBBS, etc.), hackers can analyze them for weak filtering mechanisms. Such vulnerabilities are generally impossible to defend against until patches are released. These are often discovered by highly skilled individuals and sometimes referred to as "0-day" vulnerabilities. A "0-day" isn’t a "zero-day" vulnerability but rather a period when a vulnerability is known but no patch exists.

Defensive measures include using more static web pages and regularly checking official updates.

6: **Extreme Server Configuration**

Let’s break this into parts:

1. Users and Accounts:

- Rename the Windows Administrator account and completely delete the Guest account.

- Create accounts like Admin, Admin1, Admin2… Assign different permissions to each (e.g., Admin has modification rights, Admin1 has read and execute rights, Admin2 has folder listing rights, etc.).

- In Linux, everyone knows the password and user files are `/etc/passwd` and `/etc/shadow`.

passwd:

```bash

root:x:0:0:root:/root:/bin/bash

slinux:x:1000:1000:slinux,,,:/home/slinux:/bin/bash

```

Change to:

```bash

slinux:x:0:0:slinux:/home/slinux:/bin/bash

root:x:1000:1000:root,,,:/root:/bin/bash

```

shadow:

```bash

root:$1$Ihe0v$YedFPdHzMix1J0pVStefv0:13810:0:99999:7:::

slinux:$1$MT60Xwi5$xzTteUFZav55J8PRgnUOf1:13810:0:99999:7:::

```

Change to:

```bash

slinux:$1$MT60Xwi5$xzTteUFZav55J8PRgnUOf1:13810:0:99999:7:::

root:$1$Ihe0v$YedFPdHzMix1J0pVStefv0:13810:0:99999:7:::

```

This way, `slinux` becomes the root user. Even if someone gets the root account, it’s useless.

PS: When rewriting `shadow` and `password`, permission issues may occur. Use the `root` user to set permissions to `chmod 700`, modify as needed, and reset to `chmod 500` afterward.

PS: Be careful not to mess up the encrypted passwords in Linux.

2. Command Restructuring:

Windows users know commands like `cmd` or `command`.

Restructure commands:

```bash

doskey 1=dir // Define CMD's dir command as 1

doskey dir=x // Redefine dir command as x. Thus, 1 and dir have the same effect, and dir equals x.

doskey 2=del

doskey del=x2

doskey 3=net

doskey net=x3

```

Commands to modify: `doskey` itself (to prevent others from resetting), `net` (to prevent new account creation), `del`, `copy`, `format`, `command`, and `cmd`.

Restructuring commands in Linux:

```bash

alias 1="ls" // Define bash's ls command as 1

alias ls="s" // Redefine ls command as s

alias cd..="cd .."

alias cd\ \=\="s2"

alias md="mkdir"

alias mkdir="s3"

alias rd="rmdir"

alias rmdir="s4"

```

PS: Remove or redefine the `unalias` command since it restores redefined commands.

Commands to modify: `useradd`, `rd`, `vi`, `cat`, `rm`, `head`, `cp`, `userdel`, `groupadd`, `chmod`, etc.
