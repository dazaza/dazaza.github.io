Let's talk about smartphone software.

by translatec on 2010-03-31 11:53:35

Using only the baseband chip is commonly referred to as a feature phone.

The baseband chip minus the co-processor (CP, which is usually a multimedia accelerator). This type of product is typically represented by MTK solutions; essentially all MTK products fall under this category. Spreadtrum and other companies also produce similar products. This represents a feature phone with enhanced multimedia functionality.

Baseband chip + Application Processor (AP), which is what we generally refer to as a smartphone (smart phone). Some solutions integrate the application processor and the baseband processor into a single chip, such as Qualcomm's MSM7200A. It features an ARM11 core (application processor) and an ARM9 core (baseband processor), both communicating through shared memory. Of course, smartphones can also use co-processors that enhance audiovisual processing capabilities.

The subject of this article is the software on the application processor of smartphones, so we will only discuss smartphones, or the baseband+AP solution.

In smartphones, the realization of phone functions is primarily handled by the application processor (AP), while the baseband chip provides communication functions. You can think of the AP as a computer and the baseband chip as the AP's wireless modem. This wireless modem provides voice calls, SMS, internet access, UIM card functions, etc., via an AT interface (similar to the interface between a computer and a modem, but each manufacturer has its own extended commands). This article mainly discusses the software on the AP.

The "AP software" mentioned in this article refers to the combination of all software running on the application processor (AP). In this article, it is also referred to as mobile phone software or smartphone software. If you consider a mobile phone as a computer, then mobile phone software is equivalent to the collection of operating systems and all common software on a computer. Therefore, the importance of mobile phone software is self-evident.

On the other hand, mobile phone software is a major cost factor for smartphones. As one Design House executive put it: "Software and hardware should not have issues. As long as the software can fix it." Of course, AP software isn't that easy to get right. After discussing the current state of AP software, we will also look at the composition of AP hardware.

From an implementation perspective: AP software = operating system + middleware + applications. Taking Google's Android as an example:

Of course, this is just a detailed description, excluding the bootloader. From an execution perspective, AP software always starts with the bootloader, which loads the operating system kernel and then hands over control to the operating system. The operating system loads the file system and executes the system initialization program on the file system, starting all background service programs, running a window manager to provide a graphical user interface to users. Then, based on user operations, it either executes a certain function of the window manager or calls the corresponding program.

Smartphones use operating systems mainly including Windows Mobile, Symbian, Linux, and some other proprietary systems, such as RIM's BlackBerry operating system. Among these systems, only the Linux kernel is a "bare" operating system. Both Windows Mobile and Symbian's released versions include the operating system, middleware, and a few applications. Compared to BlackBerry and iPhone, we usually refer to Windows Mobile, Symbian, and Linux as "open" systems because different manufacturers can use them (with varying usage costs).

When discussing smartphone operating systems online, Palm OS is often mentioned. Originally, Palm OS was the operating system for PDAs. In late 2003, Palm acquired Handspring, gaining Handspring's Treo smartphone. Simultaneously, they split their hardware and software divisions into PalmSource (software) and PalmOne (hardware), with PalmSource owning Palm OS. In September 2005, a Japanese company named Access purchased PalmSource for $324 million in cash. Access renamed Palm OS to Garnet OS. Access's main business was an embedded browser called NetFront, which also provided China Unicom's Color E plan. In December 2006, Palm bought the exclusive permanent license for the source code of Palm OS Garnet for $44 million, to be used in products like the Palm Treo smartphone.

Shanghai Century Technology Co., Ltd. has been developing its own "Harmony" embedded operating system and announced good cooperation relations with the TD-SCDMA Industry Alliance, but currently, there are no mass-market products. "Harmony" is a proprietary system with very little public information. The main idea of "Harmony" is to break down software into self-descriptive components, which can be assembled as needed during use, akin to SOA (Service-Oriented Architecture) in the embedded field. SUN's Jini introduced a "publish and discover services" framework in the late 1990s, and "Harmony" can be seen as a C/C++ implementation of Jini. Recently, Google's Android platform also includes a similar component model where components (Activities) register their functionalities with the system, and each component requests services from other components (intents), with the system determining the appropriate components based on the intent. Perhaps, having good ideas alone is insufficient; the ability to implement them is crucial.

Some companies developing smartphone software solutions do not use open operating systems. They might use simple real-time operating system (RTOS) kernels or adopt dual-task solutions. These companies also provide their SDKs (Software Development Kits) or source code for their solutions. Using these systems may present the following problems:

- The system's stability has not been sufficiently tested.

- There is no widespread support for third-party software.

If customizing their functionalities requires significant R&D efforts to familiarize oneself with their proprietary solutions. If the developed product cannot be launched due to stability issues, the initial R&D investment would be wasted.

Middleware builds on top of the operating system, providing a development platform for application software. Good middleware can offer a comprehensive application software development framework (Application Framework). The most fundamental questions middleware addresses are:

- GUI development package. GUI stands for Graphical User Interface. Developers use the GUI development package to create the graphical interfaces for application software.

- Component communication mechanism. Middleware must provide communication methods for applications.

Middleware functionality is quite complex. So, "middleware, invisible," application software primarily handles user interactions, and much of the unseen functionality related to user interaction is implemented in middleware, encapsulated and exposed as interfaces for application software to call.

Most companies providing operating systems also offer smartphone software platforms that include middleware. Only on the Linux platform do we need to customize our own kernel, construct our own middleware, and develop application software. There are also companies that provide Linux middleware, such as South Korea's MIZI Research.

Detailed estimates indicate that among shipped smartphones, AP software is predominantly Symbian and Windows Mobile, with a large number of phones using Linux (Motorola, Samsung, etc.). BlackBerry has a significant market share domestically but has not yet entered the domestic market. Additionally, there is the iPhone, which uses a proprietary system.

Notably, the shipments of Windows Mobile in the domestic market are continuously increasing. According to Huang Nianliu's analysis, there are five types of Windows Mobile available in the market:

- "Reversion" edition: Officially released Windows Mobile Traditional Chinese version.

- "Gray market" edition: Devices originally marked for European operators, combining the functionality of the first type with relatively lower prices.

- "Commoner" edition: Uses Windows CE with some added applications, masquerading as Windows Mobile.

- "Impostor" edition: A simulation of Windows Mobile on other operating systems, with only the first couple of menu layers resembling the original.

- "Pretender" edition: Stripped from other devices, reassembled with a similar design.

Why is it so chaotic? Because of sales. This also highlights the power of branding from another angle.

What is the difference between smartphones and feature phones? Actually, feature phones can perform most of the functions of smartphones. The most fundamental difference lies in their origins. Feature phones are continually reducing application functionalities of wired communication terminals (industry term: mobile station), while smartphones are handheld computers with added wired communication capabilities.

Feature phone software evolves from the wireless terminal software of the telecommunications industry, being embedded software that continually reduces application functionalities. Smartphone software evolves from the general-purpose operating systems of the computer industry, expanding and adapting to the needs of handheld devices. The software of both is developed by programmers from different industries and mindsets, but these two industries and mindsets are continuously merging. The mobile phone is the classic representative of the so-called 3C convergence: Computer (Computer), Communication (Communication), and Consumer Electronics (Consumer Electronics).

In telecommunications, there is the concept of 5W: Anyone (Whoever) can communicate with anyone (Whomever) at any place (Wherever) and any time (Whenever) any information (Whatever: language, images, text, and data). This is merely the communication function that smartphones aim to achieve.

The concepts of 3C and 5W have been around for over a decade. The more critical role that mobile phones will play is as a mobile information terminal or platform. Under mobile conditions, humans connect to the vast network of the information society through the mobile phone terminal; the information society publishes content to humans through the mobile phone platform. This is why companies like Google, which primarily focus on information services, take mobile phone platforms seriously. Or, to put it another way, besides integrating computers, communications, and consumer electronics, mobile phones also play a key role in the integration of 3C with the information service industry.

Feature phone software mostly runs in the privileged mode of the CPU, appearing to PC programmers as a monolithic kernel with a UI. On the other hand, the software system of smartphones largely mimics the PC software system, separating the kernel, drivers (which can be compiled into the kernel or stand-alone), and applications. Various applications, including the basic window manager, exist as separate applications running in user space. The distinction between smartphone software and PC system software mainly lies at the application level, such as screen size and input methods. Additionally, as portable devices, smartphones must significantly reduce power consumption.

One of the core ideas in software evolution is reusability—don't reinvent the wheel. From a software perspective, the system architecture of smartphone software is more advanced and reasonable. System architecture is the greatest advantage of smartphone software, allowing almost infinite scalability.

Developing a perfect operating system takes a long time and enormous R&D investment. It seems that in the smartphone domain, compared to traditional real-time operating systems (VRTX, VxWorks, ThreadX, etc.), systems evolved from PC operating systems like Windows Mobile and Linux are more likely to succeed.

Using Windows Mobile or Symbian requires authorization from Microsoft or Symbian companies. Blackberry and iPhone, with their proprietary systems, can only be studied by other manufacturers, not utilized. Below, we mainly discuss smartphone software based on Linux.

This article refers to smartphone application processor software based on the Linux kernel as Linux smartphone software.

Speaking of smartphone software (Part 2)

Simply put: choosing the Linux platform means less capital investment; since basically no licensing fees are required, the software cost of the product is also relatively low. The advantages of choosing the Linux platform are also quite obvious:

A company in Nanjing has a research and development team of over 30 people working on smartphones using QT, and after nearly two years, they have not yet launched a product. QT is a commercial GUI (Graphical User Interface) development package on the Linux platform, also available under the GPL license. A company in Shenzhen is developing smartphones using Linux+GTK, achieving a relatively high degree of completion, but they have not yet brought it to market. GTK is a GUI development package entirely different from QT, released as free software under the LGPL license. Although the lack of product launch might be due to various considerations, the difficulty of making Linux-based smartphones commercially viable is evident.

Both Windows Mobile and Symbian are led by single companies, offering unified and open application software development platforms. Many companies and individuals develop application software for them. However, Linux-based smartphones do not have a unified application software development platform, thus lacking a large number of third-party applications. This could be the hardest obstacle for Linux-based smartphones to overcome, as this defect is inherent to Linux as open-source software. Despite many companies or alliances attempting to unify the Linux platform, there has been no significant success so far. Moreover, even if the Linux application platform were unified, it might not align with the free spirit of Linux. This is a difficult conflict to reconcile between freedom and centralization, open-source and commercial interests.

Since 2007, open-source Linux smartphone projects like OpenMoko, Android, and GPE Phone Edition have opened their reference designs, somewhat reducing the difficulty of developing Linux-based smartphone software. However, these projects themselves still have various issues. Even with relatively mature reference designs, skilled engineering teams are needed to apply and customize these solutions. Even if a prototype is made, without sufficient testing, it would be risky to proceed with mass production.

Linux smartphones lack a unified brand, putting them far behind Windows Mobile or Nokia-led Symbian in terms of brand advantage, placing them at a disadvantage in market competition.

The primary reason Linux-based solutions can reduce software costs is the extensive use of open-source software, including the Linux kernel. Open-source software is always released under specific software licenses, and compliance with these licenses is necessary for proper use. There are many types of open-source software licenses. The most common ones are GPL and LGPL. Licenses come in different versions. For instance, the Linux kernel is released under the GPLv2 license.

Android takes a rather unique approach, not using the standard GUI development package on the Linux platform. The Android system includes a Java Virtual Machine, with its GUI development package and application development framework (framework) implemented in the Java language and executed by the Java Virtual Machine. This will be discussed further below.

The main GUI development packages on the Linux platform are QT and GTK. KDE is a PC window system based on QT development, while GNOME is a PC window system based on GTK development. Many Linux distributions' window systems are developed based on KDE or GNOME. The GNOME Mobile in Figure 1 is a mobile application development platform planned by the GNOME project group, aiming to standardize GTK-based mobile application development. This project has not yet been completed.

QT is a widely-used cross-platform GUI development package. The embedded version of QT mainly consists of two parts: Qtopia Core (originally called QT/Embedded) and Qtopia. Qtopia Phone Edition is a customized mobile phone version based on Qtopia and Qtopia Core. Figure 2 shows the architecture of Qtopia Phone Edition. QCop is the component communication method of the platform, similar to DCop on the KDE platform.

From the figure, it can be seen that the completion level of Qtopia Phone Edition is not high. Starting from Qtopia Phone Edition, a lot of work is still required to achieve a commercial smartphone, such as selecting browsers, email clients, instant messaging programs based on QT, etc.

GTK and QT have two significant differences:

- Documentation is too long. This is also a characteristic of open-source hardware. Open-source software developers are usually fewer in number, and they prefer spending time coding rather than writing documentation. Since code development progresses quickly, even if there is documentation, it is hard to keep it synchronized with the code.

OpenMoko focuses mainly on hardware and software component design, not paying enough attention to a unified API interface. This differs significantly from LiPS, Android, Nokia's Maemo, etc. In fact, open-source mobile phone projects should prioritize unifying the application software development platform. Although it is difficult to achieve, Linux-based mobile phones indeed need a unified application software development platform. Even a tripartite division would be better than the fragmented landscape we see now.

As previously introduced, one of Android's main characteristics is that its application framework and GUI library are implemented in the Java programming language. Inside Android, there is a Java Virtual Machine called Dalvik, which interprets and runs Java programs. Applications on the Android platform must also be developed in the Java language. Online comments suggest that Android's virtual machine uses Apache's Harmony class library. The Harmony class library is an open-source Java class library developed by the Apache organization, released under the Apache license.

The Android application framework adopts a Mash-up component model: Components (Activity) register their functionalities with the system, and each component makes its own request (intent) when needing services from other components. The system determines the appropriate component based on the intent from the registered components. The API documentation for the Android platform is quite comprehensive. To developers, creating application hardware based on the Android platform should be a serious and enjoyable task. However, choosing the Android platform presents the following issues:

Even without source code, Android can be ported to compatible development boards (using the ARMv5 instruction set), but due to the inability to customize, it cannot be used in actual products. Google has not clearly stated when it will release the source code for the middleware. In fact, Google has not explicitly said that it will release the source code for the middleware.

Google says Android is "the first complete, open, and free mobile platform." In English, "free" can mean both "freedom" and "free of charge." How it is interpreted depends entirely on Google. I believe what Google most wants to capture is the application software developer community, hoping that companies and individuals will learn its SDK and develop application software for its platform. Google will certainly collaborate with manufacturers to develop Android-supported smartphones. However, whether Google will fully open the middleware code on the web remains unknown.

If Google completely opens this platform, similar to J2ME, various companies will inevitably add their own extensions, leading to incompatible platforms. Therefore, Google might use commercial means to ensure the consistency of the Android platform.
