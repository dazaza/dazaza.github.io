Slashdot Interview with the Creator of Python

by anonymous on 2013-09-04 14:40:51

Interviews: Guido van Rossum Answers Your Questions

The creator of Python, Guido van Rossum, officially left Google in January 2013 and joined Dropbox. On August 19, 2013, the Slashdot website initiated an interview topic with Guido, where netizens could ask questions in the comments. On August 25, Slashdot compiled a post summarizing questions and answers related to "object-oriented programming, functional programming, PyPy, Python 3," among other topics.

From Google to Dropbox

nurhussein asked: "Hi, what prompted you to leave Google for Dropbox? What did you do at Google, and what will you be doing at Dropbox?"

Guido: After seven years at Google, I was ready for some changes in my life, and Dropbox's opportunity aligned well with my thoughts. At a high level, my job hasn't changed much:

Spending half my time on tasks as Python's BDFL (Benevolent Dictator For Life).

Working as a regular engineer within the company (not a manager or team leader).

Engaging in code reviews, architecture, and design work.

Handling lots of emails.

Completing much of my work using Python.

Of course, some details are different. At Google, I only worked on two projects: Initially, I spent two years developing Mondrian, an online code review tool. Although it was never open-sourced, it inspired Rietveld, which is used by the Python, Go, and Chromium communities. After joining Google App Engine, I worked on many different things, mostly related to Python. My last big project in Python was a new Python database API called NDB.

I've been at Dropbox for seven months now, and my primary task has been designing the Dropbox data storage API. It's somewhat ironic to use this term to describe the data storage, but it's not my fault—there’s some overlap between Dropbox data storage and Google App Engine data storage.

Even more ironically, despite all the design work I’ve done and completing two prototypes in Python, the SDK we released last month only supports Java, Objective-C, and JavaScript. But I'm working on improving it; this interview has slowed me down a bit.

Why Python Avoids Some Common Object-Oriented Styles

i_ate_god asked: "Interfaces, virtual classes, private members, etc... why doesn't Python have these features?"

Guido: There are two reasons that come to mind: You don’t really need them, and they're hard to implement without compile-time type checking. Python started as a skunkworks project (unsupported and unencouraged by management but also not stopped), and I wanted quick results. This led me to remove features that weren't truly necessary or urgent; it also made me focus on runtime type checks, which limited the features Python could support. I wasn't a hardcore object-oriented enthusiast—I just wanted a simple language that ended up being somewhat object-oriented by accident.

In modern Python, there are rough equivalents for these features, but they don't always work well or lead to significant overhead, so they’re generally avoided, though they do have their uses.

Functional Languages

ebno-10db asked: "Some people argue that Python is, at least partially, a functional language. You disagree, and so do I. Just having some map and filter-like functions doesn't make it a functional language. From my understanding, these functions were added by some list enthusiasts, and you've tried to remove them several times. Overall, you aren't a fan of functional programming, at least from Python's perspective.

Question: Do you feel that functional programming paradigms are not particularly useful overall, or are they just not suitable for Python? I'd love to hear your reasoning from various angles."

Guido: I’m not a believer in taking ideas to extremes; I try to take a pragmatic approach in design choices (but not "too" pragmatic). I weigh the readability and usability of real-world code. In some places, map() and filter() are appropriate, but on the other hand, Python has list comprehensions. I no longer hate reduce() because I once used (a) sum() instead, or (b) its readability was poor. So we added the built-in sum(), moved reduce() out of the built-ins, and into a utility function.

My view on functional languages is that they rely heavily on powerful compilers, like Haskell. For such compilers, functional generics are very useful because they enable numerous transformations, including parallelization. However, the Python interpreter doesn't know enough about your code, making this less useful. Therefore, I don't think it's reasonable to add certain functional ideas to Python, as they're useful in functional languages but not suitable for Python, and they would make the code less readable for those unfamiliar with functional programming (i.e., most programmers).

I also don't think functional languages have matured enough to become mainstream. While I admit I'm not deeply familiar with all aspects of Haskell, any language less popular than Haskell still has practical applications, and I haven't heard of any functional language surpassing Haskell in popularity. Regarding Haskell, I think it's great for proving compiler techniques, but its "purity" might be its biggest barrier to widespread adoption. Its singularity makes it unsuitable for most people.

Multiline Lambda Expressions

NeverWorker1 asked: "One common complaint about Python is the limitation on lambda expressions, specifically that they can't span multiple lines. Clearly, Python's handling of whitespace is the main reason for this. I've spent some time thinking about how to implement multiline lambdas, and the best method I could come up with involves cramming unnecessary symbols, like C-style braces, which would make it messy. Is there a better way, and do you think this feature will be added?"

Guido: Really? I've basically never heard those complaints, except from someone asking during a Slashdot interview.

This is indeed a better way: using the def keyword to define a regular function in the local scope. The defined function object becomes a local variable, which has the same semantics as using lambda, unless it references a local variable, and there are no syntactical restrictions. For example, the following two snippets are equivalent:

```

def make_adder(n):

def adder(x):

return x + n

return adder

```

Then here is the expression using lambda:

```

def make_adder(n):

return lambda x: x + n

```

Andrew Koenig once pointed out to me a scenario where lambda is quite fitting: when you have a long list or dict containing many lambda expressions. Trying to replace lambda with named functions would be cumbersome, as defining and naming a large number of functions, then referencing them via the list or dict, would be overwhelming. However, in such cases, the lambda expressions tend to be simple, and if there are exceptions, using def before the list or dict is a good compromise.

PyPy

Btrot69 asked: "Do you think PyPy represents the future? Are you skeptical? If so, why?"

Guido: I remain skeptical for two reasons: (1) They don't yet support Python 3. (2) Many extension modules aren't fully compatible. However, I hope they resolve these issues. As competitors to PyPy, Jython and IronPython help keep the CPython project moving forward.

Running Python in the Browser?

For years, there have been attempts to create a sandboxed version of Python that runs in the browser, mainly due to issues with JavaScript. Now that we have CoffeeScript as a good alternative to JavaScript, is it time to make Python run in the browser?

Guido: I gave up on that idea in 1995. And please don't compile Python into JavaScript. Their semantics are very different, and the result would be a Python runtime written in JavaScript, which would be too slow.

Python 3

MetalliQaZ asked: "How do you feel about the current migration to Python 3? From a user's perspective, the transition of some popular libraries is still far off, which hinders this transition. In my professional experience, almost all systems I use don't even have a 3.x interpreter installed. In fact, 2.7 isn't common either. I'd like to hear your thoughts."

Guido: I'm curious where you work. I agree that the migration to Python 3 will take a long time, but if your system isn't even using version 2.7, that's pretty outdated! When I left Google, all transitions to Python 2.7 were complete (after successfully migrating from 2.4 to 2.6 in previous years), and at Dropbox, both client-side and server-side use 2.7. Both companies are considering Python 3.

Regarding the migration to Python 3, I'm actually quite optimistic. Many popular libraries are starting to address this. It will indeed take a long time, but there's been a lot of progress, and in a few years, I hope all code will have migrated to Python 3. Fully eradicating Python 2 usage might take even longer, but then again, hasn't Windows XP refused to die completely too?
