Security and Permissions of Android Applications

by dream2012 on 2012-03-05 12:48:30

Android is a multi-process system, where each application (and part of the system) runs in its own process. Security between applications and the system is enforced at the process level through standard Linux devices, such as the user and group IDs assigned to applications. Additional fine-grained security features are provided by the "permission" mechanism, which can constrain specific operations that a given process can perform.

### Security Architecture

An important design point in Android security is that by default, applications do not have permission to perform actions that could harm other applications, the operating system, or the user. These actions include reading/writing users' private data (e.g., contacts or emails), reading/writing other applications' files, accessing the network, keeping the device awake, etc.

The process of an application is a secure sandbox. It cannot interfere with other applications unless it explicitly declares permissions when it needs to add functionality that the original sandbox does not provide. These permission requests can be handled in different ways, especially based on certificates and user prompts, being automatically allowed or denied. Permission requests are declared statically in the application through an application's manifest, so they will be known in advance at installation time or when there are no changes afterward.

### Application Signing

All Android applications (.apk files) must be signed with a certificate whose private key is held by the developer. The identity of this certificate represents the author of the application. This certificate does not need to be signed by a certificate authority: Android applications are fully permitted and specifically allowed to use self-signed certificates. This certificate is only used to establish trust relationships with applications, not for large-scale control over whether applications can be installed. The most critical aspect is how it affects security by determining access to original signature permissions and the ability to share user IDs.

### User Identification and File Access

Every Android package file (.apk) installed on the device is assigned a unique Linux user ID, and a sandbox is created for it to prevent interference with other applications (or from other applications). The user ID is assigned when the application is installed on the device and remains permanent on that device.

Since security enforcement occurs at the process level, code from different packages cannot run normally in the same process if they need to run under different Linux user identities. You can use the `sharedUserId` attribute of the `` tag in each package's AndroidManifest.xml file to assign them the same user ID. By doing so, the security issue of two packages being treated as the same application is resolved; note that for security reasons, only two applications signed with the same signature (and requesting the same `sharedUserId` tag) will be assigned the same user ID.

Any files created by the application are granted the application's user identifier and, under normal circumstances, cannot be accessed by other packages. When you create a new file using `getSharedPreferences(String, int)`, `openFileOutput(String, int)` or `openOrCreateDatabase(String, int, SQLiteDatabase.CursorFactory)`, you can use the `MODE_WORLD_READABLE` and/or `MODE_WORLD_WRITEABLE` flags simultaneously or separately to allow other packages to read/write this file. When these flags are set, the file still belongs to your application, but its global read, write, and read/write permissions have been set so that any other application can see it.

No checks are made to the user during application execution: either the application is granted specific permissions at installation time and uses the desired features, or it is not granted permission and all attempts to use the features fail without prompting the user.

For example, the `sendBroadcast(Intent)` method checks permissions when data is sent to each receiver after the method call returns, so you won't get an exception when the permission fails. However, in almost all cases, permission failures will be printed to the system log. Typically, multiple permission errors will result in a `SecurityException` being thrown back to the application.

Permissions provided by the Android system can be found in `Manifest.permission`. Each reference can also define and enforce its own permissions, so this is not a comprehensive list of all possible ones.

During program operation, individual permissions may be enforced in some places:

- Preventing an application from performing specific functions when the system receives a call.

- Preventing an application from starting Activities of other applications when launching an Activity.

- Controlling who can receive your broadcasts or send broadcasts to you when sending and receiving Intent broadcasts.

- Accessing and manipulating content providers.

- Binding or starting services.

The `android:description` attribute is optional and is only used to help the system display permissions to the user. Typically, you should set these to a standard system group (listed in `android.Manifest.permission_group`), or in more cases, customize them. It is more preferable to use an existing group to simplify the display of the permission user interface for the user.

Note: Labels (`android:label`) and descriptions (`android:description`) should be provided for each permission. When users browse the list of permissions, they can display character resources such as `(android:label)` or a detailed description of the permission `(android:description)` to the user. The label is short and describes the key function protected by the permission in a few words. The description should be a set of sentences that describe what users can do when granted the permission. Our habit of writing descriptions is in two sentences: the first states the permission, and the second warns the user what bad things might happen if the application is granted the permission.

Here is an example of a label and description for the `CALL_PHONE` permission:

```

directly call phone numbers

Allows the application to call phone numbers without your intervention. Malicious applications may cause unexpected calls on your phone bill.

Note that this does not allow the application to call emergency numbers.

```

You can view the current definition of permissions in the system via the shell command `adb shell pm list permissions`. In particular, the `-s` option displays permissions to the user in a simple, rough manner:

```

$ adb shell pm list permissions -s

All Permissions:

Network communication: view Wi-Fi state, create Bluetooth connections, full Internet access, view network state

Your location: access extra location provider commands, fine (GPS) location, mock location sources for testing, coarse (network-based) location

Services that cost you money: send SMS messages, directly call phone numbers ...

```

### Supporting Permissions in AndroidManifest.xml

Through the AndroidManifest.xml file, advanced permissions can be set to restrict access to all components of the system or the use of the application. All of these requests contain the `android:permission` attribute in the component you need, naming the permission to control access to that component.

#### Activity Permissions (Using `` Tag)

Restricts the components or applications that can start the Activity associated with the permission. This permission is checked during `Context.startActivity()` and `Activity.startActivityForResult()`. If the caller does not have the requested permission, a security exception (`SecurityException`) will be thrown for the call.

#### Service Permissions (Using `` Tag)

Restricts the components or applications that can start, bind, or both start and bind to the associated service. This permission is checked during `Context.startService()`, `Context.stopService()`, and `Context.bindService()`. If the caller does not have the requested permission, a security exception (`SecurityException`) will be thrown for the call.

#### BroadcastReceiver Permissions (Using `` Tag)

Restricts the components or applications that can send broadcasts to the associated receiver. This permission is checked after `Context.sendBroadcast()` returns while the system tries to deliver the broadcast to the relevant receivers. Therefore, permission failure will result in an exception being thrown back to the caller; it will not be delivered to the destination. Similarly, `Context.registerReceiver()` can support a permission to control which components or applications can deliver broadcasts to the registered receiver. Otherwise, when calling `Context.sendBroadcast()`, a permission can be restricted to broadcast receiver objects that are allowed to receive broadcasts (see below).

#### ContentProvider Permissions (Using `` Tag)

Used to restrict the components or applications that can access the data in the ContentProvider. (Content providers have an important additional security facility available to them called URI permissions, described after their calls.) Unlike other components, it has two unrelated permission attributes to set: `android:readPermission` to restrict components or applications that can read the provider, and `android:writePermission` to restrict components or applications that can write to the provider. Note that if a provider's read and write permissions are protected, it means you can only read from the provider, but do not have write permission. When you first retrieve the provider (if you have no permissions, a `SecurityException` will be thrown), the permissions will be checked, and the action you execute on this provider will be performed accordingly.

Request read permission using `ContentResolver.query()`;

Request write permission using `ContentResolver.insert()`, `ContentResolver.update()`, and `ContentResolver.delete()`. In all these cases, a `SecurityException` will be thrown from the caller if the requested permission is not stored.

### Supporting Permissions When Sending Broadcasts

When sending a broadcast, you can always specify a requested permission that allows others to send an Intent to a registered `BroadcastReceiver` except for permission enforcement.

By calling `Context.sendBroadcast()` with a permission string, you request that a receiver application must hold that permission to receive your broadcast.

Note that both the receiver and broadcaster can request a permission. When this happens, both permission checks for the Intent must pass for delivery to the common destination.

### Other Permission Support

Any good granularity permission can be enforced within some callers' services. This is accomplished together with the `Context.checkCallingPermission()` method. Call it and pass a required permission string; it will return an integer to determine whether the current calling process is permitted. Note that these are only used when the call you're executing enters another process, usually through an IDL interface published from a service or some other way notifying another process.

There are many other useful ways to check permissions. If you have the PID of another process, you can use the context's method `Context.checkPermission(String, int, int)` to check permission violations against the PID. If you have the package name of another application, you can use the direct package manager method `PackageManager.checkPermission(String, String)` to see if a specific package is granted the specified permission.

### URI Permissions

So far, the standard permission system described is often insufficient when used with content providers. A content provider protects itself and read/write permissions, but when they come into effect, its direct clients always need to manually specify URIs for other applications. A typical example is attachments in email applications. Access to mail should be protected because it contains sensitive user data. However, if a URL image attachment is provided to an image viewer, the image viewer will not have permission to open the attachment because it has no reason to have a permission that would allow it to access all emails.

The solution to this problem is through URI permissions: when starting an activity or returning a result to an activity, the caller can set one or both of `Intent.FLAG_GRANT_READ_URI_PERMISSION` and `Intent.FLAG_GRANT_WRITE_URI_PERMISSION`. This allows the receiving activity to access the specific data URI in the Intent, regardless of whether it has permission to access data in the corresponding content provider.

This mechanism allows a common functional model that enables user interaction (opening an attachment, selecting a contact from a list, etc.) driven by ad-hoc granular permission grants. This may be a major device feature that applications need to reduce this permission requirement, relating only directly to their behavior.

Granular URI permission grants work, however, requiring some collaboration and content providers to maintain those URIs. It is strongly recommended that content providers implement this mechanism and declare support for it through the `android:grantUriPermissions` attribute or `` tag.
