Cracking Windows EFS Decryption

by xloong on 2008-01-17 14:33:11

The EFS (Encrypting File System) encryption is a type of encryption technology based on the NTFS disk technology. EFS encryption is based on public-key strategy. When using EFS to encrypt a file or folder, the system first generates a FEK (File Encryption Key), which consists of pseudo-random numbers. Then it uses the FEK and the DESX algorithm to create the encrypted file and stores it on the hard drive while deleting the unencrypted original file. Next, the system encrypts the FEK with your public key and stores the encrypted FEK within the same encrypted file. When accessing the encrypted file, the system first decrypts the FEK using the current user's private key, then uses the FEK to decrypt the file. The first time you use EFS, if the user does not yet have a public/private key pair (commonly referred to as a key), the system will first generate the key, then encrypt the data. If you log into a domain environment, the generation of the key depends on the domain controller; otherwise, it depends on the local machine.

Although this sounds very complicated, in practice, it isn't that troublesome. The user validation process for EFS encryption occurs when logging into Windows. Once logged in, you can open any authorized encrypted file. In other words, the EFS encryption system is transparent to the user. This means that if you encrypt some data, your access to that data will be fully allowed without any restrictions. However, unauthorized users attempting to access your encrypted data will receive an "access denied" error message.

My computer is generally not used by others, and I often reinstall the system, but I'm too lazy to back up the keys, so I've never used the EFS feature in Windows 2003 or Windows XP. Today, I read some posts about people losing their EFS keys and being unable to recover their data, so I suddenly came up with an idea to try and break the EFS encryption.

I constructed my test environment on an NTFS disk in a Windows XP Pro SP2 system, creating a test folder with EFS encryption enabled. Inside the folder is an encrypted text file, 1.txt. Now I'll first attempt to read this file from another account, then attempt again in a second system (equivalent to reinstalling the system without certificates).

Step one: Enable the GUEST account on my system.

At this point, it's impossible to access the test folder from the Explorer.

Open cmd, terminate the explorer.exe process in Task Manager, and try to log in as SYSTEM using PsExec.

Failed. Prompted that the process could not be created. It seems there aren't enough permissions.

Return to the administrator account, create a new administrator account named test, and log in with it.

In the test account, running Explorer allows access to the test folder but cannot open the 1.txt encrypted file.

At this point, attempt to log in as SYSTEM using the previous method. When opening the file, it appears as gibberish!

Run IceSword.exe, locate the test folder under Files. Right-click and choose 1.txt, copy it to the desktop with any filename but keep the extension unchanged.

Double-click to open the file, and it reads normally! First step of breaking EFS succeeded!

Step two: Log into a Windows Server 2003 SP1 system (as an administrator).

Use the aforementioned method to copy 1.txt to the desktop again. After opening, it shows gibberish, consistent with the situation when SYSTEM reads it. Second attempt failed.

Summary:

Significance of this method:

Currently, it is only applicable for viewing files encrypted by EFS by other users in the system (please readers do not engage in illegal activities or actions harmful to others' rights!). File recovery after system reinstallation or loss of private keys requires further exploration.

Two software tools used in this method:

PsExec and IceSword. The former is a very popular remote control software abroad with a command-line interface. The latter is IceSword, a well-known domestic hidden process inspection software made by PJF.

Conditions applicable for this method:

1. Requires sufficient permissions to run the above two software programs (if combined with the net user command, it shouldn't be difficult, just a small hint, readers should exercise self-discipline ^_^).

2. The system still has the corresponding key for the EFS-encrypted file (this condition is based on my preliminary speculation).

Brief analysis of why this method succeeds:

1. Utilizes the kernel-level privileges unique to the SYSTEM account, which may be the condition for reading the keys of administrators or other normal users.

2. Special technology of IceSword for reading encrypted files.
