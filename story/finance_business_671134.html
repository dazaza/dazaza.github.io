Four Criteria for Evaluating Virtual Hosting Servers

by xwkj0983 on 2012-02-21 16:23:07

How many virtual hosts can you fit on a single host? This is a common question IT personnel face when considering what hardware to purchase for virtual hosts. In this article, virtualization expert Eric Siebert will explain how to select servers for different types of virtual machines and how to meet the current and future needs of your virtual environment.

You might be able to fit over 100 virtual machines on a single host or just two. The type of applications running on the virtual machines will greatly determine how many virtual machines can reside on a host server. For example, servers with minimal resource requirements such as web, file, and print servers can run more virtual machines compared to resource-intensive servers like SQL and Exchange servers. Generally, we should analyze the performance usage of the current environment to better understand the needs of the virtual environment.

Four Criteria for Evaluating Host Servers

When deciding on server hardware size, there are four main criteria: memory, CPU, network, and disk resources. Let’s start with memory, which is usually consumed first on host servers.

Memory: When it comes to calculating how much RAM to put on a host server for virtual machines, I recommend as much as possible.

However, when it comes to allocating memory to virtual machines, the situation is completely opposite—less is more. Only allocate the memory that a virtual machine actually needs. Typically, for physical servers, allocating more memory ends up being wasted. For virtual machines, increasing RAM at any time is simple, so start with a small amount of memory that meets immediate needs and increase as demand grows. It's easy to allocate more physical memory than the host actually has, but if the host runs out of memory, the virtual machine may swap to disk, risking performance degradation.

CPU: With the advent of multi-core CPUs, adding more CPUs to host servers has become easier and more affordable. Nowadays, almost all servers have two or four cores per physical CPU. A general rule is that each CPU core can support four single-CPU virtual machines. This number varies based on the average CPU usage of the applications running on the virtual machines, potentially ranging from 12 per core to a maximum of 8-10 virtual machines per core.

There is often a misconception about virtual servers: that virtual machines can use as many CPU megahertz as the total available. For example, a dual-core 2.6GHz server with four CPUs has a total of 20800 megahertz (i.e., 8 x 2.6 GHz) of resources. However, a virtual machine with one virtual CPU cannot use more megahertz than a single CPU or core provides. If a virtual machine has two virtual CPUs, it cannot use more than the maximum megahertz of each CPU or core. The number of cores needed also depends on whether virtual machines use multiple virtual CPUs.

Typically, you should have more cores than the maximum number of virtual CPUs allocated to a single virtual machine. For example, to run a virtual machine with four virtual CPUs, do not purchase a dual-core, dual-processor server because such a server only has four cores. This is because the hypervisor's CPU resource scheduler needs to find four idle cores simultaneously every time the virtual machine requests CPU resources. If there are only four available cores in total, performance will degrade. I recommend using quad-core CPUs because more cores provide more flexibility for CPU scheduling.

Network: The number of network interface cards (NICs) required for virtual servers varies based on redundancy needs, whether network storage is used, and the selected features. Using 802.1Q VLAN tagging provides the flexibility to use multiple virtual local area networks (VLANs) on a single NIC, eliminating the need for a separate NIC for each VLAN on the host server. For small servers, you should be able to use two NICs, but ideally, there should be at least four NICs on the host server. If you use network storage like iSCSI, it's best to have more than four NICs, especially if you plan to use VMware's vMotion feature. When creating switches, it's best to assign multiple NICs to them to provide redundancy and increase available capacity for virtual machines.

Disk: Finally, you need to evaluate disk resources. There are several options to consider, depending on your budget and whether you have a storage area network (SAN) in your environment. Using local disks is the cheapest option, but it does not offer shared storage with advanced features between virtual host servers, such as vMotion. SAN (Fibre Channel) disks generally provide the best-performing disk solution, but they are usually the most expensive. Network disks are a good alternative, offering performance close to that of SANs. Additionally, using 15K hard drives performs better than using 10K hard drives, but having larger RAID groups is also important, as it helps spread disk I/O across drive spindles as much as possible.

When deciding how many disks to purchase, ensure all virtual machines have enough disk resources available, and add an extra 10-20% disk space for additional virtual machine files and snapshots. If you plan to use snapshots frequently, you may need even more disk space. In most cases, disk resources are combined with host usage, such as storing development and testing virtual machines on local disks and production virtual machines on shared storage.

In general, you want your virtual machines to use at least 80% of the capacity of the virtual host server to maximize your investment. However, leave enough capacity for future growth and ensure there are sufficient resources to support additional virtual machines in case of a host failure. More capacity is always better, as it avoids resource contention and the need to purchase additional virtual host servers.
