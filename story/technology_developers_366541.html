Routing loop resolution

by fdisk54 on 2010-09-29 11:14:25

One evening, I asked a question in the group: "Why does a route become unreachable after 16 hops?" Some people answered that it was due to the fact that networks at that time were not as large as they are now. Others said it was simply a rule. Summarizing this, the main reason is to prevent routing loops. Below are some methods to solve this problem:

When maintaining routing table information, if topology changes result in slow network convergence and conflicting or inconsistent routing entries, a routing loop issue may occur. Under these conditions, routers ignore routes to unreachable networks, causing user packets to endlessly circulate on the network, ultimately resulting in severe waste of network resources. Therefore, methods to solve routing loop problems have been developed.

The solutions to routing loop problems can be summarized into six main categories:

1. **Defining a Maximum Value**

Distance vector routing algorithms can self-correct through the Time-to-Live (TTL) field in the IP header, but routing loop problems might initially require an infinite count. To avoid this delay issue, distance vector protocols define a maximum value, which represents the largest metric value (with the maximum set to 16), such as hop count. This means that routing update information can be sent up to 15 times to routers for unreachable networks. Once the maximum value of 16 is reached, the network is considered unreachable and faulty, and no further routing updates for accessing that network will be accepted.

2. **Split Horizon Technique**

A method to eliminate routing loops and accelerate network convergence is achieved through the "split horizon" technique. The rule is not to send routing update information back in the direction from which the original routing update came (understood as unidirectional updates and feedback). For example, with three routers A, B, and C, after B learns the path to network 10.4.0.0 from C, it no longer advertises to C that it can access the 10.4.0.0 network via C. Similarly, after A learns the path to the 10.4.0.0 network from B, it no longer advertises this path back to B. If the 10.4.0.0 network becomes inaccessible due to a fault, C sends routing updates to A and B indicating that the network is unreachable but will not relearn incorrect information about reaching 10.4.0.0 from A or B.

3. **Route Poisoning**

Defining a maximum value partially solves the routing loop problem but does not completely eliminate it. Before reaching the maximum value, routing loops can still exist. Route poisoning can fully resolve this issue. Here's how it works: Assume there are three routers A, B, and C. When the 10.4.0.0 network fails and becomes unreachable, router C sends relevant routing updates to neighboring routers and sets its metric value to infinity, informing them that the 10.4.0.0 network is unreachable. Router B, upon receiving the poisoned message, marks the link in its routing table as infinity, indicating that the path has failed, and advertises this to neighbor A. In this way, each router is poisoned sequentially, informing neighbors that the 10.4.0.0 network is invalid and no longer accepting updates, thus avoiding routing loops.

4. **Poison Reverse (also known as Reverse Poisoning)**

Continuing with the example, when router B sees that the metric to the 10.4.0.0 network is infinity, it sends a "poison reverse" update to router C, indicating that the 10.4.0.0 network is unreachable. This is an exception to split horizon, ensuring that all routers receive the poisoned routing information.

5. **Controlling Update Timing (i.e., Holddown Timer)**

The holddown timer prevents regular update messages from resetting a broken route at inappropriate times. It tells the router to temporarily hold any changes that might affect the route for a period longer than the time it takes for update messages to propagate across the entire network. When a router receives an update from a neighbor indicating that a previously accessible network is now unreachable, it marks the route as unreachable and starts a holddown timer. If it receives another update from a neighbor with a better metric than the original path during this time, it marks the route as accessible and cancels the holddown timer. If it receives updates with worse metrics from different neighbors before the holddown timer expires, these updates are ignored, allowing more time for the update information to propagate across the entire network.

6. **Triggered Updates**

Normally, routers periodically send their routing tables to neighboring routers. Triggered updates, however, immediately send routing updates in response to certain changes. Routers detecting a network failure immediately send an update to neighboring routers, which then trigger updates to their own neighbors, ensuring that all routers on the network receive the update information in the shortest possible time, quickly understanding the changes in the network. However, this approach also has potential issues, as update packets may be lost or corrupted on certain links in the network, preventing other routers from receiving the triggered update in time. Thus, a combination of triggered updates with holddown timers is used. The holddown rule requires that once a route becomes invalid, any routes to the same destination with the same or worse metric values within the holddown time will be ignored. This allows the triggered update to propagate throughout the network, preventing damaged routes from being reinserted into neighbors that have already received the triggered update, thereby solving the routing loop problem.

The above-discussed solutions can work simultaneously to prevent routing loops in more complex network designs.
