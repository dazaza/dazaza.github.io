Analysis of the Working Principle of Penetrating Restoration (Reverse Engineering) -- Machine Dog

by xloong on 2007-11-24 19:52:45

Originally, I saw the virus named "Machine Dog" on the Long Group forum, but since it was a Jiangmin virus report, which was obviously formulaic, I didn't pay much attention to it. However, I just saw a related analysis on Gui Zai's blog, so I've decided to repost it here.

Gui Zai's Note: The Machine Dog, IGM, and the tool for write-through restoration (which can now be detected as a virus by Kaspersky) were seen at Wudi Xiaolong's place, and I added a backup address.

Sample Unpacking:

Load the sample explorer.exe in OD,

Set a breakpoint on GetModuleHandleA, with NULL parameter being the entry point call of this function.

After exiting CALL, you can obtain the entry point at 004016ED.

Reload the sample, set a memory-write breakpoint at 004016ED, after interruption, step over one step, then set a breakpoint at 004016ED, press F9 to run to the entry point, DUMP. After dumping, don't close OD, keep the sample suspended, use ImportREC to repair the import table of the dumped file.

After repairing, if loading the dumped file in OD fails, rebuild the PE using PEDITOR's rebuilder function, and then it can be loaded in OD, indicating that the unpacking is basically successful, but there are still issues with the resource section, unable to view with ResHacker.

Extraction of pcihdd.sys:

Load the sample explorer.exe in OD, set an interrupt when a new module is loaded, press F9 to run.

When ADVAPI32.DLL is loaded, set a breakpoint on CreateServiceA, press F9 to run.

When CreateServiceA interrupts, pcihdd.sys can be extracted.

The basic flow of pcihdd.sys is as follows:

1) Check the addresses of IDT's 09 (NPX Segment Overrun) and 0E (Page Fault) handlers.

If the 09 interrupt handler exists, and the high 8 bits of the handler address are different from the high 8 bits of the 0E handler, set the high 16 bits of IDT 0E to 0. It is estimated to check whether 0E has been HOOKed.

I find this operation dirty and hard to understand. Wouldn't this cause BSOD? Please let those who understand post a reply.

2) Search addresses to find its own loading address.

Search the driver file's resources for 1000/1000, and copy them to a global buffer.

3) Create \Device\PhysicalHardDisk0 and its symbolic link \DosDevices\PhysicalHardDisk0.

4) Only respond to IRP_MJ_CREATE

IRP_MJ_CLOSE

IRP_MJ_DEVICE_CONTROL

In IRP_MJ_CREATE, detach the device attached to \Device\Harddisk0\DR0. This operation will make disk filter drivers, file system drivers (provided by OS, but some antivirus software also monitors the file system through this channel), and file system filter drivers (most file access control and monitoring are at this level) invalid (see http://blog.csdn.net/joshua_yu/archive/2006/02/04/591636.aspx).

Recover attachments on DR0 in IRP_MJ_CLOSE.

Respond to 0xF0003C04 in IRP_MJ_DEVICE_CONTROL. Just decrypt the resource data found in step 2 and return it to the application. The decryption key is generated by looking up a string (key seed?) passed by the application (KEY: 0x3f702d98).

The role of 0xF0003C04:

Perform a checksum-like operation on the entire code body passed from user mode using it as a key seed, resulting in a 4-byte decryption KEY. Then use this decryption key to decrypt the driver's carried resources (just XOR), and return the decrypted result to user mode.

Regarding detaching devices on DR0:

This operation should affect the normal file system operations of the system, but because the interval between opening and closing this driver during actual operation is very short, there should be no obvious impact.

Flow of explorer.exe:

1. Release pcihdd.sys from resources and create a service named pcihdd, start the service.

2. Locate the position of userinit.exe on the hard disk. The method is as follows:

1) Obtain the distribution information of the file data through FSCTL_GET_RETRIEVAL_POINTERS.

2) Directly access the MDR and the boot sector of the first partition of the hard disk (\\\\.\\PhysicalHardDisk0) to get the partition parameters (sectors per cluster), and use the information obtained in 1) to determine the absolute offset of the file on the hard disk.

Here is a small bug, the sector size uses a fixed 512 bytes instead of obtaining it from the boot sector.

3) Compare the file data read by ReadFile and the file data obtained directly after self-positioning to confirm the correctness of the positioning.

3. Pass the entire code body as a parameter to pcihdd.sys, control code 0xF0003C04, and directly write the data returned by pcihdd into the first cluster of userinit.exe.

Modified userinit.exe:

1) Query the Shell key value under SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon.

2) Create a Shell process.

3) Wait for network connection, when the network connection is smooth, download the list from http://yu.8s7.net/cert.cer.

4) For each file in the list, create a new thread to download and execute, increment the thread count (INC).

5) Wait for all threads to end (thread count is 0) and terminate the process.

The operation on the thread count is not atomic, theoretically, there is a small probability of problems in multi-CPU situations.

However, this is a virus targeting ordinary PCs, multi-CPU is uncommon, and stability is not required.
