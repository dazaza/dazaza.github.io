We don't need any documents outside the code.

by anonymous on 2013-07-24 14:46:53

　　English Original: We Don’t Need That Documentation

　　"You need to write more detailed documentation." Did you hear that? I’ve heard it, many times, in many companies. Most people feel guilty about not writing enough documentation and think they should be writing more of it. But I don't.

　　There are two types of documentation — inside the code and outside the code. In-code documentation includes javadoc (or any language-specific tool used to describe classes and class methods) and code comments. External documentation includes product descriptions and internal materials.

　　The biggest problem with external documentation: it becomes outdated. Keeping it in sync with the code is a tedious and time-consuming task.

　　The second biggest problem with external documentation: no one really uses it.

　　Programmers write code. They are good at reading code. Code has special meaning for them. Writing explanations for important classes and adding comments within methods for significant scenarios is considered a best practice in programming, and good programmers do this. This makes the code more understandable for others, and when combined with self-explanatory coding practices, it forms a complete and functional documentation. It doesn't require separate updates (you update it as you modify the code).

　　Do we need external documentation? Maybe. There should be a brief description of the overall system architecture to provide context for these codes. What modules exist, and how they interact — that's sufficient, just one page. However, this too can become outdated, but at least it’s easier to maintain, and team leaders and architects should regularly check if it needs updating.

　　If you're a tester or a product manager, you might say you don't understand these programs and need these documents for your work. I may sound a bit rude, but if you need these, you should write them yourself. Programmers are not technical writers; writing external documentation is neither their job nor their interest. If you don't want to write these documents but still want to know what the program does, ask the programmer. They will be happy to explain the comments to you and clarify their functions. If the functionality of the code or the expected behavior isn't clear, check the issue tracking system to see what the requirements were. You don't need external documentation to describe what the code does.

　　Of course, software is used by users, so some user manuals are needed, but those should actually be written by someone else.

　　So, next time someone insists that programmers need to write documentation to describe the functionality of the program, refute them. Stand firm that it's a waste of time and effort, and that having the code and comments is enough. If that's not enough, it means you need better coding standards and habits, not more documentation.
