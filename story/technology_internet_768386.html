Programmer, what's the hardest bug you've ever debugged?

by anonymous on 2013-11-17 17:46:51

Debugging bugs? It's a must for every programmer. There’s a popular question on Quora related to bugs: "What’s the hardest bug you’ve debugged?". Among many responses, Dave Baggett’s experience is the most astonishing, receiving over 2,400 upvotes.

Thank you to @cugbabyebar for the enthusiastic translation.

Recalling this bug still brings me some pain. As a programmer, when you find a bug, you first learn to look for issues in your own code. After testing ten thousand times, you might blame the compiler. Only after all that doesn’t work do you start blaming the hardware.

This is my story of encountering a hardware bug.

Without going into details, I once wrote save card (read/write) code for *Crash Bandicoot*. For an arrogant game programmer, it was as easy as a walk in the park, and I thought it would take only a few days. But debugging got interrupted for six weeks. During this time, I worked on other things but kept coming back to deal with this bug—spending several hours a day for days. The bug was really annoying.

The symptom of the bug was that when you needed to save your progress, the code accessed the memory card, and most of the time, there were no problems... but occasionally, the read/write would timeout... with no apparent reason. A short write operation often corrupted the memory card. Players wanted to save their progress, but not only did we fail to save, we also erased everything on their memory cards. Oh no.

After a while, our producer at Sony, Connie Booth, panicked. We obviously couldn’t release the game with this bug, and after six weeks, I still had no clue where the problem was. Through Connie, we sought help from other PS1 developers: has anyone encountered something like this? No. Absolutely no one had any issues with the memory card system.

After exhausting your mind, the only debugging method left is divide and conquer: gradually removing parts of the code until only a small amount remains, yet the problem persists. Like wood carving, you remove the parts that are not problematic, leaving behind the source of the bug.

The challenge here is that video games are hard to strip down part by part. How do you run the game after deleting the code for simulating gravity or displaying characters?

What you need to do is replace entire modules with something that pretends to do the real thing but actually does something very simple that won’t cause the bug. You have to write new support code to make these things work properly. It’s a slow and painful process.

To cut a long story short: I did it. I removed large chunks of code, quite a lot, leaving only the initialization code—just preparing the game runtime system, initializing the underlying hardware, etc. Of course, I couldn’t display the load/save menu because I had removed all the graphics code. But I could pretend the user used (invisible) load/save screens and requested saves, then write to the card.

In the end, I ended up with a small amount of code that still had the bug—but the problem still occurred randomly! In most cases, there were no issues, but occasionally it failed. Basically, all of Crash’s actual code was removed, but it still happened. It was completely baffling: the remaining code didn’t seem to be doing much.

At that point—probably around 3 a.m.—an idea popped into my head. Read/write (I/O) involves precise timing. Whether it’s a hard drive, memory card, or Bluetooth transmitter—anything—the underlying code for reading/writing is based on clocks.

Clocks synchronize hardware devices not directly connected to the CPU with the code running on the CPU. Clocks determine the baud rate—the rate at which data moves from one end to the other. If there’s a timing issue, either the hardware, software, or both will malfunction. This is really, really bad and usually leads to data corruption.

What if our initialization code somehow messed up the timing? I reviewed the timing-related code in the test program again and noticed that we had set the PS1's programmable timer to 1kHz (1000 ticks per second). That's pretty fast; when the PS1 boots, the default state is about 100Hz. Therefore, most games set their timers to 100Hz.

The lead developer (and the only one besides me), Andy, set the timer to 1kHz to make Crash’s action calculations more accurate. Andy liked overkill—if we were simulating gravity, we should increase the precision as much as possible!

But what if increasing the timer frequency mysteriously interfered with the overall program's timing, thus affecting the baud rate of the memory card?

I commented out the timer code. Then I couldn’t reproduce the bug. But that didn't mean the bug was fixed—it happened randomly. What if I just got lucky?

Days passed, and I kept playing with my test program. The bug didn’t reappear. I went back to the full Crash code, modified the load/save code, reset the programmable timer to the default setting (100Hz) before accessing the memory card, and then set it back to 1kHz. From then on, the problem didn’t reoccur.

But… why?

I returned to the test program, trying to detect the pattern of errors that appeared when the timer was set to 1kHz. Finally, I noticed these errors occurred when using the PS1 controller. Since I rarely did this myself, I hadn’t noticed (why would I use the controller while testing load/save code?). But one day, our artist waited for me to finish testing (I'm sure I was swearing at the time), and he nervously fiddled with the controller. The card got corrupted. “Wait, what happened? Hey, do it again!”

Once I realized these two things were linked, it was easy to reproduce the bug: start writing to the memory card, move the controller slightly, and the card gets corrupted. To me, it was clearly a hardware bug.

I went to Connie and told him about my findings. She relayed them to the hardware engineers who designed the PS1. She was told: “Impossible, it can’t be a hardware issue.” I asked her if I could speak directly to him.

The engineer called me. He spoke broken English, and I spoke even worse Japanese, so we argued for a while. I finally said, “I’ll give you a 30-line test program that reproduces the problem when you move the controller.” He agreed. He assured me it was a waste of time, and he was busy with a new project, but since we were important developers for Sony, he would try it.

The next evening (we were in Los Angeles, and he was in Tokyo, so it was evening for me and the next day for him), he called me back, apologizing sheepishly. It was a hardware issue.

I still don’t fully understand the exact problem, but from what I gathered from Sony headquarters, if you set the programmable timer to a sufficiently high clock frequency, it affects something near the clock crystal on the motherboard. One of those things is the baud rate controller for the memory card, which also sets the baud rate for the controller. I’m not a hardware guy, so I’m quite vague on the details.

But the gist is that interference between two independent parts of the motherboard, combined with data transmission from the controller interface and memory card interface at a 1kHz clock frequency, caused bit loss, leading to data loss and card corruption.

This is the only time in my entire programming career that I debugged a problem due to quantum mechanics.
