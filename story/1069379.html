最小可行治理

2021-07-24 08:51:13

在政策团队中，我经常听到开发人员说起草项目治理是一个痛点。出于这个原因，我们制作了最小可行治理 (MVG)，这是一个简单、易于实施的治理框架，适用于您的免费和开源项目。它应该适用于所有项目——那些在几个朋友或几个大型企业竞争对手中的项目。它是开源的，因此您可以对其进行修改以满足您的需要。当您的项目从 1 个维护者变为 1+N 个维护者时，事情会变得复杂。突然，您需要弄清楚您将如何做出决定，如何添加其他维护者，如何分工并就愿景达成一致，以及谁拥有商标。通常对这些决定的谈判要么太少要么太多。对于志愿者运营的项目，人们通常在出现问题之前什么都不做，而这些问题会使社区紧张或破裂。对于有大型企业贡献者的项目，律师会参与其中，并花费数月时间协商减慢实际工作的重量级法律治理结构。 MVG 是维护者之间签署并存储在存储库中的协议。 MVG 为一组开源项目提供了一个两层结构。在顶层（在 GitHub 上称为“组织”），您选择一组人作为技术指导委员会，就组织所有项目之间的总体方向和协调做出决策。在该顶层之下是各个项目，维护者之间具有轻量级、基于共识的治理。该协议涵盖五项内容，默认设置如下。由于 MVG 是开源的，您当然可以修改文档以满足您的需要。做决定。 MVG 在组织层面默认采用基于共识的决策过程。共识并不意味着不反对——它意味着利益相关者基于其主导观点的松散同意，并考虑到突出的反对意见。这就是构建网络的方法。根据我们的经验，争取共识可以减少摩擦并减少敌对的项目分叉。在组织/指导委员会层面，无法通过共识做出的决定会退回到多数票。单个存储库仅达成共识，因此如果无法通过共识做出决定，则可以向指导委员会提出上诉。商标政策。中立的商标所有权是真正开放的协作过程的有力指标。当项目有一个非营利性公司主页时，该公司主页拥有商标（如项目名称和徽标），以便为更广泛的社区的利益进行管理。如果没有公司住所，商标所有权就很复杂。 MVG 通过让与项目相关的每个人都同意受与与项目无关的相同商标政策的约束来解决这些复杂问题。 MVG 默认商标政策基于示范商标指南。

反垄断政策。该政策明确规定，使用 MVG 进行项目合作的参与者不会从事违反反垄断或竞争法的活动。这是良好的法律卫生，特别是如果有来自公司的维护者。行为守则。 MVG 默认为贡献者契约。这是 GitHub 开源社区使用最广泛的行为准则。项目标准。 MVG 适用于 GitHub 上的开源软件、开放标准、开放硬件和开放数据项目。默认情况下，MVG 包含一项策略，项目必须从有限的开源许可证列表中进行选择。对于开源项目，这些是 Open Source Initiative 的流行许可证列表，MVG 的模板项目许可证是 MIT——使用最广泛的开源许可证。 MVG 默认为标准的社区规范或开放 Web 基金会协议和开放知识基金会的数据推荐一致性许可列表。 MVG 的设计既简单又强大。一旦您查看并确保它适合您，请复制一份副本，填写空白，然后开始工作。 MVG 是开源的，遵循 CC-BY 许可，因此您可以修改文档以满足您的需要并重新共享它们。请记住，这些是合同，因此您应该确保它们适合您。 MVG 是一个入口匝道。如果您的项目取得成功并需要为聚会、会议演讲、托管费或其他任何事情存钱，则很容易将此结构转变为免费和开源软件基金会或其他公司形式。 MVG 仍处于测试阶段。我们需要您的反馈！有几个 alpha 版本正在使用中，但与所有事情一样，还有改进的余地。在我们发布 1.0 版本之前，我们将在接下来的几个月内在 MVG 存储库中接受任何人的反馈。最后，感谢许多 GitHub 团队成员以及在此 Beta 版发布之前审核并提供反馈的其他人，尤其是 Pam Chestek 和 Aaron Williamson。
