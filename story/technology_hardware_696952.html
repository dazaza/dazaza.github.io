Use the Dalvik Debug Monitor Service (DDMS) tool

by dream2012 on 2012-03-08 11:42:21

Android comes with a debugging tool called Dalvik Debug Monitor Service (DDMS), which provides port forwarding services, screen capture on the device, thread and heap information on the device, logcat, process, and wireless status information, call and SMS spoofing, location data spoofing, and more. This article only introduces the general features of DDMS and does not provide detailed descriptions of its features and functions. DDMS is located in the SDK's tools/ directory. To run it, navigate to this directory from a terminal or command prompt and type ddms (or ./ddms on Mac/Linux). DDMS will run on the emulator or a connected device; if both the emulator and a connected device are running, DDMS defaults to running on the emulator.

* How DDMS Works

DDMS acts as an intermediary that connects the IDE to applications running on the device. Each Android application runs in its own process, managing its own virtual machine (VM), and each process listens for a debugger on a different port. When it starts, DDMS connects to adb and establishes a device monitoring service between them. When devices are connected or disconnected, they notify DDMS. A virtual monitoring service is established between adb and DDMS, notifying DDMS when a VM on the device starts or terminates. Once the VM is running, DDMS obtains the process ID through adb and establishes a connection to the VM debugger via the adb daemon (adbd) program on the device. DDMS can communicate with the VM using a custom wire protocol.

On each VM on the device, DDMS opens a port to listen for debuggers. For the first VM, DDMS listens on port 8600, the next one on port 8601, and so on. When a debugger connects to one of these ports, all communication is forwarded between the debugger and the corresponding VM. DDMS can operate like a remote debugging session.

DDMS also opens another local port, the DDMS "base port" (default 8700), which also serves as the listening port for the debugger. When a debugger connects to this port, all traffic currently selected in DDMS is forwarded to the VM, making it the default connection port for the debugger.

Tip: You can set the DDMS configuration port in the File > Preferences menu. The configuration is saved in the "$HOME/.ddmsrc" file.

Understanding Dalvik Debugging Knowledge

Debugging applications on the Dalvik Virtual Machine is similar to debugging on other VMs. However, when stepping out of synchronized code, the "current line" cursor jumps to the last line of the method.

* Left Frame

The left side of the Debug Monitor displays the current running status of each emulator/device, showing each emulator/device's VM in a list. The VM is uniquely identified by the package name of the application running on it. Use this list to find and bind the process you want to debug to the VM. Near the VM name in the list is a "debug communication" port (in the far right column). If you connect a debugger to one of the ports in the list, you will be connected to the corresponding VM on the device. However, when using DDMS, you only need to connect to port 8700, at which point DDMS forwards all traffic here to the currently selected VM (Note: When you select a VM in the list, the listed port includes 8700), so you don't have to reconfigure the debugger's port every time you switch VMs.

When an application running on the device calls waitForDebugger() (or you select this option in developer settings), a red icon will appear next to the client name while it waits for the debugger to bind to the VM, and when a debugger connects, the icon turns green.

If you see a small bug icon with a strike-through, it means that DDMS cannot establish a connection between the debugger and the VM because it cannot open the local port of the VM. If you see this icon on all VMs on the device, it is likely because another instance of DDMS is running (including the Eclipse plugin).

If you see a question mark in an application's package space, it means that DDMS received the application's PID from adb but somehow could not handshake with the VM process. Try restarting DDMS.

* Right Frame

On the right side, the Debug Monitor displays useful information and some pretty cool tools via tabs.

- Information

This view shows general information related to the currently selected VM, including process ID, package name, and VM version.

- Threads

The Thread view has a list of threads running on the target VM. To reduce the amount of data sent, only thread update information is sent when the "Threads" button on the toolbar is pressed. This button corresponds to each VM. This list contains the following information:

ID - A unique thread ID assigned by the VM. In Dalvik, these start from 3 and are odd numbers.

Tid - Linux thread ID. As the main thread of a process, it equals the process ID.

Status - VM thread status. Daemon threads are displayed with a star and will be one of the following states:

running - Executing application code

sleeping - Calling Thread.sleep()

monitor - Waiting to acquire a monitor lock

wait - In Object.wait()

native - Executing native code

vmwait - Waiting for a VM resource

zombie - Thread in a dead process

init - Thread is initializing (you won't see this step)

starting - Thread is preparing to run (you won't see this either)

utime - Total time executing user code, in "jiffies" (usually 10ms), available only on Linux.

stime - Total time executing system code, in "jiffies" (usually 10ms).

Name - Name of the thread.

"ID" and "Name" are set when the thread starts and retained areas are updated periodically (default is every 4 seconds).

- VM Heap

Displays some heap state information, updated during garbage collection. If a VM is selected, the VM Heap view prompts to disable heap updates. Click the "Show Heap Updates" button in the top-left toolbar to return to the VM Heap view, click Cause GC to perform garbage collection and update the heap state.

- Allocation Tracking

In this view, you can track memory allocations for each VM. Select a VM on the left, then click Start Tracking, then click Get Allocations to view memory allocation information since tracking started. Corresponding data will be displayed in the table below, click it again to refresh the list.

- Competition Control

Through these controls, you can simulate special device states and activities, including the following features:

Phone State - Change the voice phone state and data plan (local, roaming, searching, etc.) and simulate different types of network speeds and delays (GPRS, EDGE, UTMS, etc.)

Phone Activity - Simulate phone calls and SMS messages to the emulator.

Location Control - Send false location data to the emulator, allowing you to form a positioning operation like GPS mapping.

To use Location Control, start your application in the Android emulator and open DDMS, click the Emulator Control tab and press Location Control. Afterward, you can:

Manually send individual longitude/latitude coordinates to the device.

Click Manual, select coordinate format, fill in data in the fields and click Send.

Use a GPX file to describe path playback on the device.

Select and load a GPX file, once loaded, click the Play button to replay the path of the location operation program.

When starting playback from GPX, you can adjust the playback speed and control playback via pause and skip buttons on the DDMS panel. DDMS will parse track points (in the first table) and tracks (in the second table, via simple field combinations to support multi-segments), only tracks can be played. Clicking a track point in the first list sends only its coordinates to the device, after selecting a track, you can play it.

Use a KML file to describe sequential playback of location points on the device.

Click and open a KML file, once the file is loaded, click the Play button to send coordinates to your location program.

When using a KML file, it parses an element whose value consists of a set of longitude, latitude, and altitude values. Example: -122.084143,37.421972,4

In your file, you can include multiple elements, each containing an element. When you do this, the set of location markers will be added to the track, and DDMS will send one location marker per second to the device.

A suitable way to generate a KML file is: Find a location on Google Earth, right-click this location point, select the "Save Location As" menu item in the pop-up menu on the left, and choose the storage format as Kml.

Note: DDMS does not support creating paths via lat1, long1, lat2, long2, ... methods. Similarly, nodes in the middle are currently unsupported. Future versions may support time-based locations and paths with a single location element.

For more location data spoofing methods, refer to the Location-Based Services APIs documentation.

* File Browser

Using the File Browser, you can browse the device file system and perform basic management, such as dragging and pulling files, which revolves around the drag-and-drop commands of adb, providing a GUI experience.

Open DDMS, select Device > File Browser... to open the File Browser window. You can drag files into the device directory but cannot drag them out. To copy files from the device, select the file and click the Fetch File from Device button on the toolbar. To delete files, use the Delete button on the toolbar.

If you are interested in the SD card image on the emulator, you can create an image using the mksdcard command and load it when starting the emulator. For example, in the /tools directory, execute:

```bash

$ mksdcard 1024M ./img

$ emulator -sdcard ./img

```

Now, when the emulator is running, the DDMS File Browser will be able to read and write to the sdcard directory. However, your files will not automatically appear. For example, if you add an MP3 file to the sdcard directory, the media player will not see this file until you restart the emulator. (When restarting the emulator from the command line, make sure to reload the sdcard.)

* Screen Capture

You can capture a screen image from the device or emulator by selecting Device > Screen Capture from the menu or pressing the CTRL-S shortcut.

* Browse Processes

You can get the same output as using the ps -x command on a specific VM by selecting Device > Show Process Status from the menu bar.

* Trigger GC

Press the Trash Can button on the toolbar to trigger garbage collection.

Run Dumpsys and Dumpstate on the Device (logcat)

From Dalvik, run dumpsys (logcat) by selecting Device > Run logcat... from the menu bar.

From Dalvik, run dumpstate by selecting Device > Show Device Status... from the menu bar.

* Test Wireless Status

By default, on a standard logcat (which contains a lot of information), there is no output for wireless status. To view wireless information, click Device > Show Wireless Status... from the menu, or run the logcat described in Record Wireless Information.

* Stop a VM

You can stop a VM by selecting Action > Suspend VM from the menu, which causes the VM to call System.exit(1).

* Learn About DDMS Defects

DDMS has the following known defects:

If you connect and disconnect a debugger, after DDMS drops and reconnects to the client, the VM thinks the debugger has closed. This will eventually be fixed.
