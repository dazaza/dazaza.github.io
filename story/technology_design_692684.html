Development Exercise of Notepad Application (Part I)

by dream2012 on 2012-03-06 11:10:40

In this exercise, you will learn how to add a second Activity to your notepad application. This Activity allows users to create, edit, and delete notes. It assumes that a new note is created in response to user input and packages itself into a return value Bundle provided by an intent.

**Objective:**

- Build a new Activity and add it to the Android manifest.

- Call another Activity using the `startActivityForResult()` method asynchronously.

- Pass data between Activities using various Bundle objects.

- Learn how to use a more advanced screen layout.

If you encounter errors related to AndroidManifest.xml or android.zip files, right-click on the project and select **Android Tools > Fix Project Properties** to resolve them.

The member variable used to store the note pointer is called `mNotesCursor`. The prefix 'm' indicates it's a member variable, adhering to Android coding style standards.

- Implement two new overridden interfaces: `onListItemClick()` and `onActivityResult()`.

mDbHelper.deleteNote(getListView().getSelectedItemId());

Here, we delete the specified ID note through the `deleteNote()` interface. To get the specified note ID, we call `getListView().getSelectedItemId()`. Then, we call `fillData()` to keep the data updated. The full definition of the `onMenuItemSelected()` interface is as follows:

**Step 4:**

Implement the `createNote()` interface:

Create a new Intent with the class `NoteEdit` to create a note (ACTIVITY_CREATE). Then call `startActivityForResult()` to start the Intent:

```java

Intent i = new Intent(this, NoteEdit.class);

startActivityForResult(i, ACTIVITY_CREATE);

```

This form of Intent call is specific to a particular class within our Activity, in this case, `NoteEdit`. Since the Intent class needs to send request information through the Android operating system, we must provide a Context (`this`). The `startActivityForResult()` interface is triggered after the completion of a new Activity and is then called in our Activity. The interface in our Activity that receives the callback is named `onActivityResult()`, which we will implement later. Another interface named `startActivity()` is also available, but it is a fire-and-forget approach where the current Activity does not receive a notification when the other Activity completes, and no result information can be returned via the `startActivity()` interface.

Don't worry about the fact that `NoteEdit` doesn't currently exist; we will address that shortly.

**Step 5:**

Implement the overridden `onListItemClick()` interface:

`onListItemClick()` is a callback function that we will override. It is called when the user clicks on an item in the note list. It has four parameters: the ListView object calling the interface, the clicked View (contained within the ListView), the position of the clicked item (its location in the list), and the row index of the clicked item (`mRowId`). In this example, we ignore the first two parameters (since we can only have one ListView at the moment) and the row index parameter. Our focus is entirely on the position of the selected item in the list. Through this parameter, we can obtain the desired data from the corresponding row and send this data packed to the `NoteEdit` Activity.

In the implementation of this callback function interface, we create an Intent with the class `NoteEdit` to modify the note content, then add data to the additional package passed as a parameter to the called Activity's Intent. We use it to pass the title, body, and row index of the note being edited. Finally, it calls `startActivityForResult()` to start the Intent. Here is the complete code for the `onListItemClick()` interface implementation:

```java

super.onListItemClick(l, v, position, id);

Cursor c = mNotesCursor;

c.moveToPosition(position);

Intent i = new Intent(this, NoteEdit.class);

i.putExtra(NotesDbAdapter.KEY_ROWID, id);

i.putExtra(NotesDbAdapter.KEY_TITLE, c.getString(

c.getColumnIndexOrThrow(NotesDbAdapter.KEY_TITLE)));

i.putExtra(NotesDbAdapter.KEY_BODY, c.getString(

c.getColumnIndexOrThrow(NotesDbAdapter.KEY_BODY)));

startActivityForResult(i, ACTIVITY_EDIT);

```

`putExtra()` is an interface that adds data to the additional package when triggering an Intent. Here, we use the package to pass the title, body, and row index of the note we want to edit.

When we locate the details of the selected object in the corresponding position in the list through `moveToPosition()`, all the details of the note are available in the query pointer.

By adding information to the Intent and using `startActivityForResult()`, we trigger the Intent and request type code (the request type code will be returned as a parameter `requestCode` to the `onActivityResult()` interface) in the `NoteEdit` class.

**Step 6:**

Open the `note_edit.xml` file and browse through it. This is the code for the note editing interface. This is the most complex interface layout we've dealt with so far. The file provided here avoids issues with fading in and out during code formatting. (XML has very strict rules regarding case sensitivity and structure, and these errors are often the root cause of screen layout problems.)

A new parameter we haven't encountered before is used here: `android:layout_weight` (in this case, set to 1 in each instance). `layout_weight` assigns importance values to multiple views in a linear layout. All views have a `layout_weight` value, defaulting to zero, meaning they take up as much screen space as needed. If a value greater than zero is assigned, the available space in the parent view is divided based on the ratio of each view's `layout_weight` value to the overall `layout_weight` value in the current screen layout and the `layout_weight` value of other views.

For example: Suppose we have a text label and two text editing elements horizontally. The text label has no specified `layout_weight` value, so it takes up the minimum required space. If both text editing elements have a `layout_weight` value of 1, they split the remaining width equally in the parent view layout (because we declared their importance to be equal). If one of the text editing elements has a `layout_weight` value of 1 and the other has a value of 2, one-third of the remaining space is allocated to the first, and two-thirds to the second (because we declared the second to have higher importance).

This layout also demonstrates how to nest multiple layouts within other layouts to achieve more complex and aesthetically pleasing layouts. In this example, a horizontal linear layout is nested within another vertical layout to align the title label and text fields horizontally.

**Step 8:**

Implement the definition of the `onCreate()` method for the `NoteEdit` class:

This method sets the title of our new Activity to "Edit Note" (a string defined in the `strings.xml` file) and sets the note content view through the `note_edit.xml` layout file. We can get handles for the note title and text body views, as well as the confirm button. This way, we can set the title and body of the note through these handles and bind the confirm button to an event responding to the user pressing the button.

Then, we can unpack the parameter values passed in the bundle attached to the Intent and use these values to pre-fill the title and note text body and view, allowing the user to edit them. Next, we can retrieve and store the note row index (`mRowId`) value, thereby knowing which note the user is currently editing.

In the `onCreate()` method, establish the layout:

```java

setContentView(R.layout.note_edit);

```

Find the necessary edit and button components:

These resources can be found through the associated IDs declared in the R class, then converted to the correct view types (two text views are of the `EditText` class, the confirm button is of the `Button` class):

```java

mTitleText = (EditText) findViewById(R.id.title);

mBodyText = (EditText) findViewById(R.id.body);

Button confirmButton = (Button) findViewById(R.id.confirm);

```

Note that `mTitleText` and `mBodyText` are class member variables (you need to declare them in the class before using them here).

Declare a private class member variable `Long mRowId` in the class to store the current row index value `mRowId` (if any).

Continue adding code in the `onCreate()` method to initialize the note title, body, and row index ID variables with the values in the Intent bundle (if present):

```java

mRowId = null;

Bundle extras = getIntent().getExtras();

if (extras != null) {

String title = extras.getString(NotesDbAdapter.KEY_TITLE);

String body = extras.getString(NotesDbAdapter.KEY_BODY);

mRowId = extras.getLong(NotesDbAdapter.KEY_ROWID);

if (title != null) {

mTitleText.setText(title);

}

if (body != null) {

mBodyText.setText(body);

}

}

```

We extract the title and body values set in the bundle when the Intent was triggered.

We also protect against assigning empty strings to the text fields.

Create an `onClickListener()` method for the button:

Listeners may be one of the more confusing aspects of UI implementation, but in this example, the effect we want to achieve is simple: call the `onClick()` method when the user presses the confirm button. Use it to do something and return some values to the caller Intent after the note has been edited. We achieve this purpose through an anonymous inner class, which may seem a bit confusing if you haven't seen it before. However, you need to set aside confusion for now because there will be opportunities to refer back to this section of code and understand how to create a Listener and associate it with a button (Listener is a common term in Java development, especially in UI design and implementation). Below is the code for an empty Listener:

```java

confirmButton.setOnClickListener(new View.OnClickListener() {

public void onClick(View view) {

}

});

```

**Step 9:**

Complete the implementation of the `onClick()` method in our Listener:

When the user presses the confirm button, this interface will run. We hope that this method can obtain the title and text fields of the edited note and put these data into a package as a return value, so that these data can be passed back to the Activity that initially triggered this `NoteEdit` Activity. If the interface responds to an edit rather than a creation operation, we also need to place the row index ID in the package as a return value, so that the `Notepadv2` class can save the edits made to the note.

Create a data package (Bundle) as a return value. Pass the values of the note title and body along with the keywords defined in `Notepadv2` to it:

```java

Bundle bundle = new Bundle();

bundle.putString(NotesDbAdapter.KEY_TITLE, mTitleText.getText().toString());

bundle.putString(NotesDbAdapter.KEY_BODY, mBodyText.getText().toString());

if (mRowId != null) {

bundle.putLong(NotesDbAdapter.KEY_ROWID, mRowId);

}

```

Pass this data package into a new Intent and end the Activity:

```java

Intent mIntent = new Intent();

mIntent.putExtras(bundle);

setResult(RESULT_OK, mIntent);

finish();

```

This new Intent simply acts as a carrier for the data package (including the note title, body, and row index ID that we want to pass).

The `setResult()` method sets the result type code and passes the Intent back to the caller of this Intent. In this example, everything works, so we return the result code `RESULT_OK`.

Calling the `finish()` method signals that the Activity is complete (equivalent to a return call). Any interfaces set in the results are returned together with the execution control information to the result caller.

The complete implementation of the `onCreate()` method (along with some fields supporting the class) is as follows:

```java

private EditText mTitleText;

private EditText mBodyText;

private Long mRowId;

@Override

protected void onCreate(Bundle savedInstanceState) {

super.onCreate(savedInstanceState);

setContentView(R.layout.note_edit);

mTitleText = (EditText) findViewById(R.id.title);

mBodyText = (EditText) findViewById(R.id.body);

Button confirmButton = (Button) findViewById(R.id.confirm);

mRowId = null;

Bundle extras = getIntent().getExtras();

if (extras != null) {

String title = extras.getString(NotesDbAdapter.KEY_TITLE);

String body = extras.getString(NotesDbAdapter.KEY_BODY);

mRowId = extras.getLong(NotesDbAdapter.KEY_ROWID);

if (title != null) {

mTitleText.setText(title);

}

if (body != null) {

mBodyText.setText(body);

}

}

confirmButton.setOnClickListener(new View.OnClickListener() {

public void onClick(View view) {

Bundle bundle = new Bundle();

bundle.putString(NotesDbAdapter.KEY_TITLE, mTitleText.getText().toString());

bundle.putString(NotesDbAdapter.KEY_BODY, mBodyText.getText().toString());

if (mRowId != null) {

bundle.putLong(NotesDbAdapter.KEY_ROWID, mRowId);

}

Intent mIntent = new Intent();

mIntent.putExtras(bundle);

setResult(RESULT_OK, mIntent);

finish();

}

});

}

```

**Step 10:**

Finally, define the new Activity in the manifest file:

Before the new Activity is recognized by the Android operating system, it needs its own Activity entry in the `AndroidManifest.xml` file. This way, the Android operating system knows that the Activity exists in the `AndroidManifest.xml` file and can be called. We can also specify which IntentFilters are implemented by the Activity, but we'll skip that part for now and just let the Android operating system know that the Activity is defined.

The Manifest Editor included in the Eclipse plugin provides a convenient way to edit the `AndroidManifest` file, which we will use. Double-click the `AndroidManifest.xml` file in the folder tree to open it.

Click the Application tab at the bottom of the Manifest Editor.

In the application section, click Add....

If you see a radio button at the top of the dialog box, select the radio button at the top labeled "Create a new element at the application level."

Before clicking "OK," make sure you have selected "(A) Activity" in the dialog box tab.
