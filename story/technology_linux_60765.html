Handling font files in JDK 5.0 on Linux (Setting the default font)

by wszdw142 on 2008-04-07 20:18:55

(Partially referenced other people's documents on the internet)

Apache Batik is a library for processing SVG. I originally intended to use it to create a filtering servlet that would automatically render a PNG image and send it to the client when the browser does not support SVG. I thought this would be a simple task, but... frustratingly, it cost me two days of work. Looking back, the reasons for the errors were different at various stages, but the resulting issues were similar, leading me to subjectively believe both errors had the same cause, which wasted a lot of time debugging. Initially, when drawing images, Chinese characters appeared as "???", obviously due to a Chinese encoding issue. After fixing it, I drew again, and everything seemed normal. However, after running it formally, I discovered that some Chinese text in the graphics was missing, showing as "口口". I assumed it was still an encoding issue, so I delved deep into Batik, checking for hours, but found no improper handling of encoding. Eventually, I traced it all the way to `graphics2d.drawXXX`. With no further options, going deeper would involve native code. So, I retraced my steps, but still couldn't find the problem and almost gave up.

This morning, feeling refreshed, I suddenly wondered if it could be a JDK bug. Thus, I wrote a JSP test page with the following code:

I used the system to list all fonts and drew pictures with each one, checking them one by one. Sure enough, the problem was identified:

1. There was no SimSun font! Sweat... It turns out the Chinese characters that could be drawn were using some unknown font.

2. The situation was the same as with Batik—some Chinese characters couldn't be rendered, appearing as squares.

After searching online, I discovered that JDK 5.0 on Linux differs from previous versions—it doesn't support Chinese fonts by default. You need to manually configure a `fontconfig` file located in `$JAVA_HOME/jre/lib/`. Indeed, there were many `fontconfig.XX.properties` files. Rather than reading too much documentation, I decided to create my own configuration file, copying SimSun.ttc from Windows to the `fonts` directory under `lib`.

Then, I created a simple configuration file:

```properties

# @(#)linux.fontconfig.SuSE.properties 1.2 03/10/17

#

# Copyright 2003 Sun Microsystems, Inc. All rights reserved.

#

# Version

version=1

# Component Font Mappings

serif.plain.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

serif.bold.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

serif.italic.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

serif.bolditalic.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

sansserif.plain.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

sansserif.bold.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

sansserif.italic.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

sansserif.bolditalic.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

monospaced.plain.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

monospaced.bold.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

monospaced.italic.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

monospaced.bolditalic.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

dialog.plain.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

dialog.bold.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

dialog.italic.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

dialog.bolditalic.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

dialoginput.plain.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

dialoginput.bold.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

dialoginput.italic.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

dialoginput.bolditalic.zh-1=Simsun GB-medium-r-normal--0-0-0-0-c-0-gb2312.1988-0

# Search Sequences

sequence.allfonts=zh-1

# Exclusion Ranges

# Font File Names

```

The file was named `fontconfig.RedHat.properties`, where the middle name corresponds to the operating system (e.g., `RedHat` for Red Hat systems, or `XP` for Windows XP).

After restarting Java, the issue was resolved.

**Lessons Learned:**

1. Don't easily use unfamiliar third-party libraries.

2. When encountering problems, don't get stuck on one idea—take breaks when appropriate.

3. If Google isn’t helping, avoid debugging third-party libraries directly.
