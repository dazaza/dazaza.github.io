APK and Android

by wujizhui on 2010-12-19 12:11:18

APK stands for Android Package, which refers to an Android installation package. The APK file format is similar to Symbian's Sis or Sisx formats. By transferring the APK file directly to the Android emulator or an Android phone and executing it, you can install the application.

An APK file is essentially in ZIP format but with its extension changed to .apk. After decompressing using UnZip, you will find Dex files. Dex stands for Dalvik VM Executes, meaning Android Dalvik executable programs. These are not Java ME bytecode but rather Dalvik bytecode.

The structure of an APK file includes:

- META-INF: Commonly seen in JAR files.

- res: A directory storing resource files.

- AndroidManifest.xml: The global configuration file for the program.

- classes.dex: Contains Dalvik bytecode.

- resources.arsc: Compiled binary resource files.

In summary, we observe that when Android runs a program, it first needs to unzip the APK file. This process is somewhat similar to how Symbian operates, differing from the PE (Portable Executable) files used in Windows Mobile. This approach does not offer high levels of confidentiality or reliability for programs, as they can be decompiled using the dexdump command. However, this conforms to development trends; Microsoft’s Windows Gadgets or WPF also adopt a similar architecture.

In the Android platform, the execution files of the Dalvik VM are packaged in APK format. During runtime, the loader unzips the file and accesses the permission-related security settings in the compiled androidmanifest.xml file. Nevertheless, many security restrictions still exist. If you transfer the APK file to the /system/app folder, you may find that execution is unrestricted. Typically, the files we install daily might not reside in this folder, whereas in Android ROMs, system APK files are by default placed in this folder and possess root permissions.
