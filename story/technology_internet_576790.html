A blood case caused by a code

by diglog on 2011-11-22 06:10:25

The Internet resolves multilingual compatibility through various encoded character sets. Due to historical reasons, the vast majority of Chinese websites use either GBK or GB2312 encoding. As web applications become more complex with increasing Ajax data interactions, the rigid GBK encoding no longer meets modern requirements. Therefore, it's clear that UTF-8 remains the best choice.

In the case of IE6/IE7 browsers, when the encoding of an external JavaScript file does not match the page's encoding, it can lead to undefined variables. Sounds shocking? Suppose the page uses GBK encoding and includes the following code: `Foo.init({...});`. Meanwhile, the referenced external JS file is in UTF-8 encoding and contains the following code: `var Foo = {};`. In this situation, IE6/IE7 will immediately crash, kindly informing you that the variable `Foo` is undefined.

All of this stems from a file encoding issue. If the encoding of external resources is not specified, IE6/IE7 defaults to using the page's encoding. The solution is simple: just add the `charset="utf-8"` attribute when referencing the external JS file. For example:

```html

```

Conversely, if the external resource is GBK encoded, you can use `charset="gbk"`.
