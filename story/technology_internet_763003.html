Reasons to Choose or Abandon MySQL

by anonymous on 2013-07-29 12:07:36

MySQL, as an open-source database, has been under the watchful eye of the development community since it was acquired by Oracle. There have been concerns about its future direction. Recently, two developers shared their reasons for choosing and abandoning MySQL, providing valuable insights for those in the database field.

Andy Patrizio outlined five reasons to abandon MySQL:

- MySQL is not as mature as other relational databases. Initially, MySQL wasn't designed as a RDBMS but evolved over time with added functionalities. In many eyes, other established and long-standing relational databases still offer more features than MySQL. If you need a feature-rich RDBMS, PostgreSQL or commercial options like Oracle or Microsoft SQL Server might be better choices. PostgreSQL code contributor Selena Deckelmann stated that Postgres is the right choice for web developers needing a relational database. With support for JSON data structures and PLV8, Postgres could become the default choice for NoSQL.

- Although MySQL is open-source, it only appears so. Technically, MySQL is an open-source database, but practically, developers feel otherwise. Under Oracle's management, MySQL now includes some patent-protected and proprietary modules. Paula Rooney noted that MySQL's activity is superficial; Oracle controls the development process, refuses to release test cases, and tightly manages security patches. Should Oracle donate MySQL to Apache? You have other open-source alternatives like MariaDB, which remains truly open-source. All of MariaDB's code is released under GPL, LGPL, or BSD licenses without any private modules like MySQL 5.5 Enterprise Edition.

- MySQL's performance does not match that of its competitors. The MariaDB blog provides benchmark results comparing MySQL and MariaDB, where the latter performs better. PostgreSQL developers claim Heroku Postgres is more appealing: "It’s the largest hosted environment for Postgres, automatically handling application scaling and supporting plugins..."

- MySQL is controlled by Oracle rather than being community-driven. Since Oracle took over, although there hasn't been a clear change in direction, Oracle's presence makes developers uneasy. Michael “Monty” Widenius, co-founder of MySQL and MariaDB, expressed that the worst part is that the community cannot collaborate with Oracle's developers on MySQL. Moreover, Oracle rejects patches and refuses to disclose roadmaps.

- More people are abandoning MySQL. This year alone, Redhat, Slackware Linux, OpenSUSE, and Wikipedia have all announced plans to switch from MySQL to MariaDB. Asher Feldman, Wikimedia architect, explained that enhancements in MariaDB's optimizer and Percona's XtraDB configuration were key factors behind the decision. "We welcome and support the MariaDB Foundation as a non-profit organization."

Rikki Endsley presented five reasons to choose MySQL:

- MySQL continues to invest and innovate more than ever before. Since the acquisition, Oracle has increased the number of MySQL developers and provided a more mature engineering model, unlike the typical dispersed nature of previous open-source projects. Additionally, MySQL's code has become more modular, meaning short-term investments yield long-term returns. In MySQL 5.6, they broke down the critical LOCK_open in the MySQL server, improving performance more than twofold. Furthermore, MySQL's storage engine is InnoDB, which Oracle acquired in 2005, enabling better integration between the two.

- The MySQL product remains robust. Despite complaints from MariaDB and other open-source products about the lack of test cases and certain enterprise features being proprietary in MySQL 5.5, it continues to perform well. Oracle spent two years releasing DMRs (Development Milestone Releases) to the MySQL community for testing and feedback. MySQL 5.6 supports up to 64 CPU threads compared to 4 previously and offers additional new features.

- MySQL focuses on Web, Cloud, and Big Data. To support cloud services, MySQL backup capabilities have significantly improved, including Global Transaction Identifiers (GTIDs), which allow easy tracking and comparison of progress states between master and slave servers. In April 2013, Oracle released MySQL Applier for Hadoop. Nokia initially used MySQL in big data environments, including centralized Hadoop clusters.

- MySQL Enterprise Edition 5.6 adds high availability features such as backups and an enterprise monitor that continuously monitors your database and provides best practices. It also includes a Query Analyzer for monitoring application performance and rich management tools for server configuration and user management.

- MySQL projects continue to grow. Before the acquisition, MySQL had 400 employees across 25 countries, but communication issues existed. Now, Oracle has organized new teams working together, some focusing on clustering software, others on database algorithms, and others on backup functionality.

Recently, InfoQ discussed system scalability issues, mentioning MySQL in two instances:

- Using a database for queues – While MySQL excels in many areas, it isn’t ideal for handling application queues. Do you have a JOBS table in your database with a status column containing values like "queued," "working," and "completed"? If so, you're likely using your database to manage application queues, which isn't optimal due to locking issues and difficulties in searching and scanning for the next task. Consider RabbitMQ or Amazon SQS as these external services scale more easily.

- Using a database for full-text search – Although Oracle provides full-text search support, why can't MySQL do the same? MySQL does have this feature, but in many versions, it only works with the older MyISAM storage engine. It's recommended to use proven search solutions like Apache Solr, specifically designed for search tasks with excellent libraries, allowing developers to use various modern web languages and scale easily. For those interested in cutting-edge technology, MySQL 5.6 offers crash-safe and transactional InnoDB storage engines. However, external solutions like Solr or Sphinx and the MySQL Sphinx SE plugin are still advised.

Despite these points, Google continues to favor MySQL, currently transforming it into a fully managed service within the cloud. This includes a set of JSON APIs for programmatically managing MySQL. Google also offers Cloud SQL to enhance its cloud services. This "fully managed" web service provides a scalable MySQL database running on Google's infrastructure. Users can connect to MySQL instances in the cloud via several methods: command prompt, API console, management and reporting tools, external applications, Java and Python applications on GAE, and GAE scripts. Users can also import and export instances using mysqldump. Google provides an experimental JSON API, a RESTful interface for programmatically managing MySQL: creating, deleting, restarting, restoring from backups, importing/exporting, and retrieving various instance information. Although both command-line and API management options are available, MySQL instances are still automatically managed, including synchronous or asynchronous replication across different geographical locations, on-demand scaling, automatic backups, and patching when needed.
