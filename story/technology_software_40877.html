Network security configuration in CMD

by xloong on 2007-12-29 21:43:15

The built-in CMD command-line tools related to networking are numerous, such as the familiar `ping`, `tracert`, `ipconfig`, `telnet`, `ftp`, `tftp`, `netstat`, and less commonly known ones like `nbtstat`, `pathping`, `nslookup`, `finger`, `route`, `netsh`...

These CMD commands can be categorized into three types: network diagnostics (e.g., `ping`), network connection (e.g., `telnet`), and network configuration (e.g., `netsh`). The first two categories are relatively straightforward, so this article will only cover two network configuration tools.

### **Netsh**

When using `netsh` in a remote shell, one must first address an interaction mode issue. As mentioned earlier, many shells cannot redirect output again, so interactive command-line tools like `ftp` cannot be used interactively in such environments. The solution is that most interactive tools allow the use of scripts (or response files). For example, `ftp -s:filename`. Similarly, `netsh` also supports this: `netsh -f filename`.

The functionality of `netsh` commands is extensive, allowing configuration of IAS, DHCP, RAS, WINS, NAT servers, TCP/IP protocols, IPX protocols, routing, etc. Since we're not administrators, it's usually unnecessary to know all these details. We only need to use `netsh` to understand the network configuration information of the target host.

1. **TCP/IP Configuration**

```cmd

echo interface ip > s

echo show config >> s

netsh -f s

del s

```

This allows you to understand whether the host has multiple network cards and IPs, if the IP is dynamically assigned (DHCP), and what the internal IP is (if applicable). This command is similar to `ipconfig /all`.

Note: The following commands require the `remoteaccess` service to be running on the target host. If it is disabled, you must first enable it by importing a registry file, then:

```cmd

net start remoteaccess

```

2. **ARP Table**

```cmd

echo interface ip > s

echo show ipnet >> s

netsh -f s

del s

```

This provides more information than the `arp -a` command.

3. **TCP/UDP Connections**

```cmd

echo interface ip > s

echo show tcpconn >> s

echo show udpconn >> s

netsh -f s

del s

```

These commands are equivalent to `netstat -an`.

4. **Network Interface Information**

If all `netsh` commands could be replaced by other commands, why would it still be necessary? Below is something that cannot be easily replaced:

```cmd

echo interface ip > s

echo show interface >> s

netsh -f s

del s

```

Functions like modifying IP addresses with `netsh` are generally unnecessary (in case changing the IP makes the system unreachable, resulting in "no help from heaven or earth"). Therefore, they are omitted.

### **IPSec**

It’s important to note that IPSec and TCP/IP filtering are different things; do not confuse them. TCP/IP filtering has very limited functionality compared to the flexibility and power of IPSec. Below, we discuss how to control IPSec via the command line.

In XP, use `ipseccmd`; in Windows 2000, use `ipsecpol`. Unfortunately, neither is included by default. `ipseccmd` is located in the `SUPPORT\TOOLS\SUPPORT.CAB` folder on the XP installation disc, while `ipsecpol` is found in the Windows 2000 Resource Kit. Additionally, to use `ipsecpol`, you need two additional files: `ipsecutil.dll` and `text2pol.dll`. Together, these three files total 119KB.

IPSec can be controlled via Group Policy, but after searching MSDN, I couldn't find the syntax for the corresponding security templates. Moreover, pre-configured IPSec policies cannot be exported as templates. Thus, configuring IPSec via Group Policy isn't feasible. IPSec settings are stored in the registry (`HKEY_LOCAL_MACHINE\SOFTWARE\Policies\Microsoft\Windows\IPSec\Policy\Local`), and theoretically, IPSec could be configured by modifying the registry. However, much of the information is stored in binary form, making reading and modification difficult. Uploading command-line tools is therefore more convenient.

There is plenty of documentation available online about `ipsecpol` and `ipseccmd`, so this article won’t go into detail but will provide some practical examples. The syntax of `ipseccmd` for setting IPSec policies is almost identical to `ipsecpol`, so only `ipsecpol` will be demonstrated:

1. **Defend Against RPC-DCOM Attacks**

```cmd

ipsecpol -p myfirewall -r rpc-dcom -f *+0:135:tcp *+0:135:udp *+0:137:udp *+0:138:udp *+0:139:tcp *+0:445:tcp *+0:445:udp -n BLOCK -w reg -x

```

This command blocks local host ports TCP 135, 139, 445, and UDP 135, 137, 138, 445.

Explanation:

- `-p myfirewall`: Specifies the policy name as "myfirewall".

- `-r rpc-dcom`: Specifies the rule name as "rpc-dcom".

- `-f ......`: Creates seven filters. `*` represents any address (source); `0` represents the local machine address (destination); `+` indicates mirror (bidirectional) filtering. Detailed syntax can be seen with `ipsecpol -?`.

- `-n BLOCK`: Specifies the filter action as "block". Note that BLOCK must be uppercase.

- `-w reg`: Writes the configuration to the registry, making it persistent after reboot.

- `-x`: Activates the policy immediately.

2. **Prevent Being Pinged**

```cmd

ipsecpol -p myfirewall -r antiping -f *+0::icmp -n BLOCK -w reg -x

```

If the policy named "myfirewall" already exists, the "antiping" rule will be added to it.

Note: This rule also prevents the host from pinging others.

3. **Restrict Access to Backdoors**

Suppose you have installed DameWare Mini Remote Control on a host. To protect it from brute-force password attacks or buffer overflows, restrict access to its service port (6129).

```cmd

ipsecpol -p myfw -r dwmrc_block_all -f *+0:6129:tcp -n BLOCK -w reg

ipsecpol -p myfw -r dwmrc_pass_me -f 123.45.67.89+0:6129:tcp -n PASS -w reg -x

```

This ensures only 123.45.67.89 can access port 6129 on the host.

If you have a dynamic IP, set rules based on the range of IP addresses allocated to you. For example:

```cmd

ipsecpol -p myfw -r dwmrc_block_all -f *+0:6129:tcp -n BLOCK -w reg

ipsecpol -p myfw -r dwmrc_pass_me -f 123.45.67.*+0:6129:tcp -n PASS -w reg -x

```

This allows IP addresses from 123.45.67.1 to 123.45.67.254 to access port 6129.

Be extremely careful when writing rules to avoid blocking yourself. If you're unsure about the effect of a rule, you can use Task Scheduler to "leave a backdoor." For example:

```cmd

c:\> net start schedule

Task Scheduler service is starting ..

Task Scheduler service started successfully.

c:\> time /t

12:34

c:\> at 12:39 ipsecpol -p myfw -y -w reg

A new job has been added, Job ID = 1

```

You now have five minutes to set up and test the "myfw" policy. After five minutes, the scheduled task will stop the policy.

If the test results are unsatisfactory, delete the policy:

```cmd

c:\> ipsecpol -p myfw -o -w reg

```

Make sure the policy is stopped before deleting it; otherwise, it may remain active for a period of time depending on the policy refresh interval (default is 180 minutes).

If the test passes, enable the policy:

```cmd

c:\> ipsecpol -p myfw -x -w reg

```

Finally, here's how to check the IPSec policy:

- For XP, it's simple: `ipseccmd show filters`

- `ipsecpol` does not have query functionality. Use another command-line tool, `netdiag`, which is located in `SUPPORT\TOOLS\SUPPORT.CAB` on the Windows 2000 installation disc.

`netdiag` requires the `RemoteRegistry` service to be running. Start it:

```cmd

net start remoteregistry

```

If `RemoteRegistry` is not started, you'll get an error:

```

[FATAL] Failed to get system information of this machine.

```

The command to view the IPSec policy is:

```cmd

netdiag /debug /test:ipsec

```

This produces a long string of output. The IPSec policy is at the end.

### **Software Installation**

The installation process of a software/tool generally involves two things: copying files to specific directories and modifying the registry. Once you understand the specifics, you can implement the installation via the command line (excluding cases where activation or registration is required after installation).

WinPcap is a commonly used tool, but it must be installed in a graphical user interface. Non-GUI versions can be found online (though they still have copyright pages), but we can create our own version.

For example, with WinPcap 3.0a, comparing the file system and registry snapshots before and after installation reveals the entire installation process. Excluding uninstallation parts, the key files are three: `wpcap.dll`, `packet.dll`, and `npf.sys`. The first two are located in the `system32` directory, and the third is in `system32\drivers`. Registry changes involve adding a system service called `NPF`. Note that this is a system service (i.e., driver) and not a Win32 service.

As a system service, keys must be added under both `HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services` and `HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Enum\Root`. By default, only the `SYSTEM` account can modify the latter, but fortunately, manual addition isn't necessary—`winpcap` handles it automatically when invoked. Even the registry doesn't need to be manually modified; `winpcap` takes care of everything. All you need to do is copy the three files to their appropriate locations.

As an example, here's how to modify the registry using an INF file:

```ini

[Version]

Signature="$WINDOWS NT$"

[DefaultInstall.Services]

AddService=NPF,,winpcap_svr

[winpcap_svr]

DisplayName=Netgroup Packet Filter

ServiceType=0x1

StartType=3

ErrorControl=1

ServiceBinary=%12%\npf.sys

```

Save the above content as `_wpcap_.inf`. Then write a batch file `_wpcap_.bat`:

```cmd

rundll32.exe setupapi,InstallHinfSection DefaultInstall 128 %CD%\_wpcap_.inf

del _wpcap_.inf

if /i %CD%==%SYSTEMROOT%\system32 goto COPYDRV

copy packet.dll %SYSTEMROOT%\system32\

copy wpcap.dll %SYSTEMROOT%\system32\

del packet.dll

del wpcap.dll

:COPYDRV

if /i %CD%==%SYSTEMROOT%\system32\drivers goto END

copy npf.sys %SYSTEMROOT%\system32\drivers\

del npf.sys

:END

del %0

```

Package all files (five in total) into a self-extracting EXE with WinRAR, and set the "Run after extraction" option to `_wpcap_.bat`. The command-line WinPcap installer is now ready.

Note: The last line of the batch file should not have a carriage return. Otherwise, the file will be in use and cannot be deleted.

This approach can be applied to most software installations. However, system patches are exceptions because they might replace files currently being executed or accessed, so the `copy` command won't work. Fortunately, Windows patch packages support command-line installation. For example:

```cmd

KB824146.exe -n -z -q

```

- `-n`: Do not keep backups.

- `-z`: Do not restart.

- `-q`: Quiet mode.

If you have multiple patches to install, package them into a self-extracting RAR file with a batch script:

```cmd

for %%f in (KB??????.exe) do %%f -n -z -q

for %%f in (KB??????.exe) do del %%f

del %0

```

### **Windows Scripts**

Many tasks can be accomplished concisely with scripts. Below are some common scripts in echo format.

1. **Display System Version**

```cmd

@echo for each ps in getobject _ >ps.vbs

@echo ("winmgmts:\\.\root\cimv2:win32_operatingsystem").instances_ >>ps.vbs

@echo wscript.echo ps.caption^&" "^&ps.version:next >>ps.vbs

cscript //nologo ps.vbs & del ps.vbs

```

2. **List Processes**

```cmd

@echo for each ps in getobject _ >ps.vbs

@echo ("winmgmts:\\.\root\cimv2:win32_process").instances_ >>ps.vbs

@echo wscript.echo ps.handle^&vbtab^&ps.name^&vbtab^&ps.executablepath:next >>ps.vbs

cscript //nologo ps.vbs & del ps.vbs

```

3. **Terminate a Process**

```cmd

@echo for each ps in getobject _ >pk.vbs

@echo ("winmgmts:\\.\root\cimv2:win32_process").instances_ >>pk.vbs

@echo if ps.handle=wscript.arguments(0) then wscript.echo ps.terminate:end if:next >>pk.vbs

```

To terminate a process with PID 123, use the following syntax:

```cmd

cscript pk.vbs 123

```

If it displays `0`, termination was successful. Then:

```cmd

del pk.vbs

```

4. **Restart System**

```cmd

@echo for each os in getobject _ >rb.vbs

@echo ("winmgmts:{(shutdown)}!\\.\root\cimv2:win32_operatingsystem").instances_ >>rb.vbs

@echo os.win32shutdown(2):next >>rb.vbs & cscript //nologo rb.vbs & del rb.vbs

```

5. **List Auto-Start Services**

```cmd

@echo for each sc in getobject("winmgmts:\\.\root\cimv2:win32_service").instances_ >sc.vbs

@echo if sc.startmode="Auto" then wscript.echo sc.name^&" - "^&sc.pathname >>sc.vbs

@echo next >>sc.vbs & cscript //nologo sc.vbs & del sc.vbs

```

6. **List Running Services**

```cmd

@echo for each sc in getobject("winmgmts:\\.\root\cimv2:win32_service").instances_ >sc.vbs

@echo if sc.state="Running" then wscript.echo sc.name^&" - "^&sc.pathname >>sc.vbs

@echo next >>sc.vbs & cscript //nologo sc.vbs & del sc.vbs

```

7. **Show Last Boot Time**

```cmd

@echo for each os in getobject _ >bt.vbs

@echo ("winmgmts:\\.\root\cimv2:win32_operatingsystem").instances_ >>bt.vbs

@echo wscript.echo os.lastbootuptime:next >>bt.vbs & cscript //nologo bt.vbs & del bt.vbs

```
