Excel-like Report Designer

After summarizing the pros and cons of traditional strip reports (word-type editors) and grid controls, Java reporting software FineReport innovatively proposed using an Excel-style interface to edit reports, completely solving the efficiency problem of editing complex Chinese reports while reducing the learning cost for domestic users who use Excel for reports, pioneering a new generation of report editing modes.

An Excel-like report design interface

The report editing area of the Java reporting software FineReport supports unlimited rows and columns. Users will not need to manually add rows or columns due to insufficient row or column numbers when editing reports.

Reports support multi-sheet design. The sheet editing interface, adding and deleting sheets methods are similar to Excel. This ensures that multi-sheet Excel files are fully imported into one report file. At the same time, content-related files within a single sheet are completely exported as corresponding sheet Excel files.

Cell naming is entirely similar to Excel. Columns are represented by A, B, C……… Rows are represented by 1,2,3,4,5,6,7….. Cells are identified by A1, B1, A2 one by one.

The font, size, color, border, foreground color, background color, etc., of data in cells, as well as the whole row, whole column, whole table modification editing style and format brush properties correspond completely with Excel.

Formula adjustments are similar to Excel; if the content in cell A1 is =SUM(A2, A3), inserting a row before the first row adjusts the formula content to =SUM(A3,A4).

Expressions fully include Excel formula writing methods, supporting SUM(A1,A2), SUM(A1+A2), and SUM(A1:A2), etc.

Chart elements in reports support floating, achieving arbitrary relative positioning with tables, solving user challenges brought by random reports to the greatest extent.

Multi-source Report Fragmentation

Chinese characteristic reports often need to display data existing in several database tables in one report. Traditional Western Java reporting software uses unified views creation, stored procedures writing, and program preparation of data or sub-reports to solve the problem.

As the first type of solution, creating a unified view, writing stored procedures, and preparing data with programs mainly handle horizontal direction report fragmentation. However, this solution has overly complex SQL writing, multiple joins and unions causing low processing efficiency.

The second type of solution—sub-reports, mainly addresses vertical direction fragmentation problems. However, sub-reports have the following issues: 1) Sub-reports operate independently, and every adjustment to the main table and sub-table formats requires adjusting all related reports, making alignment between reports very difficult to solve. 2) In terms of data, there is no communication between the main and sub-tables, nor between sub-tables. Only the data from the main table can be passed as parameters to sub-tables, and sub-tables cannot return information or provide information to other sub-tables. At this point, mixed calculations across the entire table become very difficult, requiring separate calculations, unable to adopt intermediate calculation results from some sub-tables, not only complicating writing but also lowering computational efficiency. As shown in the figure below: in traditional Java reporting software, total amounts are hard to statistically derive from sub-reports.

FineReport's multi-source fragmentation concept thoroughly resolves these issues. One report can extract data from multiple databases or from multiple database tables within one database. Report-making efficiency increases by more than five times. Simultaneously, it can achieve complex fragmented reports that traditional Java reporting software basically cannot accomplish.

Symmetry of Rows and Columns in Reports

Traditional Java reporting software is stronger in handling vertical directions than horizontal ones. There are two serious defects: 1) Data can only expand from top to bottom, making it difficult to handle non-standard cross-tab reports that combine dynamic and static columns. 2) Header repetition can only repeat the top header, not the side headers.

Java reporting software FineReport adopts the principle of symmetry of rows and columns, compensating for the shortcomings of traditional Java reporting software, making horizontal direction processing as powerful as vertical direction processing. For non-standard cross-tab reports that combine dynamic and static columns, FineReport can handle them just like vertical direction processing, allowing these complex cross-tab reports that require programming in traditional Java reporting software to be easily achieved using the designer.

Hierarchical Coordinates in Reports

Traditional Java reporting software generally only provides inter-cell operations within the same row and aggregate operations for a certain group (or all). For the following situations, there are deficiencies:

Cross-row operations, such as referencing data from the previous row, typical applications include cumulative ratio and year-on-year ratio.

Cross-group operations, if year-on-year ratios span groups, they are powerless.

For aggregate operations, only a few fixed functions are provided, such as taking the first place and calculating cumulative values, unable to form general aggregate operations, such as taking the second place and calculating the product of cumulative values. Certain conditional operations are also unmanageable, such as counting how many data points in the same group are larger than the current number, which is data ranking. Java reporting software FineReport adopts hierarchical coordinates, allowing precise positioning of each extended cell. After extension, the positioning rules are divided into two types:

Absolute hierarchical coordinates. The naming rule for extended report cells is as follows:

Lk is the left parent cell of a certain cell Cellx, lk is the sequence number of the extended left parent cell, i.e., the ordinal number of the extended cell, if lk is not specified or is 0, it indicates the current left parent Lk of the cell where the current expression resides; similarly, the principle for the upper parent cell is the same;

If there is no left parent cell, only the upper parent cell exists, the semicolon cannot be omitted, i.e., it is written as Cellx [; Tk : tk, Tk-1 : tk-1,...T1 : t1];

If there is no upper parent cell, only the left parent cell exists, the semicolon can be omitted, i.e., it is written as Cellx [Lk : lk, Lk-1 : lk-1,...L1 : l1];

The order of Lk is from far to near, starting from the furthest parent cell from the current cell, or understood as starting from the highest level grouping cell.

Relative hierarchical coordinates. The naming rule for extended cells is as follows:

Lk is the left parent cell of Cellx, lk is the offset of the cell, i.e., the lkth cell above or below the left parent cell Lk where the current expression resides, if lk is not specified, it indicates the current left parent cell Lk, Cellx is the target cell, usually an affiliated cell of Lk, Lk-1,…,L1, i.e., a child cell;

If there is no left parent cell, only the upper parent cell exists, the semicolon cannot be omitted, i.e., it is written as Cellx [; Tk : ±tk, Tk-1 : ±tk-1,...T1 : ±t1];

If there is no upper parent cell, only the left parent cell exists, the semicolon can be omitted, i.e., it is written as Cellx [Lk : ±lk, Lk-1 : ±lk-1,...L1 : ±l1];

The order of Lk is from far to near, starting from the furthest parent cell from the current cell, or understood as starting from the highest level grouping cell.

Report Filling

Traditional Java reporting software can only read database data to the front end, solving the data presentation problem, summarized as data reading. FineReport not only achieves data reading but also implements data writing, realizing bidirectional data flow.

Key aspects of report filling include:

Matching data tables that reflect business rules with report cells. No need to consider database structure when designing report templates; no need to consider future filling table design when designing database structure.

Rich front-end display styles for reports. Implement complete cell data validation to ensure data validity; enable automatic calculations upon modifying a cell.

Filling and presentation of data are independent; the data flow paths for filling and presentation tables are completely separated. Data from database A can be displayed on the front end, and then filled back into database B.

Through the above technology, Java reporting software FineReport can easily implement three typical filling tables:

Detail filling table: Achieves CRUD (Create, Read, Update, Delete) operations on data.

Employee Information Table for Java reporting software detail filling: All data on the report corresponds to a single record in the database, but drawing the table is complicated. Additionally, it can test the support of various form vendors for editors.

Multi-source filling table for Java reporting software: Data entered in a single filling table is simultaneously saved into multiple physical tables, maintaining database consistency among the tables.

The most common application of filling tables is combining with workflows. FineReport provides numerous interactive interfaces and has successful cases with collaborative workflows like Xi'an and JBPM.
