Mobile Application Design on Android Platform

Android Design Philosophy

Even though there are significant differences between platforms, the process of learning how to use APIs to create applications is largely similar. Generally, there are two steps: first, you should know how to use the API to implement your functionality. Second, you need to understand the subtle differences between platforms. In other words, first, you should learn how to create an application (understand the basic structure of the application, etc.), and then you need to learn how to implement this application based on specific situations.

In comparison, the second phase (learning to implement the application correctly) usually takes a long time, during which you will constantly write code, make mistakes, and learn from those mistakes. Clearly, this is not an effective learning method. This section and some of the links below aim to help you by teaching you how to learn to create your application.

Before we proceed, here's a key point: successful applications typically provide an outstanding user experience. When the Android team built a system with a robust core, most of the user experience will come from the interaction between users and applications. Therefore, we encourage you to spend time building excellent user experiences for your applications.

Significant user experiences are characterized by three core features: fast, responsive, and seamless. Of course, since the advent of computers, every platform has had these three characteristics in one form or another. However, each platform implements these features differently; the following will briefly introduce how your application can achieve these requirements on the Android platform.

1. Fast

Android applications should execute quickly. More accurately, their programs should run efficiently (efficiency leads to speed). In today's computer world, there is a tendency to assume that Moore's Law will eventually solve all our problems. When this tendency encounters embedded applications, Moore's Law becomes more complex.

Unlike desktop and server applications, Moore's Law is not truly applicable to mobile device applications. Moore's Law is actually about the integration density of electronic transistors, and its original meaning is: over time, more circuits can be integrated into a given circuit size. For desktop and server applications, it means: you can pack more "speed" into a chip of roughly the same size, and as speed increases, other related performance metrics also improve significantly. For embedded applications like smartphones, however, Moore's Law is used to make chips smaller. Thus, as chip density increases, chips with the same functionality become smaller and consume less power, allowing phones to get smaller and batteries to last longer. This results in handheld embedded devices growing at a slower rate compared to desktop systems. Therefore, for embedded devices, Moore's Law means more functions and less power consumption, with speed being secondary.

This is why we need to write efficient code: you cannot naively assume that the speed increase in telephones is the same as in desktops or server applications. Generally, efficient code means minimal memory usage, compact style, and avoidance of performance impacts caused by certain languages and coding habits. We use the concept of object-oriented programming to understand that most of this work is done at the method level, similar to actual code, loops, etc.

2. Responsive

We may be able to write code that wins all the performance tests in the world, but users may find it very frustrating because the application lacks sufficient responsiveness, feels sluggish, freezes at critical moments, or processes input too slowly. On the Android platform, unresponsive applications often trigger the fatal message "Application Not Responding" (ANR).

Typically, this happens when the application fails to respond to user input. For example, if your application blocks on some I/O operations (such as network interface calls), the main thread will not process user input events, and after some time, the system will consider your program hung and offer an option to the user asking whether they want to terminate it. Similarly, if your application spends a lot of time constructing a structure in memory or calculating the next move in a game, the system may also consider the program hung. When encountering such situations, ensure computational efficiency, but even the most efficient code still requires time.

In both examples above, the solution is to create a sub-thread to handle most of the work, ensuring that your main thread (which handles user interface events) continues running, thus preventing the system from thinking your program has frozen. Since the implementation of such threads generally occurs at the "class" level, you can think of responsiveness as a class-level issue (this can be compared to the basic performance described at the method level).

3. Seamless

Even if your application runs quickly and has high responsiveness, it can still frustrate users. A common example is when background processes (like Android's Service and BroadcastReceiver) suddenly pop up a UI response for certain events. This might seem trivial, and developers often consider it normal because they spend a lot of time testing and using their own applications. However, the Android application model is designed to allow users to switch smoothly between different applications. This means that when your background process pops up that UI, the user may be doing something else elsewhere in the system, such as taking a call. Imagine if the SMS service popped up a dialog box every time a text message arrived; this would quickly frustrate users. This is why Android standards use the notification (Notifications) mechanism for such events; it gives users control.

This is just one example, and there are countless similar cases. For instance, if Activities do not properly implement the onPause() method and other lifecycle methods, it could lead to data loss. Or, if your application intentionally exposes data for use by other applications, you should use a ContentProvider rather than an unprotected raw file or database visible to everyone.

These examples share a common characteristic: they all involve interactions between programs or between programs and the system. The system is designed to treat multiple applications as a loosely coupled set of components rather than large black-box codes. This allows you, as a developer, to view the entire system as a large union of these components. This allows you to cleanly encapsulate and seamlessly integrate with other applications, enabling you to design the applications you like.
