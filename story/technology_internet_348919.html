It is well known that while the running speed of Java applications is not slow, it is also not fast. With the latest performance of JRE 1.6, at most, it just outperforms some purely interpreted languages, still lagging behind the execution efficiency of compiled languages like C/C++.

If we use Java to create some simple desktop applications, the current drawing speed of Java components can barely be acceptable; however, if Java is used for game development, especially for creating dynamic effects requiring high FPS, technical optimization must be applied to solve its drawing efficiency problem before discussing other aspects.

What is FPS?

The FPS mentioned here does not refer to shooting games but rather the commonly known "frame rate" (Frames per Second, abbreviation: FPS) or "Hertz" (Hz). The author has provided a specific explanation and Java implementation example in previous blog posts, which will not be repeated here.

Due to the physiological structure of the human eye, as long as the frame rate we see exceeds 16 frames per second, we perceive the image as continuous. This phenomenon is called "persistence of vision" in biology, explaining why movie films are shot frame by frame yet appear continuous to us. Of course, the so-called 16 frames is just an average value and may vary depending on the specific visual object. In the worst case, animations should have at least 12 frames per second, modern movies at least 24 frames per second, and action games at least 30 frames per second to be perceived as continuous and uninterrupted by the human eye.

In summary, the higher the FPS value, the smoother the picture; the lower the FPS value, the more stagnant the picture appears. To improve the running efficiency of Java games, the focus must be on how to increase the program's FPS.

Where should we start to improve FPS?

Java ultimately draws through native code, a fact known to everyone. This means unless we learn SWT and rewrite the local graphic interface, we cannot interfere with the display part; thus, optimizing FPS must occur before the local graphics system calls, not after. Therefore, improving the FPS of Java applications involves repeatedly addressing Image and Graphics and their related subclasses.

So, what parts of these well-known Java drawing components can we improve?

In a recent blog post titled "Ten Basic Principles That Should Always Be Followed in Java Game Development," I shared some insights on building efficient Java games. The seventh point, "Always double-buffer game images," is not only key to solving Java image flickering problems but also one of the winning strategies for increasing Java game frame rates. Yes, the optimization and improvement of Java drawing functions hinge on how to optimize the handling of buffer areas.

Below, I will demonstrate three different Java buffering drawing methods.

1. Buffering using BufferedImage objects

This method is the simplest and most common way to build double buffering. A BufferedImage buffers the current drawing, and all Graphics operations are performed on it. Only when necessary is the texture painted onto the form. It is straightforward to use, but how efficient is it? This remains unknown at this point in the article.

2. Building a buffer using BufferStrategy

Using BufferStrategy for buffering allows access to hardware acceleration provided by the system. The Java system selects the most suitable BufferStrategy based on the local environment. To create a BufferStrategy, you need to inform the system of the desired number of buffers (usually double buffering, i.e., entering "2") using the createBufferStrategy() method and switch between buffers using the getDrawGraphics() method, which returns the next buffer to be used. The biggest disadvantage and advantage of BufferStrategy lie in its dependence on the local graphic environment: neither very fast graphic environments producing slow FPS nor very slow graphic environments producing fast FPS are expected.

3. Completely processing images within the DataBuffer of BufferedImage

Each BufferedImage has a corresponding WritableRaster object (obtained via the getRaster method), from which we obtain the specified BufferedImage's DataBuffer (via the getDataBuffer method). Similar to method 1, we construct a BufferedImage to buffer all current drawings, performing all operations on it and painting the texture onto the form only when necessary. However, since DataBuffer can be converted into arrays representing BufferedImage pixel points such as int[], byte[], or short[], we no longer use the Graphics object provided by Java but directly manipulate pixels for all drawing implementations. But, would array operations be faster?

Now, we will construct three examples for each method, trying to build them with similar processing flows, respectively measuring the specific efficiencies of the three methods.

In actuality, in Java online games like Pirate Age and Shadow World, NetBeans, and various other not-so-slow Java applications, there exists a large amount of image rendering using DataBuffer; however, in terms of usage, only the JGnet engine used in Shadow closely resembles the common pairing mode of Image and Graphics. To provide convenience for those unwilling to decompile, here I briefly analyze part of JGnet (the part in homepage.jar).

From the code, all UI in JGnet inherits from GUIWidget (GUIWidget inherits from Container), and all JGnet-generated GUI components should also inherit from it. JGnet provides a GUIRoot as the bottom panel, to which all derived GUIWidget components should be attached. Additionally, the engine provides AbstractGameClient, ClientContext interfaces, and related implementations to schedule relevant components. The carrier of JGnet's GUIRoot can be either an Applet or a Frame.

Most resources in JGnet are packaged in zip files, though single images can also be read. JGnet resource loading depends on its provided ImageFactory class. Operations of ImageFactory can be divided into the following two steps:

Step one is addArchive, during which the specified zip resources are loaded. Thereafter, the read image resources are transformed into ShortBufferImage and cached in the HashMap provided by the daff.utill package according to the corresponding path in the zip file.

Step two is getImage, which retrieves the ShortBufferImage object for the specified path. This step prioritizes the cache. If retrieval fails, it attempts to call the loadImage method again for loading and caches it in the HashMap object provided by the daff package. The loadImage method determines the type of resource to load based on the suffix. Files with jpg, gif, or png suffixes invoke the AwtImageLoader class provided by the daff package to convert them into ShortBufferImage, while files without these suffixes are directly loaded in ShortBufferImage format.

JGnet mainly uses its internally provided ShortBufferImage and ShortBufferGraphics and related derived classes for image drawing. ShortBufferImage is an abstract class without a parent class, and all graphical resources in JGnet will be represented as ShortBufferImage. Drawing ShortBufferImage requires the specialized canvas ShortBufferGraphics provided by the daff.gui package. Essentially, ShortBufferGraphics is a pixel renderer for ShortBufferImage, internally preserving the DataBuffer of a blank BufferedImage in short array form, with all drawing based on it.

Like most online games, JGnet extensively uses custom graphic formats (suffix .img), whose interpretation and conversion are completed through the CompressedImage class. Since custom formats typically declare element relationships in the image more succinctly, JGnet processes its custom format resources faster than ordinary image resources.

JGnet adopts either MsJvmGraphics or SunJvmGraphics (both inheriting from ShortBufferGraphics) depending on the environment, allowing us to preliminarily determine that it is compatible with Microsoft JVM, meaning it can run on JRE 1.1 or higher.

Through simple comparison, we can discover that Graphics drawing, whether directly using BufferedImage double buffering or utilizing BufferStrategy for hardware acceleration, cannot match the ShortBufferGraphics that directly performs pixel drawing using DataBuffer. BufferStrategy even resulted in a shocking 4 frames when drawing a thousand people, and changing BufferedImage to VolatileImage had little effect in the test. There might be slight improvements on microcomputers with higher configurations, but user machine types are not under our control.

Since JGnet directly obtains the short[] form of the image DataBuffer and uses it for image drawing, ShortBufferImage and ShortBufferGraphics can be considered a set of innate Java graphic buffer objects. In terms of efficiency, using ShortBufferGraphics for drawing yields a significantly higher FPS than using Graphics2D for drawing, determined by the inherent efficiency advantage of array operations. However, the disadvantage lies in that ShortBufferImage and ShortBufferGraphics do not directly inherit from Image and Graphics, and their functions cannot fully correspond. Some commonly used methods still need to be implemented, and many commonly used methods require the ShortBufferGraphics user to resolve themselves.

In terms of efficiency alone, using DataBuffer for buffered drawing is clearly superior to simply using BufferStrategy or obtaining Graphics from BufferedImage for drawing.

Overall, if we pursue the highest FPS, using BufferedImage to generate DataBuffer for pixel processing is the most efficient. Mixing it with method two can further improve efficiency, but unfortunately, there is currently a lack of relevant open-source components, making available resources somewhat insufficient. We can consider functional mixing, where the main part uses DataBuffer rendering and unimplemented parts are supplemented by Graphics, but this will definitely affect operational efficiency.
