Introduction to Android Application Modules

by dream2012 on 2012-03-07 13:02:57

In most operating systems, there is a standalone executable file (such as .exe in Windows) that can generate processes, allowing users to interact with the application via interface icons. However, this is not fixed in Android, and it is crucial to understand how these scattered parts are combined together. Due to Android's flexible nature, understanding some fundamental techniques when implementing different parts of an application is important:

- **Android Package (.apk)**: This contains the code and resources for an application. It is a file that users can download and install on their devices.

- **Task**: Typically, what users perceive as an "application" that can be launched. There is usually an icon on the desktop to start the task. A higher-level application can bring your task to the foreground, placing it above other applications.

- **Process**: This is the core process where the code runs. All the code within an .apk typically runs in one process, with one process corresponding to one .apk; however, the `` tag can be used to change where the code runs—either for the entire .apk or for individual components like activities, receivers, services, or providers.

### Key Points about Tasks:

When users see an "application," it is actually a task. If you create an .apk with just some activities, at least one will be the top-level entry point (via `intent-filter` and category `android.intent.category.LAUNCHER`). This creates a single task, and any activity started will be part of this task.

If an activity is started without the `Intent.FLAG_ACTIVITY_NEW_TASK` flag, it will run in the same task (unless the activity is started in a special mode, which we will discuss later). If you use the `FLAG_ACTIVITY_NEW_TASK` flag and the task has already been started, the task will be brought to the foreground instead of reloading. The `FLAG_ACTIVITY_NEW_TASK` must be used carefully because it appears to the user as if a new application is being started. If this is not your intention, you won't create a new task. Additionally, if the user needs to return to their original place from the desktop and open a new task with the same intent, you should use the new task flag. Otherwise, if the user presses the HOME key instead of the BACK key after launching your task, your task and its activities will be placed behind the desktop program, making it impossible to switch back.

### Task Affinities:

Sometimes Android needs to know which task an activity belongs to, even if that task hasn't been started yet. This is done through task affinity, which provides a unique static name for one or more activities that may run in a task. By default, the task affinity name for an activity is the name of the .apk package that implements the activity. This provides a general feature where all activities in an .apk appear as part of a single application to the user.

When starting a new activity without the `Intent.FLAG_ACTIVITY_NEW_TASK` flag, task affinity has no effect on the newly started activity—it will always run in the task it was started from. However, if the `NEW_TASK` flag is used, affinity checks if an existing task has the same affinity. If so, that task is brought to the foreground, and the new activity is started at the top of the task stack.

You can set your own task affinity for all activities in your .apk by using the application tag in your manifest file, or you can set it individually for each activity. Here are some examples of how to use them:

- If your .apk contains multiple top-level applications that users can launch, you might want to assign different affinities to each activity. A good convention is to append different string names with a colon to the .apk package name. For example, the affinity names for "com.android.contacts" could be "com.android.contacts:Dialer" and "com.android.contacts:ContactsList".

- If you want to replace an internal activity from another application (e.g., notifications or shortcuts), you need to explicitly set the task affinity (`taskAffinity`) for the activity you want to replace. For instance, if you want to replace the contact details view (which users can access directly or via a shortcut), set the task affinity to "com.android.contacts".

### Launch Modes and Flags:

The primary way you control communication between activities and tasks is by setting launch mode attributes and intent flags. These two parameters can work together in various combinations to control the result of activity launches, as described in the relevant documentation. Here, we'll cover some common usage scenarios and combinations.

- The most commonly used mode is `singleTop` (besides the default `standard` mode). This doesn't affect the task much; it simply prevents the same activity from being launched multiple times on top of the stack.

- The `singleTask` mode affects tasks by ensuring the activity always opens in a new task (or brings an already opened task to the foreground). Using this mode requires careful consideration of how the process interacts with other system components, as it may affect all activities. This mode is best suited for the launcher activity of an application (supporting MAIN action and LAUNCHER category).

- The `singleInstance` launch mode is more specialized and should only be used when the entire application consists of a single activity.

There are cases where other entities (like the SearchManager or NotificationManager) start your activity. In such cases, you need to use the `Intent.FLAG_ACTIVITY_NEW_TASK` flag because the activity is started outside of a task (this app/task hasn't been started yet). As previously described, the standard behavior is that the current task matching the affinity of the new activity will be brought to the foreground, and the new activity will be started at the top of the stack. Of course, you can implement other types of behaviors.

A common approach is to combine `Intent.FLAG_ACTIVITY_CLEAR_TOP` with `NEW_TASK`. This way, if your task is already running, it will be brought to the foreground, and all activities except the root activity will be cleared. The `onNewIntent(Intent)` method of the root activity will be called with the new intent. When using this method, `singleTop` or `singleTask` launch modes are often used so that the current instance receives a new intent rather than destroying the original task and creating a new instance.

Another option is to set the activity's task affinity to an empty string (indicating no affinity) and set the `finishOnBackground` attribute. This is useful if you want to notify the user with a separate activity description rather than returning to the main task of the application. To specify this attribute, the activity will end whether the user uses BACK or HOME. If this attribute is not specified, pressing HOME will leave the activity and task in the system, making it impossible to return to that task.

Make sure to read the documentation on the `launchMode` attribute and `Intent` flags for detailed information on these options.

### Processes:

An Android process is a complete implementation of an Android application, though not necessarily understood by users in the conventional sense. Their primary purpose is straightforward:

As described earlier, the process attribute is used to control which processes special application components run in. Note that this attribute cannot violate system security: if two .apk packages cannot share the same user ID but attempt to run in the same process, this is not allowed, and in fact, the system will create two separate processes. Refer to the security documentation for more information on security restrictions.

### Threads:

Each process contains one or more threads. In most cases, Android avoids creating unnecessary threads within a process unless it creates its own thread, maintaining the single-threaded nature of applications. An important conclusion is that all instances of calls, broadcast receivers, and service instances are created by the main thread running in this process.

Note that new threads are not created for activities, broadcast receivers, services, or content providers: these application components are instantiated within the process (usually in the same process unless otherwise stated), which is actually the main thread of the process. This means that when these components (including services) are called by the system, they do not require remote procedure calls or blocking operations (such as network calls or computational loops) because doing so would block all other components in the process. You can use standard thread classes or Android's `HandlerThread` class to perform remote operations on other threads.

Here are some exceptions to the rule of creating threads:

- Move untrusted or unstable code to other processes to enhance stability and security.

- Run multiple .apk files in the same process to reduce system overhead.

- Help the system manage resources by placing important code in a separate process, allowing other parts of the application to be destroyed independently.

- Call IBinder or interfaces implemented by IBinder. If the call comes from another process, you can call them via threads sent by IBinder or the thread pool in the local process, but not from the main thread of the process. In special cases, calling an IBinder of a service can be handled this way (although methods in the service have already been executed in the main thread).

- Call the main methods of content providers dispatched by the thread performing the call or the main thread and IBinder. The designated methods are documented in the content provider class. This means that implementing these methods must follow a thread-safe pattern to allow simultaneous access by any other thread.

- Call views and subclasses composed of threads running in the view. Usually, this will be the main thread of the process. If you create a thread and display a window, the inherited window view will start from that thread.
