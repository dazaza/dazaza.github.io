Why aren't Macbook hard drives partitioned?

by anonymous on 2013-10-15 21:00:45

Why is Linux's filesystem designed as a single directory tree?

Can someone explain to me why Linux's file system is structured as a "single hierarchical tree" instead of having drive letters like Windows (C:\, D:\, etc.)? Is there a specific reason for this design choice?

(Editor's note: Both Apple's macOS and Linux are branches of Unix, so... ╮(╯▽╰)╭)

In short: A single-directory structure is better. Windows uses drive letters primarily for backward compatibility.

Doug O'Neal (163 votes)

The history of Unix's file system predates Windows by many years, so your question could be rephrased as: "Why did Windows introduce its own disk partition notation?"

A hierarchical file structure has an advantage in that any subdirectory or file can be traced back to the root directory. For example, if you need to migrate local files to a network drive, software won't notice any difference.

Suppose you have a system where the operating system cannot be modified, but you have a program with high I/O demands. You can easily mount the /usr directory as read-only and move the /opt (application installation directory) to an SSD. From the operating system's perspective, the overall file system structure remains unchanged. In contrast, handling such cases in Windows is more cumbersome, especially when dealing with paths like C:\Program Files\...

Gilles (70 votes)

This design has historical reasons and was partly more efficient at the time.

Multics

Multics, to my knowledge, was the first operating system to adopt a "hierarchical tree file structure," allowing directories to contain subdirectories. As stated in R.C. Daley and P.G. Neumann's paper *A General-Purpose File System For Secondary Storage*:

The second part of the paper describes a "hierarchical tree file structure" that is very efficient for operating systems. This structure balances reliability and flexibility...

To understand it better, imagine the file system as a tree where some directories are also files. However, there is one special directory whose pointer only points to a single, fixed directory—the root directory. The root directory is not explicitly referenced by other directories; instead, it implicitly represents a virtual branch we call the "file system."

At any given time, users are located in a specific directory called the "working directory," from which they can access files by specifying their entry points. Multiple users can work simultaneously within the same working directory.

Multics emphasizes flexibility, allowing users to work on just one branch of the entire file system while maintaining the convenience of managing files through directories. Directories are also used to control access permissions—READ permission allows users to access files in the directory, while EXECUTE permission allows them to modify files (a feature inherited by later Unix systems).

Multics adheres to the principle of a "single storage object pool." Although the paper doesn't elaborate on this, it was well-suited to the hardware conditions of the time when removable hard drives were either non-existent or ignored. Multics did have a backup storage pool, but it was invisible to users.

Unix

Unix was heavily influenced by Multics, but while Multics focused on flexibility, Unix prioritized simplicity.

A "single directory tree hierarchy" fits perfectly with Unix. Like Multics, the details of storage pools are often abstracted away from users. However, with the advent of removable hard drives, Unix introduced the mount and umount commands to expose device operations to users (only those with "administrative privileges" can execute these). In *The Unix Time-Sharing System*, Dennis Ritchie and Ken Thompson explained:

While the system root directory is usually on the same storage device, other branches may not be. Two considerations necessitate the mount command: ensuring consistent filenames for regular files and accommodating independent file structures on different devices. Through the mount command, a file can be replaced by the complete directory structure of a removable storage device. Essentially, the mount command substitutes a portion of the file hierarchy with a new structure by referencing external data. From the perspective of the entire file system, no changes occur. For example, during system installation, the root directory occupies a small space on the hard drive, while other user data resides on different disks. These are sequentially mounted to form an initial system. A "mountable" system is entirely achieved through reading and writing special files. Various applications can create their own file systems or simply copy one from elsewhere.

(Editor's note: The central idea here is that in Unix, everything is a file.)

Another advantage of the "hierarchical tree file system" is that it delegates the complexity of managing storage devices to the kernel. This means that while the kernel becomes increasingly complex, the applications built on top remain simple. Since the kernel handles the dirty work, most applications don't need to worry about these details, making it a more natural design.

Windows

Windows inherits two ancient lineages: the VMS operating system designed for VAX microcomputers and the CP/M operating system for early Intel microcomputers (such as the 8080 and Z80).

VMS had a distributed hierarchical file system called Files-11. In Files-11, a full file path includes a "node name," an associated user account, device name, directory tree, filename, file type, and version number. VMS has a powerful Logical Names feature that allows custom "shortcuts" to specific paths, so users don't need to care about the "real" directory paths.

CP/M was designed for 64kB memory and floppy disk systems, emphasizing simplicity from the start. It lacked the concept of "directories," but files were accessed via "drive locations," such as A: or B:.

MS-DOS 1.0 inherited this characteristic from CP/M. When MS-DOS 2.0 introduced the concept of "directories," it retained the "drive letter" prefix for backward compatibility with DOS 1.0. The slash "/" symbol was originally used in VMS and CP/M systems to indicate "command parameters," so DOS and Windows later adopted the backslash "\" as the path separator to differentiate between the two.

Windows inherited the habits of DOS and VMS, though they are largely obsolete today. It still retains the tradition of "drive letters." Underneath, Windows uses UNC paths (Universal Naming Convention, i.e., \servername\sharename), which are mainly relevant to advanced Windows users. Nowadays, Windows can indeed mount storage devices using "reparse points," eliminating the need for drive letters like C:.
